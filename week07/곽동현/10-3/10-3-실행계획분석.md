## 10.3 실행 계획 분석

MySQL 8.0 버전부터는 `EXPLAIN` 명령의 결과로 출력되는 실행 계획의 포맷을 기존 테이블 포맷과 `JSON`, `TREE` 형태로 선택할 수 있다.
포맷보다 더 중요한 것은 **실행 계획이 어떤 접근 방법을 사용해서 어떤 최적화를 수행하는지, 어떤 인덱스를 사용하는지**를 이해하는 것이다.
이 책에서는 가독성이 높은 **테이블 포맷**을 사용해 실행 계획을 표시한다.

```sql
mysql> EXPLAIN
       SELECT *
       FROM employees e
         INNER JOIN salaries s ON s.emp_no=e.emp_no
       WHERE first_name='ABC';

+----+-------------+-------+------------+------+--------------------+------------+---------+-------------------+------+----------+-------+
| id | select_type | table | partitions | type | possible_keys      | key        | key_len | ref               | rows | filtered | Extra |
+----+-------------+-------+------------+------+--------------------+------------+---------+-------------------+------+----------+-------+
|  1 | SIMPLE      | e     | NULL       | ref  | PRIMARY,ix_firstname | ix_firstname | 58    | const             |    1 |   100.00 | NULL  |
|  1 | SIMPLE      | s     | NULL       | ref  | PRIMARY            | PRIMARY    |       4 | employees.e.emp_no|   10 |   100.00 | NULL  |
+----+-------------+-------+------------+------+--------------------+------------+---------+-------------------+------+----------+-------+
```

**실행 계획 테이블의 각 라인(레코드)은** 쿼리 문장에서 사용된 테이블(서브쿼리로 임시 테이블을 생성한 경우 그 임시 테이블까지 포함)의 개수만큼 출력된다.

- 실행 순서는 **위에서 아래로** 순서대로 표시된다 (UNION이나 상관 서브쿼리와 같은 경우 순서대로 표시되지 않을 수도 있다).
- 위쪽에 출력된 결과일수록 (`id` 칼럼의 값이 작을수록) 쿼리의 바깥(Outer) 부분이거나 먼저 접근한 테이블이다.
- 아래쪽에 출력된 결과일수록 (`id` 칼럼의 값이 클수록) 쿼리의 안쪽(Inner) 부분 또는 나중에 접근한 테이블이다.

### 10.3.1 id 칼럼

하나의 `SELECT` 문장은 다시 1개 이상의 하위(SUB) `SELECT` 문장을 포함할 수 있다.

```sql
mysql> SELECT ...
       FROM (SELECT ... FROM tb_test1) tb1, tb_test2 tb2
       WHERE tb1.id=tb2.id;
```

이렇게 `SELECT` 키워드 단위로 구분한 것을 **"단위(SELECT) 쿼리"** 라고 표현한다.

실행 계획에서 가장 왼쪽에 표시되는 **`id` 칼럼은 단위 SELECT 쿼리별로 부여되는 식별자 값**이다.

**같은 id 값** — 여러 테이블이 조인되는 경우 (SELECT 문은 하나이지만 조인된 테이블 수만큼 실행 계획 레코드가 출력됨)

```sql
mysql> EXPLAIN
       SELECT e.emp_no, e.first_name, s.from_date, s.salary
       FROM employees e, salaries s
       WHERE e.emp_no=s.emp_no LIMIT 10;

+----+-------------+-------+-------+--------------+------+--------------------+
| id | select_type | table | type  | key          | ref  | rows               |
+----+-------------+-------+-------+--------------+------+--------------------+
|  1 | SIMPLE      | e     | index | ix_firstname | NULL | 300252             |
|  1 | SIMPLE      | s     | ref   | PRIMARY      | employees.e.emp_no | 10 |
+----+-------------+-------+-------+--------------+------+--------------------+
```

**다른 id 값** — 쿼리 문장이 3개의 단위 SELECT 쿼리로 구성된 경우

```sql
mysql> EXPLAIN
       SELECT
         ( (SELECT COUNT(*) FROM employees) + (SELECT COUNT(*) FROM departments) ) AS total_count;

+----+-------------+-------------+-------+------------+------+--------+------------------+
| id | select_type | table       | type  | key        | ref  | rows   | Extra            |
+----+-------------+-------------+-------+------------+------+--------+------------------+
|  1 | PRIMARY     | NULL        | NULL  | NULL       | NULL |   NULL | No tables used   |
|  3 | SUBQUERY    | departments | index | ux_deptname| NULL |      9 | Using index      |
|  2 | SUBQUERY    | employees   | index | ix_hiredate| NULL | 300252 | Using index      |
+----+-------------+-------------+-------+------------+------+--------+------------------+
```

> **주의**: `id` 칼럼이 테이블의 **접근 순서를 의미하지는 않는다.**
> 다음 예시에서 `dept_emp` 테이블의 `id` 값은 1이고 `employees` 테이블의 `id` 값은 2로 표시됐지만,
> 실제 실행 순서는 `employees` 테이블을 먼저 읽고 그 결과를 이용해 `dept_emp` 테이블을 읽는다.
> 실행 순서를 정확히 파악하려면 `EXPLAIN FORMAT=TREE` 명령으로 확인하는 것이 좋다.

```sql
mysql> EXPLAIN FORMAT=TREE
       SELECT * FROM dept_emp de
       WHERE de.emp_no= (SELECT e.emp_no
                         FROM employees e
                         WHERE e.first_name='Georgi'
                           AND e.last_name='Facello' LIMIT 1);

-> Filter: (de.emp_no = (select #2)) (cost=1.10 rows=1)
    -> Index lookup on de using ix_empno_fromdate (emp_no=(select #2)) (cost=1.10 rows=1)
    -> Select #2 (subquery in condition; run only once)
        -> Limit: 1 row(s)
            -> Filter: (e.last_name = 'Facello') (cost=70.49 rows=25)
                -> Index lookup on e using ix_firstname (first_name='Georgi') (cost=70.49 rows=253)
```

TREE 포맷 실행 계획에서 들여쓰기가 가장 많이 된 부분(employees의 ix_firstname 인덱스 읽기)이 가장 먼저 실행된다.

### 10.3.2 select_type 칼럼

각 단위 `SELECT` 쿼리가 **어떤 타입의 쿼리**인지 표시되는 칼럼이다.

#### 10.3.2.1 SIMPLE

UNION이나 서브쿼리를 사용하지 않는 단순한 SELECT 쿼리인 경우 `select_type`이 `SIMPLE`로 표시된다. 쿼리에 조인이 포함된 경우에도 마찬가지다. 실행 계획에서 `SIMPLE`인 단위 쿼리는 **하나만 존재**하며, 일반적으로 제일 바깥 SELECT 쿼리가 `SIMPLE`로 표시된다.

#### 10.3.2.2 PRIMARY

UNION이나 서브쿼리를 가지는 SELECT 쿼리의 실행 계획에서 **가장 바깥(Outer)에 있는 단위 쿼리**는 `PRIMARY`로 표시된다. `SIMPLE`과 마찬가지로 `PRIMARY`인 단위 쿼리는 하나만 존재한다.

#### 10.3.2.3 UNION

UNION으로 결합하는 단위 SELECT 쿼리 가운데 **첫 번째를 제외한 두 번째 이후 단위 SELECT 쿼리**의 `select_type`이 `UNION`으로 표시된다. UNION의 첫 번째 단위 SELECT는 `UNION`이 아니라 UNION 결과를 모아 저장하는 임시 테이블(`DERIVED`)의 `select_type`으로 표시된다.

```sql
mysql> EXPLAIN
SELECT * FROM (
    (SELECT emp_no FROM employees e1 LIMIT 10) UNION ALL
    (SELECT emp_no FROM employees e2 LIMIT 10) UNION ALL
    (SELECT emp_no FROM employees e3 LIMIT 10) ) tb;

+----+-------------+------------+-------+-------------+------+--------+-------------+
| id | select_type | table      | type  | key         | ref  | rows   | Extra       |
+----+-------------+------------+-------+-------------+------+--------+-------------+
|  1 | PRIMARY     | <derived2> | ALL   | NULL        | NULL |     30 | NULL        |
|  2 | DERIVED     | e1         | index | ix_hiredate | NULL | 300252 | Using index |
|  3 | UNION       | e2         | index | ix_hiredate | NULL | 300252 | Using index |
|  4 | UNION       | e3         | index | ix_hiredate | NULL | 300252 | Using index |
+----+-------------+------------+-------+-------------+------+--------+-------------+
```

#### 10.3.2.4 DEPENDENT UNION

UNION이나 UNION ALL로 결합된 단위 쿼리가 **외부 쿼리에 의해 영향을 받는** 경우를 의미한다. 내부 쿼리가 외부 값을 참조해서 처리될 때 `select_type`에 `DEPENDENT` 키워드가 표시된다.

```sql
mysql> EXPLAIN
SELECT *
FROM employees e1 WHERE e1.emp_no IN (
    SELECT e2.emp_no FROM employees e2 WHERE e2.first_name='Matt'
    UNION
    SELECT e3.emp_no FROM employees e3 WHERE e3.last_name='Matt'
);

+------+--------------------+------------+--------+---------+------+------+-----------------+
| id   | select_type        | table      | type   | key     | ref  | rows | Extra           |
+------+--------------------+------------+--------+---------+------+------+-----------------+
|    1 | PRIMARY            | e1         | ALL    | NULL    | NULL | 300252 | Using where  |
|    2 | DEPENDENT SUBQUERY | e2         | eq_ref | PRIMARY | func |    1 | Using where   |
|    3 | DEPENDENT UNION    | e3         | eq_ref | PRIMARY | func |    1 | Using where   |
| NULL | UNION RESULT       | <union2,3> | ALL    | NULL    | NULL | NULL | Using temporary |
+------+--------------------+------------+--------+---------+------+------+-----------------+
```

#### 10.3.2.5 UNION RESULT

UNION 결과를 담아두는 테이블을 의미한다. MySQL 8.0부터는 **UNION ALL의 경우 임시 테이블을 사용하지 않도록** 개선됐다. 하지만 UNION(DISTINCT)은 여전히 임시 테이블에 결과를 버퍼링한다. `UNION RESULT`는 실제 쿼리에서 단위 쿼리가 아니기 때문에 별도의 `id` 값은 부여되지 않는다.

```sql
mysql> EXPLAIN
SELECT emp_no FROM salaries WHERE salary>100000
UNION DISTINCT
SELECT emp_no FROM dept_emp WHERE from_date>'2001-01-01';

+------+-------------+------------+-------+-----------+--------+--------------------------+
| id   | select_type | table      | type  | key       | rows   | Extra                    |
+------+-------------+------------+-------+-----------+--------+--------------------------+
|    1 | PRIMARY     | salaries   | range | ix_salary | 191348 | Using where; Using index |
|    2 | UNION       | dept_emp   | range | ix_fromdate|  5325 | Using where; Using index |
| NULL | UNION RESULT| <union1,2> | ALL   | NULL      | NULL   | Using temporary          |
+------+-------------+------------+-------+-----------+--------+--------------------------+
```

`table` 칼럼의 `<union1,2>`는 id 값이 1과 2인 단위 쿼리의 조회 결과를 합쳤다는 의미다. UNION DISTINCT를 UNION ALL로 변경하면 `UNION RESULT` 라인이 사라진다.

#### 10.3.2.6 SUBQUERY

`select_type`의 `SUBQUERY`는 **FROM 절 이외에서 사용되는 서브쿼리**만을 의미한다. FROM 절에 사용된 서브쿼리는 `DERIVED`로 표시된다.

서브쿼리는 사용 위치에 따라 이름이 다르다:
- **중첩된 쿼리(Nested Query)**: SELECT되는 칼럼에 사용된 서브쿼리
- **서브쿼리(Subquery)**: WHERE 절에 사용된 경우
- **파생 테이블(Derived Table)**: FROM 절에 사용된 서브쿼리 (MySQL에서는 인라인 뷰, 서브 셀렉트라고도 부름)

반환하는 값의 특성에 따른 구분:
- **스칼라 서브쿼리(Scalar Subquery)**: 하나의 값만 반환하는 쿼리
- **로우 서브쿼리(Row Subquery)**: 칼럼 개수와 관계없이 하나의 레코드만 반환하는 쿼리

#### 10.3.2.7 DEPENDENT SUBQUERY

서브쿼리가 바깥쪽(Outer) SELECT 쿼리에서 정의된 칼럼을 사용하는 경우 `DEPENDENT SUBQUERY`가 표시된다. 외부 쿼리가 먼저 수행된 후 내부 쿼리(서브쿼리)가 실행되므로 일반 서브쿼리보다 처리 속도가 느릴 때가 많다.

```sql
mysql> EXPLAIN
SELECT e.first_name,
    (SELECT COUNT(*)
     FROM dept_emp de, dept_manager dm
     WHERE dm.dept_no=de.dept_no AND de.emp_no=e.emp_no) AS cnt
FROM employees e
WHERE e.first_name='Matt';

+----+--------------------+-------+--------+-------------------+------+-------------+
| id | select_type        | table | type   | key               | rows | Extra       |
+----+--------------------+-------+--------+-------------------+------+-------------+
|  1 | PRIMARY            | e     | ref    | ix_firstname      |  233 | Using index |
|  2 | DEPENDENT SUBQUERY | de    | ref    | ix_empno_fromdate |    1 | Using index |
|  2 | DEPENDENT SUBQUERY | dm    | ref    | PRIMARY           |    2 | Using index |
+----+--------------------+-------+--------+-------------------+------+-------------+
```

#### 10.3.2.8 DERIVED

MySQL 5.5 버전까지는 서브쿼리가 FROM 절에 사용된 경우 항상 `DERIVED` 실행 계획을 만든다. MySQL 5.6 버전부터는 옵티마이저 옵션에 따라 FROM 절의 서브쿼리를 외부 쿼리와 통합하는 최적화가 수행되기도 한다.

`DERIVED`는 단위 SELECT 쿼리의 실행 결과로 **메모리나 디스크에 임시 테이블(파생 테이블)을 생성**하는 것을 의미한다.

```sql
mysql> EXPLAIN
SELECT *
FROM (SELECT de.emp_no FROM dept_emp de GROUP BY de.emp_no) tb,
     employees e
WHERE e.emp_no=tb.emp_no;

+----+-------------+------------+--------+-------------------+--------+-------------+
| id | select_type | table      | type   | key               | rows   | Extra       |
+----+-------------+------------+--------+-------------------+--------+-------------+
|  1 | PRIMARY     | <derived2> | ALL    | NULL              | 331143 | NULL        |
|  1 | PRIMARY     | e          | eq_ref | PRIMARY           |      1 | NULL        |
|  2 | DERIVED     | de         | index  | ix_empno_fromdate | 331143 | Using index |
+----+-------------+------------+--------+-------------------+--------+-------------+
```

> **튜닝 팁**: 실행 계획 확인 시 `select_type` 칼럼의 값이 `DERIVED`인 것이 있는지 확인해야 한다. 서브쿼리를 조인으로 해결할 수 있는 경우라면 서브쿼리보다는 **조인을 사용할 것을 강력히 권장**한다.

#### 10.3.2.9 DEPENDENT DERIVED

MySQL 8.0부터 래터럴 조인(LATERAL JOIN) 기능이 추가되면서, FROM 절의 서브쿼리에서도 외부 칼럼을 사용할 수 있게 됐다. 래터럴 조인이 사용된 경우 `DEPENDENT DERIVED` 키워드가 표시된다.

```sql
mysql> SELECT *
FROM employees e
LEFT JOIN LATERAL
    (SELECT *
     FROM salaries s
     WHERE s.emp_no=e.emp_no
     ORDER BY s.from_date DESC LIMIT 2) AS s2 ON s2.emp_no=e.emp_no;

+----+-------------------+------------+--------+--------------+-----------------------------+
| id | select_type       | table      | type   | key          | Extra                       |
+----+-------------------+------------+--------+--------------+-----------------------------+
|  1 | PRIMARY           | e          | ALL    | NULL         | Rematerialize (<derived2>)  |
|  1 | PRIMARY           | <derived2> | ref    | <auto_key0>  | NULL                        |
|  2 | DEPENDENT DERIVED | s          | ref    | PRIMARY      | Using filesort              |
+----+-------------------+------------+--------+--------------+-----------------------------+
```

#### 10.3.2.10 UNCACHEABLE SUBQUERY

하나의 쿼리 문장에 서브쿼리가 하나만 있더라도 그 서브쿼리가 한 번만 실행되는 것은 아니다. 조건이 똑같은 서브쿼리가 실행될 때는 이전 결과를 캐시해서 재사용하는데, 캐시 자체가 불가능한 경우 `UNCACHEABLE SUBQUERY`가 표시된다.

캐시를 사용하지 못하게 하는 요소:
- 사용자 변수(`@var`)가 서브쿼리에 사용된 경우
- NOT-DETERMINISTIC 속성의 스토어드 루틴이 서브쿼리 내에 사용된 경우
- `UUID()`나 `RAND()`와 같이 결괏값이 호출할 때마다 달라지는 함수가 서브쿼리에 사용된 경우

```sql
mysql> EXPLAIN
SELECT *
FROM employees e WHERE e.emp_no = (
    SELECT @status FROM dept_emp de WHERE de.dept_no='d005' );

+----+----------------------+-------+------+---------+--------+-------------+
| id | select_type          | table | type | key     | rows   | Extra       |
+----+----------------------+-------+------+---------+--------+-------------+
|  1 | PRIMARY              | e     | ALL  | NULL    | 300252 | Using where |
|  2 | UNCACHEABLE SUBQUERY | de    | ref  | PRIMARY | 165571 | Using index |
+----+----------------------+-------+------+---------+--------+-------------+
```

#### 10.3.2.11 UNCACHEABLE UNION

`UNION`과 `UNCACHEABLE`의 속성이 합쳐진 `select_type`을 의미한다.

#### 10.3.2.12 MATERIALIZED

MySQL 5.6 버전부터 도입된 `select_type`으로, 주로 FROM 절이나 `IN(subquery)` 형태의 쿼리에 사용된 서브쿼리의 최적화를 위해 사용된다. MySQL 5.7 버전부터는 서브쿼리의 내용을 임시 테이블로 구체화(Materialization)한 후, 임시 테이블과 테이블을 조인하는 형태로 최적화되어 처리된다.

```sql
mysql> EXPLAIN
SELECT *
FROM employees e
WHERE e.emp_no IN (SELECT emp_no FROM salaries WHERE salary BETWEEN 100 AND 1000);

+----+-------------+-------------+--------+-----------+------+--------------------------+
| id | select_type | table       | type   | key       | rows | Extra                    |
+----+-------------+-------------+--------+-----------+------+--------------------------+
|  1 | SIMPLE      | <subquery2> | ALL    | NULL      | NULL | NULL                     |
|  1 | SIMPLE      | e           | eq_ref | PRIMARY   |    1 | NULL                     |
|  2 | MATERIALIZED| salaries    | range  | ix_salary |    1 | Using where; Using index |
+----+-------------+-------------+--------+-----------+------+--------------------------+
```

### 10.3.3 table 칼럼

MySQL 서버의 실행 계획은 단위 SELECT 쿼리 기준이 아니라 **테이블 기준으로 표시**된다. 별칭이 부여된 경우에는 별칭이 표시된다.

- `FROM` 절이 없거나 `FROM DUAL`인 경우 `table` 칼럼에 `NULL`이 표시된다.
- `<derived N>`, `<union M,N>` 같이 `<>` 로 둘러싸인 이름은 **임시 테이블**을 의미한다.
- `<>` 안의 숫자는 단위 SELECT 쿼리의 `id` 값을 지칭한다.

```
+----+-------------+------------+--------+-------------------+--------+-------------+
| id | select_type | table      | type   | key               | rows   | Extra       |
+----+-------------+------------+--------+-------------------+--------+-------------+
|  1 | PRIMARY     | <derived2> | ALL    | NULL              | 331143 | NULL        |  <- id=2의 결과로 만든 파생 테이블
|  1 | PRIMARY     | e          | eq_ref | PRIMARY           |      1 | NULL        |
|  2 | DERIVED     | de         | index  | ix_empno_fromdate | 331143 | Using index |
+----+-------------+------------+--------+-------------------+--------+-------------+
```

**실행 계획 읽는 순서 분석**:
1. 첫 번째 라인의 table이 `<derived2>`이므로, id=2인 라인이 먼저 실행되어 파생 테이블이 준비된다.
2. id=2인 라인은 `dept_emp` 테이블을 읽어서 파생 테이블을 생성한다.
3. 첫 번째, 두 번째 라인은 같은 id를 가지므로 두 테이블(`<derived2>`와 `e`)이 조인된다.
4. `<derived2>`가 위에 표시됐으므로 드라이빙 테이블이고, `e`가 드리븐 테이블이다.

MySQL 8.0에서 `select_type`이 `MATERIALIZED`인 실행 계획에서는 `<subquery N>` 형태가 `table` 칼럼에 표시된다.

---

### 10.3.4 partitions 칼럼

MySQL 8.0부터는 `EXPLAIN` 명령으로 파티션 관련 실행 계획까지 모두 확인할 수 있다. 파티션 키 칼럼을 WHERE 조건으로 가진 쿼리의 경우, 옵티마이저가 필요한 파티션들의 목록만 모아서 `partitions` 칼럼에 표시해준다.

**파티션 프루닝(Partition Pruning)**: 파티션이 여러 개인 테이블에서 불필요한 파티션을 빼고 접근해야 할 것으로 판단되는 파티션만 골라내는 과정이다.

```sql
mysql> EXPLAIN
SELECT *
FROM employees_2
WHERE hire_date BETWEEN '1999-11-15' AND '2000-01-15';

+----+-------------+-------------+------------------------+------+-------+
| id | select_type | table       | partitions             | type | rows  |
+----+-------------+-------------+------------------------+------+-------+
|  1 | SIMPLE      | employees_2 | p1996_2000,p2001_2005  | ALL  | 21743 |
+----+-------------+-------------+------------------------+------+-------+
```

`type` 칼럼의 값이 `ALL`이지만, 이는 파티션 전체가 아닌 **해당 파티션만 풀 스캔**한다는 의미다. 파티션은 물리적으로 개별 테이블처럼 별도의 저장 공간을 가지기 때문이다.

---

### 10.3.5 type 칼럼

실행 계획에서 `type` 이후의 칼럼은 MySQL 서버가 각 테이블의 레코드를 어떤 방식으로 읽었는지를 나타낸다. MySQL 매뉴얼에서는 `type` 칼럼을 "조인 타입"으로 소개하지만, 실제로는 **각 테이블의 접근 방법(Access type)** 으로 해석하면 된다.

성능이 빠른 순서대로:

| type | 설명 |
|------|------|
| `system` | 레코드가 1건 이하인 테이블 (MyISAM, MEMORY만 해당) |
| `const` | PK나 Unique Key 전체를 동등 조건으로 1건만 반환 |
| `eq_ref` | 조인에서 두 번째 읽는 테이블의 PK/Unique Key로 1건 검색 |
| `ref` | 인덱스 종류 무관, 동등 조건으로 검색 (1건 보장 불필요) |
| `fulltext` | 전문 검색(Full-text Search) 인덱스 사용 |
| `ref_or_null` | ref + IS NULL 비교 추가 |
| `unique_subquery` | IN(subquery)에서 서브쿼리가 중복 없는 값 반환 |
| `index_subquery` | IN(subquery)에서 서브쿼리가 중복값이 있을 수 있으나 인덱스로 제거 가능 |
| `range` | 인덱스 레인지 스캔 (`<`, `>`, `BETWEEN`, `IN`, `LIKE` 등) |
| `index_merge` | 2개 이상의 인덱스로 검색 후 결과 병합 |
| `index` | 인덱스 풀 스캔 |
| `ALL` | 풀 테이블 스캔 |

`ALL`을 제외한 나머지는 모두 인덱스를 사용하는 접근 방법이다. `index_merge`를 제외한 나머지 접근 방법은 하나의 인덱스만 사용한다.

#### 10.3.5.1 system

레코드가 1건만 존재하는 테이블 또는 한 건도 존재하지 않는 테이블을 참조하는 형태의 접근 방법이다. **InnoDB 스토리지 엔진에서는 나타나지 않고**, MyISAM이나 MEMORY 테이블에서만 사용된다.

#### 10.3.5.2 const

테이블의 레코드 건수와 관계없이 쿼리가 **프라이머리 키나 유니크 키 칼럼 전체를 이용하는 WHERE 조건**을 가지고 있으며, 반드시 1건을 반환하는 쿼리의 처리 방식이다.

```sql
mysql> EXPLAIN
SELECT * FROM employees WHERE emp_no=10001;
-- type: const

mysql> EXPLAIN
SELECT * FROM dept_emp WHERE dept_no='d005';
-- type: ref (PK의 일부만 조건으로 사용, 1건 보장 안됨)

mysql> EXPLAIN
SELECT * FROM dept_emp WHERE dept_no='d005' AND emp_no=10001;
-- type: const (PK 모든 칼럼 동등 조건)
```

`const` 타입의 실행 계획은 옵티마이저가 쿼리를 최적화하는 단계에서 쿼리를 먼저 실행해서 **통째로 상수화**한다.

#### 10.3.5.3 eq_ref

여러 테이블이 조인되는 쿼리의 실행 계획에서만 표시된다. 조인에서 처음 읽은 테이블의 칼럼값을, **다음 읽어야 할 테이블의 프라이머리 키나 유니크 키 칼럼의 검색 조건**에 사용할 때를 가리켜 `eq_ref`라고 한다. 두 번째 이후에 읽히는 테이블에서 반드시 **1건만 존재한다는 보장이 있어야** 사용할 수 있다.

```sql
mysql> EXPLAIN
SELECT * FROM dept_emp de, employees e
WHERE e.emp_no=de.emp_no AND de.dept_no='d005';

+----+-------------+----------+--------+---------+---------+--------+
| id | select_type | table    | type   | key     | key_len | rows   |
+----+-------------+----------+--------+---------+---------+--------+
|  1 | SIMPLE      | de       | ref    | PRIMARY | 16      | 165571 |
|  1 | SIMPLE      | e        | eq_ref | PRIMARY | 4       |      1 |
+----+-------------+----------+--------+---------+---------+--------+
```

#### 10.3.5.4 ref

조인의 순서와 관계없이 사용되며, 프라이머리 키나 유니크 키 등의 제약 조건도 없다. **인덱스 종류에 관계없이 동등(Equal) 조건으로 검색**할 때 `ref` 접근 방법이 사용된다. 반환되는 레코드가 반드시 1건이라는 보장이 없다.

```sql
mysql> EXPLAIN
SELECT * FROM dept_emp WHERE dept_no='d005';

-- type: ref (PK 일부 칼럼만 동등 조건, ref 칼럼에 'const'가 표시됨)
```

**const, eq_ref, ref 비교**:
- `const`: 조인 순서 무관, PK/Unique Key 모든 칼럼에 동등 조건 → 반드시 1건 반환
- `eq_ref`: 조인에서 첫 번째 테이블 칼럼으로 두 번째 테이블을 PK/Unique Key 동등 검색 → 반드시 1건 반환
- `ref`: 조인 순서 및 인덱스 종류 무관, 동등 조건 검색 → 1건 보장 없음

이 세 가지 모두 `WHERE` 조건절에 동등 비교 연산자(`=` 또는 `<=>`)를 사용하며, **인덱스 분포도가 나쁘지 않다면 성능상 문제를 일으키지 않는 접근 방법**이다.

#### 10.3.5.5 fulltext

MySQL 서버의 **전문 검색(Full-text Search) 인덱스**를 사용해 레코드를 읽는 접근 방법이다. 전문 검색 인덱스는 통계 정보가 관리되지 않으며, `MATCH (...) AGAINST (...)` 구문을 사용해야 한다.

MySQL 서버에서 전문 검색 조건은 우선순위가 상당히 높다. 쿼리에 전문 인덱스를 사용하는 조건과 일반 인덱스를 사용하는 조건을 함께 사용하면, 일반 인덱스의 접근 방법이 `const`나 `eq_ref`, `ref`가 아니면 일반적으로 MySQL은 전문 인덱스를 사용하는 조건을 선택한다.

> **주의**: 전문 검색 인덱스를 이용하는 `fulltext`보다 일반 인덱스를 이용하는 `range` 접근 방법이 더 빨리 처리되는 경우가 더 많다. 전문 검색 쿼리를 사용할 때는 조건별로 성능을 확인해 보는 편이 좋다.

#### 10.3.5.6 ref_or_null

`ref` 접근 방법과 같은데, **NULL 비교(IS NULL)가 추가된 형태**다.

```sql
mysql> EXPLAIN
SELECT * FROM titles
WHERE to_date='1985-03-01' OR to_date IS NULL;
-- type: ref_or_null
```

#### 10.3.5.7 unique_subquery

`WHERE` 조건절에서 사용될 수 있는 `IN(subquery)` 형태의 쿼리를 위한 접근 방법이다. **서브쿼리에서 중복되지 않는 유니크한 값만 반환**할 때 사용한다.

> MySQL 8.0에서는 `IN(subquery)` 형태의 세미 조인을 최적화하기 위한 많은 기능이 도입됐다. `semijoin` 최적화 옵션을 비활성화한 상태에서만 `unique_subquery`가 표시된다.

#### 10.3.5.8 index_subquery

`IN(subquery)`에서 subquery가 중복된 값을 반환할 수도 있을 때, **인덱스를 이용해 중복된 값을 제거**할 수 있을 때 사용된다.

- `unique_subquery`: IN(subquery)에서 subquery 반환 값에 중복이 없으므로 별도 중복 제거 작업 불필요
- `index_subquery`: IN(subquery)에서 subquery 반환 값에 중복이 있을 수 있지만 인덱스로 중복 제거 가능

#### 10.3.5.9 range

우리가 익히 알고 있는 **인덱스 레인지 스캔** 형태의 접근 방법이다. 인덱스를 하나의 값이 아니라 **범위로 검색**하는 경우를 의미하며, 주로 `<`, `>`, `IS NULL`, `BETWEEN`, `IN`, `LIKE` 등의 연산자를 이용해 인덱스를 검색할 때 사용된다.

```sql
mysql> EXPLAIN
SELECT * FROM employees WHERE emp_no BETWEEN 10002 AND 10004;
-- type: range
```

> 인덱스 레인지 스캔이라고 하면 `const`, `ref`, `range` 세 가지 접근 방법을 모두 묶어서 지칭하는 것임을 기억하자.

#### 10.3.5.10 index_merge

다른 접근 방법과는 달리 **2개 이상의 인덱스를 이용해 각각의 검색 결과를 만들어낸 후, 그 결과를 병합**해서 처리하는 방식이다.

특징:
- 여러 인덱스를 읽어야 하므로 일반적으로 `range` 접근 방법보다 효율성이 떨어진다.
- 전문 검색 인덱스를 사용하는 쿼리에서는 `index_merge`가 적용되지 않는다.
- 처리된 결과는 항상 2개 이상의 집합이 되기 때문에 교집합, 합집합, 중복 제거와 같은 부가적인 작업이 필요하다.

```sql
mysql> EXPLAIN
SELECT * FROM employees
WHERE emp_no BETWEEN 10001 AND 11000
OR first_name='Smith';

+----+------------+-------------------------------+---------+----------------------------------------------+
| id | type       | key                           | key_len | Extra                                        |
+----+------------+-------------------------------+---------+----------------------------------------------+
|  1 | index_merge| PRIMARY,ix_firstname          | 4,58    | Using union(PRIMARY,ix_firstname);Using where|
+----+------------+-------------------------------+---------+----------------------------------------------+
```

#### 10.3.5.11 index

접근 방법의 이름이 `index`라서 효율적으로 인덱스를 사용하는 것으로 오해하기 쉽지만, **인덱스를 처음부터 끝까지 읽는 인덱스 풀 스캔**을 의미한다.

`index` 접근 방법은 풀 테이블 스캔과 비교하는 레코드 건수는 같다. 하지만 인덱스는 데이터 파일 전체보다 크기가 작으므로 더 빠르게 처리되며, 정렬된 인덱스의 장점을 이용할 수 있다.

사용 조건(첫 번째+두 번째, 또는 첫 번째+세 번째 충족 시):
1. `range`, `const`, `ref` 등 접근 방법으로 인덱스를 사용하지 못하는 경우
2. 인덱스에 포함된 칼럼만으로 처리할 수 있는 쿼리인 경우 (데이터 파일 접근 불필요)
3. 인덱스를 이용해 정렬이나 그루핑 작업이 가능한 경우 (별도의 정렬 작업 피할 수 있는 경우)

```sql
mysql> EXPLAIN
SELECT * FROM departments ORDER BY dept_name DESC LIMIT 10;
-- type: index (ux_deptname 인덱스 역순 읽기)
```

#### 10.3.5.12 ALL

**풀 테이블 스캔**을 의미한다. 테이블을 처음부터 끝까지 전부 읽어서 불필요한 레코드를 제거하고 반환한다. 지금까지 설명한 접근 방법으로는 처리할 수 없을 때 가장 마지막에 선택하는 가장 비효율적인 방법이다.

InnoDB는 대량의 디스크 I/O를 유발하는 작업을 위해 **리드 어헤드(Read Ahead)** 기능을 제공한다. 인접한 페이지가 연속해서 몇 번 읽히면 백그라운드로 작동하는 읽기 스레드가 최대 64개의 페이지씩 한꺼번에 디스크로부터 읽어 들인다.

MySQL 8.0에서는 **병렬 쿼리(Parallel Query)** 기능도 도입됐다:

```sql
-- innodb_parallel_read_threads=1: 2분 33초
-- innodb_parallel_read_threads=4: 21.85초
-- innodb_parallel_read_threads=32: 5.35초
mysql> SELECT /*+ SET_VAR(innodb_parallel_read_threads=4) */ COUNT(*) FROM big_table;
```

> `index`나 `ALL` 접근 방법은 작업 범위를 제한하는 조건이 아니므로 빠른 응답이 필요한 웹 서비스 등의 온라인 트랜잭션 처리 환경에는 적합하지 않다.

---

### 10.3.6 possible_keys 칼럼

옵티마이저가 최적의 실행 계획을 만들기 위해 후보로 선정했던 접근 방법에서 사용되는 인덱스의 목록이다. "**사용될 법했던 인덱스의 목록**"이다.

실제로 실행 계획을 보면 그 테이블의 모든 인덱스가 목록에 포함되어 나오는 경우가 허다하기에 쿼리 튜닝에 크게 도움이 되지 않는다. **`possible_keys` 칼럼에 인덱스 이름이 나열됐다고 해서 그 인덱스를 사용한다고 판단하지 말 것.**

---

### 10.3.7 key 칼럼

`possible_keys` 칼럼의 인덱스가 사용 후보였던 반면, **`key` 칼럼에 표시되는 인덱스는 최종 선택된 실행 계획에서 사용하는 인덱스**를 의미한다. 쿼리를 튜닝할 때는 `key` 칼럼에 의도했던 인덱스가 표시되는지 확인하는 것이 중요하다.

- `key` 칼럼에 `PRIMARY`가 표시되면 프라이머리 키를 사용한다는 의미이다.
- `type`이 `ALL`이어서 인덱스를 전혀 사용하지 못하면 `key` 칼럼은 `NULL`로 표시된다.
- `type`이 `index_merge`일 때는 2개 이상의 인덱스가 사용되며, `,`로 구분되어 표시된다.

---

### 10.3.8 key_len 칼럼

**쿼리를 처리하기 위해 다중 칼럼으로 구성된 인덱스에서 몇 개의 칼럼까지(정확히는 몇 바이트까지) 사용했는지** 알려주는 칼럼이다. 많은 사용자가 쉽게 무시하지만 매우 중요한 정보다.

```sql
-- dept_no(CHAR(4), utf8mb4) + emp_no(INT)로 구성된 복합 PK

mysql> EXPLAIN SELECT * FROM dept_emp WHERE dept_no='d005';
-- key_len: 16  (dept_no만 사용, CHAR(4) * 4바이트 = 16바이트)

mysql> EXPLAIN SELECT * FROM dept_emp WHERE dept_no='d005' AND emp_no=10001;
-- key_len: 20  (dept_no 16바이트 + emp_no 4바이트)
```

> **NULLABLE 칼럼 주의**: NULLABLE인 칼럼은 NULL 여부를 저장하기 위해 1바이트를 추가로 사용한다. 예를 들어, DATE 타입(3바이트)의 NULLABLE 칼럼이면 `key_len`에 4(3+1)가 표시된다.

---

### 10.3.9 ref 칼럼

접근 방법이 `ref`일 때, 참조 조건(Equal 비교 조건)으로 **어떤 값이 제공됐는지** 보여준다.

- 상숫값을 지정했다면 `ref` 칼럼의 값은 `const`로 표시된다.
- 다른 테이블의 칼럼값이면 그 테이블명과 칼럼명이 표시된다.
- 조인 칼럼에 콜레이션 변환이나 값 연산이 적용된 경우 `func`로 표시된다.

```sql
-- e.emp_no=(de.emp_no-1) 처럼 값을 가공하면 ref 칼럼에 'func' 표시
```

가능하다면 MySQL 서버가 변환하지 않아도 되게 조인 칼럼의 타입은 일치시키는 편이 좋다.

---

### 10.3.10 rows 칼럼

**실행 계획의 효율성 판단을 위해 예측했던 레코드 건수**를 보여준다. 통계 정보를 기준으로 예측한 값이라서 정확하지는 않다. 또한 반환하는 레코드의 예측치가 아니라 **쿼리를 처리하기 위해 얼마나 많은 레코드를 읽고 체크해야 하는지**를 의미한다.

```sql
mysql> EXPLAIN SELECT * FROM dept_emp WHERE from_date>='1985-01-01';
-- rows: 331143, type: ALL (테이블 전체 레코드를 읽어야 한다고 판단, 풀 테이블 스캔)

mysql> EXPLAIN SELECT * FROM dept_emp WHERE from_date>='2002-07-01';
-- rows: 292, type: range (전체 테이블의 약 0.088%, 인덱스 레인지 스캔 선택)
```

MySQL 8.0부터는 더 정확한 예측을 위해 **히스토그램**이 도입됐다. (10.1.2절 참조)

### 10.3.11 filtered 칼럼

각 테이블에서 일치하는 레코드 개수를 파악하기 위한 칼럼이다. **필터링되고 남은 레코드의 비율(%)**을 의미한다. (버려지는 비율이 아님)

조인 순서 결정에 활용된다. 드라이빙 테이블로 선정될 가능성이 높은 테이블은 최종적으로 일치하는 레코드 건수가 적은 테이블이다.

```sql
mysql> EXPLAIN
SELECT *
FROM employees e, salaries s
WHERE e.first_name='Matt'
    AND e.hire_date BETWEEN '1990-01-01' AND '1991-01-01'
    AND s.emp_no=e.emp_no
    AND s.from_date BETWEEN '1990-01-01' AND '1991-01-01'
    AND s.salary BETWEEN 50000 AND 60000;

+----+-------------+-------+------+--------------+------+----------+
| id | select_type | table | type | key          | rows | filtered |
+----+-------------+-------+------+--------------+------+----------+
|  1 | SIMPLE      | e     | ref  | ix_firstname |  233 |    16.03 |  <- 233건 중 16.03%만 hire_date 조건 통과 = 약 37건 조인
|  1 | SIMPLE      | s     | ref  | PRIMARY      |  10  |     0.48 |
+----+-------------+-------+------+--------------+------+----------+
```

MySQL 8.0에서는 `filtered` 칼럼의 값을 더 정확히 예측할 수 있도록 **히스토그램** 기능이 도입됐다.

### 10.3.12 Extra 칼럼

칼럼의 이름과는 달리, **쿼리의 실행 계획에서 성능에 관련된 중요한 내용**이 자주 표시된다. 주로 내부적인 처리 알고리즘에 대해 조금 더 깊이 있는 내용을 보여주는 경우가 많다.

#### 10.3.12.1 const row not found

`const` 접근 방법으로 테이블을 읽었지만 실제로 해당 테이블에 **레코드가 1건도 존재하지 않으면** 이 내용이 표시된다.

#### 10.3.12.2 Deleting all rows

WHERE 조건절이 없는 DELETE 문장의 실행 계획에서 자주 표시되며, 테이블의 모든 레코드를 삭제하는 핸들러 기능(API)을 한 번 호출함으로써 처리됐다는 것을 의미한다.

> MySQL 8.0에서는 InnoDB와 MyISAM 엔진 모두 더 이상 "Deleting all rows" 최적화는 표시되지 않는다. 테이블의 모든 레코드를 삭제하고자 한다면 **`TRUNCATE TABLE` 명령을 사용**할 것을 권장한다.

#### 10.3.12.3 Distinct

`DISTINCT` 처리를 위해 조인하지 않아도 되는 항목은 모두 무시하고 꼭 필요한 것만 조인했으며, 꼭 필요한 레코드만 읽었다는 것을 의미한다.

```sql
mysql> EXPLAIN
SELECT DISTINCT d.dept_no
FROM departments d, dept_emp de WHERE de.dept_no=d.dept_no;

+----+-------------+-------+-------+-------------+--------------------------+
| id | select_type | table | type  | key         | Extra                    |
+----+-------------+-------+-------+-------------+--------------------------+
|  1 | SIMPLE      | d     | index | ux_deptname | Using index; Using temporary |
|  1 | SIMPLE      | de    | ref   | PRIMARY     | Using index; Distinct    |
+----+-------------+-------+-------+-------------+--------------------------+
```

#### 10.3.12.4 FirstMatch

세미 조인 최적화 중에서 **FirstMatch 전략**이 사용되면 표시된다. 기준 테이블을 기준으로 다른 테이블에서 첫 번째로 일치하는 한 건만 검색한다는 것을 의미한다.

```
| id | table | type | key          | rows | Extra                               |
|  1 | e     | ref  | ix_firstname |  233 | NULL                                |
|  2 | t     | ref  | PRIMARY      |    1 | Using where; Using index; FirstMatch(e) |
```

#### 10.3.12.5 Full scan on NULL key

`col1 IN (SELECT col2 FROM ...)` 형태의 조건에서 `col1`의 값이 NULL이 된다면 SQL 표준에 따라 차선책으로 **서브쿼리 테이블에 대해 풀 테이블 스캔을 사용할 것**이라는 사실을 알려주는 키워드다.

NULL이 실제로 없다면 풀 테이블 스캔은 발생하지 않는다. NULL 비교 규칙을 무시해도 된다면 `col1 IS NOT NULL` 조건을 추가하면 된다.

#### 10.3.12.6 Impossible HAVING

쿼리에 사용된 `HAVING` 절의 조건을 만족하는 레코드가 없을 때 표시된다.

```sql
mysql> EXPLAIN
SELECT e.emp_no, COUNT(*) AS cnt
FROM employees e
WHERE e.emp_no=10001
GROUP BY e.emp_no
HAVING e.emp_no IS NULL;
-- Extra: Impossible HAVING (emp_no는 NOT NULL이므로 HAVING 조건 불가)
```

#### 10.3.12.7 Impossible WHERE

WHERE 조건이 항상 FALSE가 될 수밖에 없는 경우 표시된다.

```sql
mysql> EXPLAIN
SELECT * FROM employees WHERE emp_no IS NULL;
-- Extra: Impossible WHERE (emp_no는 NOT NULL)
```

#### 10.3.12.8 LooseScan

세미 조인 최적화 중에서 **LooseScan 최적화 전략**이 사용되면 표시된다.

```sql
mysql> EXPLAIN
SELECT * FROM departments d WHERE d.dept_no IN (
    SELECT de.dept_no FROM dept_emp de );

+----+-------+-------+---------+--------+------------------------+
| id | table | type  | key     | rows   | Extra                  |
+----+-------+-------+---------+--------+------------------------+
|  1 | de    | index | PRIMARY | 331143 | Using index; LooseScan |
|  1 | d     | ref   | PRIMARY |      1 | NULL                   |
+----+-------+-------+---------+--------+------------------------+
```

#### 10.3.12.9 No matching min/max row

`MIN()`, `MAX()` 같은 집합 함수가 있는 쿼리의 조건절에 일치하는 레코드가 한 건도 없을 때 표시된다. `MIN()`이나 `MAX()`의 결과로 `NULL`이 반환된다.

#### 10.3.12.10 no matching row in const table

조인에 사용된 테이블에서 `const` 방법으로 접근할 때 **일치하는 레코드가 없다면** 표시된다. "Impossible WHERE"와 같은 종류로, 실행 계획을 만들기 위한 기초 자료가 없음을 의미한다.

#### 10.3.12.11 No matching rows after partition pruning

파티션된 테이블에 대한 `UPDATE` 또는 `DELETE` 명령의 실행 계획에서 표시될 수 있는데, **해당 파티션에서 UPDATE하거나 DELETE할 대상 레코드가 없을 때** 표시된다.

단순히 삭제할 레코드가 없음을 의미하는 것이 아니라 **대상 파티션 자체가 없다**는 것을 의미한다.

```sql
mysql> EXPLAIN DELETE FROM employees_parted WHERE hire_date>='2020-01-01';
-- Extra: No matching rows after partition pruning
-- (파티션 정의가 2006-01-01까지만 있으므로 대상 파티션 없음)
```

#### 10.3.12.12 No tables used

FROM 절이 없는 쿼리 문장이나 `FROM DUAL` 형태의 쿼리 실행 계획에서 표시된다.

```sql
mysql> EXPLAIN SELECT 1;
mysql> EXPLAIN SELECT 1 FROM DUAL;
-- Extra: No tables used
```

#### 10.3.12.13 Not exists

아우터 조인(LEFT OUTER JOIN)을 이용해 안티-조인을 수행하는 쿼리에서 표시된다. 옵티마이저가 드리븐 테이블의 레코드가 존재하는지 아닌지만 판단하며, 조인 조건에 일치하는 레코드가 여러 건이 있다고 하더라도 **딱 1건만 조회해보고 처리를 완료**하는 최적화를 의미한다.

```sql
mysql> EXPLAIN
SELECT *
FROM dept_emp de
LEFT JOIN departments d ON de.dept_no=d.dept_no
WHERE d.dept_no IS NULL;

+----+-------------+-------+--------+---------+----------------------------+
| id | select_type | table | type   | key     | Extra                      |
+----+-------------+-------+--------+---------+----------------------------+
|  1 | SIMPLE      | de    | ALL    | NULL    | NULL                       |
|  1 | SIMPLE      | d     | eq_ref | PRIMARY | Using where; Not exists    |
+----+-------------+-------+--------+---------+----------------------------+
```

#### 10.3.12.14 Plan isn't ready yet

`EXPLAIN FOR CONNECTION` 명령을 실행했을 때, 해당 커넥션에서 아직 쿼리의 실행 계획을 수립하지 못한 상태임을 의미한다. 대상 커넥션의 쿼리가 실행 계획을 수립할 여유 시간을 좀 더 주고 다시 실행하면 된다.

```sql
-- 다른 커넥션(id=8)에서 실행 중인 쿼리의 실행 계획 확인
mysql> EXPLAIN FOR CONNECTION 8;
```

#### 10.3.12.15 Range checked for each record(index map: N)

두 개의 테이블을 조인하는데 조인 조건에 상수가 없고 둘 다 변수인 경우, 최적의 접근 방법을 레코드 단위로 결정해야 할 때 표시된다. 즉, **레코드마다 인덱스 레인지 스캔과 풀 테이블 스캔 중에서 어느 것이 효율적일지 판단**한다.

```sql
mysql> EXPLAIN
SELECT *
FROM employees e1, employees e2
WHERE e2.emp_no >= e1.emp_no;

+----+-------------+-------+------+----------------------------------------------------+
| id | select_type | table | type | Extra                                              |
+----+-------------+-------+------+----------------------------------------------------+
|  1 | SIMPLE      | e1    | ALL  | NULL                                               |
|  1 | SIMPLE      | e2    | ALL  | Range checked for each record (index map: 0x1)     |
+----+-------------+-------+------+----------------------------------------------------+
```

`index map`은 16진수로 표시되며, 이진수로 변환하면 각 비트가 SHOW CREATE TABLE에 나열된 인덱스의 순번을 나타낸다. 실행 계획의 `type` 칼럼에 `ALL`이 표시되지만, `index map`에 표시된 후보 인덱스가 도움이 되지 않을 경우에만 최종적으로 풀 테이블 스캔을 사용한다.

#### 10.3.12.16 Recursive

MySQL 8.0부터 CTE(Common Table Expression)를 이용해 재귀 쿼리를 작성할 수 있게 됐다. **재귀 CTE가 사용된 경우** Extra 칼럼에 `Recursive` 문구가 표시된다.

```sql
mysql> WITH RECURSIVE cte (n) AS
(
    SELECT 1
    UNION ALL
    SELECT n + 1 FROM cte WHERE n < 5
)
SELECT * FROM cte;

+----+-------------+------------+--------+------+------------------------+
| id | select_type | table      | type   | key  | Extra                  |
+----+-------------+------------+--------+------+------------------------+
|  1 | PRIMARY     | <derived2> | ALL    | NULL | NULL                   |
|  2 | DERIVED     | NULL       | NULL   | NULL | No tables used         |
|  3 | UNION       | cte        | ALL    | NULL | Recursive; Using where |
+----+-------------+------------+--------+------+------------------------+
```

WITH 구문이 재귀 CTE로 사용될 경우에만 `Recursive` 메시지가 표시된다. (비재귀 CTE에서는 표시되지 않음)

#### 10.3.12.17 Rematerialize

MySQL 8.0부터 추가된 래터럴 조인에서, 래터럴로 조인되는 테이블은 **선행 테이블의 레코드별로 서브쿼리를 실행해서 그 결과를 임시 테이블에 저장**한다. 매번 임시 테이블이 새로 생성될 때 표시된다.

```sql
mysql> EXPLAIN
SELECT * FROM employees e
LEFT JOIN LATERAL (SELECT *
                   FROM salaries s
                   WHERE s.emp_no=e.emp_no
                   ORDER BY s.from_date DESC LIMIT 2) s2 ON s2.emp_no=e.emp_no
WHERE e.first_name='Matt';

+----+-------------------+------------+------+--------------+----------------------------+
| id | select_type       | table      | type | key          | Extra                      |
+----+-------------------+------------+------+--------------+----------------------------+
|  1 | PRIMARY           | e          | ref  | ix_firstname | Rematerialize (<derived2>) |
|  1 | PRIMARY           | <derived2> | ref  | <auto_key0>  | NULL                       |
|  2 | DEPENDENT DERIVED | s          | ref  | PRIMARY      | Using filesort             |
+----+-------------------+------------+------+--------------+----------------------------+
```

#### 10.3.12.18 Select tables optimized away

`MIN()` 또는 `MAX()`가 SELECT 절에 사용되거나 GROUP BY에서 `MIN()`, `MAX()`를 조회하는 쿼리가 **인덱스를 오름차순 또는 내림차순으로 1건만 읽는 형태의 최적화**가 적용될 때 표시된다.

```sql
mysql> EXPLAIN
SELECT MAX(emp_no), MIN(emp_no) FROM employees;

mysql> EXPLAIN
SELECT MAX(from_date), MIN(from_date) FROM salaries WHERE emp_no=10002;

-- Extra: Select tables optimized away
-- table: NULL (인덱스에서 최솟값/최댓값 1건씩만 읽어도 되므로)
```

#### 10.3.12.19 Start temporary, End temporary

세미 조인 최적화 중에서 **Duplicate Weed-out 최적화 전략**이 사용되면 표시된다. 불필요한 중복 레코드를 제거하기 위해 내부 임시 테이블을 사용하는데, 조인의 첫 번째 테이블에 "Start temporary", 조인이 끝나는 부분에 "End temporary"가 표시된다.

```sql
mysql> EXPLAIN
SELECT * FROM employees e
WHERE e.emp_no IN (SELECT s.emp_no FROM salaries s WHERE s.salary>150000);

+----+-------------+-------+-------+-----------+-----------------------------------------------+
| id | select_type | table | type  | key       | Extra                                         |
+----+-------------+-------+-------+-----------+-----------------------------------------------+
|  1 | SIMPLE      | s     | range | ix_salary | Using where; Using index; Start temporary     |
|  1 | SIMPLE      | e     | eq_ref| PRIMARY   | End temporary                                 |
+----+-------------+-------+-------+-----------+-----------------------------------------------+
```

`salaries` 테이블부터 `employees` 테이블까지의 내용을 임시 테이블에 저장한다는 의미다.

#### 10.3.12.20 unique row not found

두 개의 테이블이 각각 유니크(프라이머리 키 포함) 칼럼으로 아우터 조인을 수행하는 쿼리에서 **아우터 테이블에 일치하는 레코드가 존재하지 않을 때** 표시된다.

```sql
-- tb_test1에 fdpk=2가 있지만 tb_test2에는 없을 때
+----+-------------+-------+-------+---------+------+----------------------+
| id | select_type | table | type  | key     | rows | Extra                |
+----+-------------+-------+-------+---------+------+----------------------+
|  1 | SIMPLE      | t1    | const | PRIMARY |    1 | Using index          |
|  1 | SIMPLE      | t2    | const | PRIMARY |    0 | unique row not found |
+----+-------------+-------+-------+---------+------+----------------------+
```

#### 10.3.12.21 Using filesort

`ORDER BY` 처리가 인덱스를 사용하지 못할 때 표시된다. 조회된 레코드를 **정렬용 메모리 버퍼에 복사해 퀵 소트 또는 힙 소트 알고리즘을 이용해 정렬**하게 된다는 의미다.

```sql
mysql> EXPLAIN
SELECT * FROM employees ORDER BY last_name DESC;
-- type: ALL, Extra: Using filesort (last_name에 인덱스 없음)
```

실행 계획의 Extra 칼럼에 "Using filesort"가 출력되는 쿼리는 많은 부하를 일으키므로 가능하다면 쿼리를 튜닝하거나 인덱스를 생성하는 것이 좋다.

#### 10.3.12.22 Using index (커버링 인덱스)

**데이터 파일을 전혀 읽지 않고 인덱스만 읽어서 쿼리를 모두 처리**할 수 있을 때 표시된다. 이렇게 인덱스만으로 처리되는 것을 **"커버링 인덱스(Covering Index)"** 라고 한다.

```sql
-- 인덱스 없이 데이터 파일도 읽어야 하는 경우
mysql> EXPLAIN
SELECT first_name, birth_date FROM employees WHERE first_name BETWEEN 'Babette' AND 'Gad';
-- type: ALL (5만여 건의 데이터 파일 페이지를 읽어야 하므로 풀 테이블 스캔이 더 효율적)

-- 커버링 인덱스로 처리되는 경우
mysql> EXPLAIN
SELECT first_name FROM employees WHERE first_name BETWEEN 'Babette' AND 'Gad';
-- type: range, Extra: Using where; Using index (인덱스만으로 처리 가능, 30~40페이지만 읽음)
```

InnoDB 테이블은 클러스터링 인덱스로 구성되어, 세컨더리 인덱스에 프라이머리 키 값이 저장되므로 커버링 인덱스로 처리될 가능성이 상당히 높다.

```sql
-- emp_no(PK)가 인덱스에 포함되어 있으므로 커버링 인덱스
mysql> SELECT emp_no, first_name FROM employees WHERE first_name BETWEEN 'Babette' AND 'Gad';
-- Extra: Using index
```

> **주의**: Extra 칼럼의 "Using index"(커버링 인덱스)와 접근 방법(type 칼럼)의 "index"(인덱스 풀 스캔)는 성능상 반대되는 개념이다. 반드시 구분해서 이해해야 한다.

#### 10.3.12.23 Using index condition

MySQL 옵티마이저가 **인덱스 컨디션 푸시 다운(Index Condition Pushdown)** 최적화를 사용하면 표시된다.

```sql
mysql> SELECT * FROM employees WHERE last_name='Acton' AND first_name LIKE '%sal';
-- Extra: Using index condition
```

#### 10.3.12.24 Using index for group-by

`GROUP BY` 처리가 인덱스를 이용할 때 표시된다. 인덱스의 필요한 부분만 듬성듬성 읽는 **루스 인덱스 스캔(Loose Index Scan)** 방식으로 처리될 때만 표시된다.

**타이트 인덱스 스캔**: `GROUP BY` 처리를 위해 인덱스를 사용하지만 모든 인덱스를 다 읽어야 하는 경우 (`COUNT()`, `SUM()`, `AVG()` 등). 이 경우 "Using index for group-by"가 표시되지 않는다.

**루스 인덱스 스캔**: `MIN()`, `MAX()` 같이 인덱스의 첫 번째 또는 마지막 레코드만 읽어도 되는 경우.

```sql
mysql> EXPLAIN
SELECT emp_no, MIN(from_date), MAX(from_date) FROM salaries GROUP BY emp_no;
-- Extra: Using index for group-by (루스 인덱스 스캔)

mysql> EXPLAIN
SELECT first_name, COUNT(*) AS counter FROM employees GROUP BY first_name;
-- Extra: Using index (타이트 인덱스 스캔, COUNT이므로 모든 레코드 읽어야 함)
```

루스 인덱스 스캔 사용 조건:
- `WHERE` 절 조건이 없는 경우: `GROUP BY` 절의 칼럼과 SELECT 칼럼이 루스 인덱스 스캔 조건만 갖추면 됨
- `WHERE` 절이 있지만 인덱스 사용 못하는 경우: 타이트 인덱스 스캔으로 처리
- `WHERE` 절이 있고 인덱스 사용하는 경우: `WHERE` 절과 `GROUP BY` 절이 **동일한 인덱스**를 공통으로 사용할 때만 루스 인덱스 스캔 가능

#### 10.3.12.25 Using index for skip scan

MySQL 옵티마이저가 **인덱스 스킵 스캔 최적화**를 사용하면 표시된다.

```sql
mysql> EXPLAIN
SELECT gender, birth_date FROM employees WHERE birth_date>='1965-02-01';
-- Extra: Using where; Using index for skip scan
-- (gender+birth_date 복합 인덱스에서 첫 번째 칼럼인 gender 없이도 birth_date 조건으로 최적화)
```

MySQL 8.0에서 루스 인덱스 스캔 최적화를 확장한 인덱스 스킵 스캔 최적화가 도입됐다.

#### 10.3.12.26 Using join buffer(Block Nested Loop / Batched Key Access / hash join)

조인에서 드리븐 테이블의 조인 칼럼에 적절한 인덱스가 없어서 **블록 네스티드 루프 조인이나 해시 조인을 사용**하면 표시된다. 조인 버퍼의 크기는 `join_buffer_size` 시스템 변수로 설정할 수 있다.

```sql
mysql> EXPLAIN
SELECT *
FROM dept_emp de, employees e
WHERE de.from_date>'2005-01-01' AND e.emp_no<10904;

+----+-------------+-------+-------+-------------+------------------------------------------+
| id | select_type | table | type  | key         | Extra                                    |
+----+-------------+-------+-------+-------------+------------------------------------------+
|  1 | SIMPLE      | de    | range | ix_fromdate | Using index condition                    |
|  1 | SIMPLE      | e     | range | PRIMARY     | Using where; Using join buffer (hash join)|
+----+-------------+-------+-------+-------------+------------------------------------------+
```

MySQL 5.6 버전부터는 "Batched Key Access"나 "Hash join"이 도입되면서 `Using join buffer` 문구에 조인 알고리즘이 추가로 표시된다.

#### 10.3.12.27 Using MRR

MySQL 서버는 **MRR(Multi Range Read)** 이라는 최적화를 도입했다. MySQL 엔진은 여러 개의 키 값을 한 번에 스토리지 엔진으로 전달하고, 스토리지 엔진은 넘겨받은 키 값을 정렬해서 **최소한의 페이지 접근만으로 필요한 레코드를 읽을 수 있게 최적화**한다.

MRR이 사용되면 Extra 칼럼에 "Using MRR"이 표시된다.

#### 10.3.12.28 Using sort_union(...) / Using union(...) / Using intersect(...)

`index_merge` 접근 방법으로 실행될 때, 두 인덱스로부터 읽은 결과를 어떻게 병합했는지 설명하기 위해 표시된다.

- **`Using intersect(...)`**: 각 인덱스를 사용할 수 있는 조건이 `AND`로 연결된 경우 → 교집합 추출
- **`Using union(...)`**: 각 인덱스를 사용할 수 있는 조건이 `OR`로 연결된 경우 → 합집합 추출
- **`Using sort_union(...)`**: `Using union`과 같은 작업이지만, 처리될 수 없는 경우(대량의 range 조건들) 사용. 프라이머리 키만 먼저 읽어서 정렬하고 병합한 이후 비로소 레코드를 읽어서 반환한다.

`Using union()`은 대체로 동등 비교처럼 일치하는 레코드 건수가 많지 않은 경우 사용되고, 각 조건이 범위 비교처럼 상대적으로 많은 레코드에 일치하면 `Using sort_union()`이 사용된다. 실제로는 WHERE 조건에 사용된 비교 조건이 모두 동등 조건이면 `Using union()`이, 그렇지 않으면 `Using sort_union()`이 사용된다.

#### 10.3.12.29 Using temporary

쿼리를 처리하는 동안 중간 결과를 담아 두기 위해 **임시 테이블(Temporary table)을 사용**했다는 것을 의미한다. 임시 테이블은 메모리 또는 디스크에 생성될 수 있다.

```sql
mysql> EXPLAIN
SELECT * FROM employees
GROUP BY gender
ORDER BY MIN(emp_no);
-- Extra: Using temporary; Using filesort (GROUP BY 칼럼과 ORDER BY 칼럼이 다름)
```

> **주의**: Extra 칼럼에 "Using temporary"가 표시되지 않더라도 내부적으로 임시 테이블을 사용하는 경우가 많다. 대표적으로 다음과 같은 경우 임시 테이블이 사용된다:
> - FROM 절에 사용된 서브쿼리(파생 테이블)
> - `COUNT(DISTINCT column1)`을 포함하는 쿼리(인덱스 사용 불가 시)
> - `UNION` 또는 `UNION DISTINCT` 쿼리 (MySQL 8.0부터 UNION ALL은 임시 테이블 미사용)
> - 인덱스를 사용하지 못하는 정렬 작업

임시 테이블이나 버퍼가 메모리에 저장됐는지, 디스크에 저장됐는지는 MySQL 서버의 상태 변수로 확인할 수 있다:

```sql
mysql> FLUSH STATUS;
mysql> -- 쿼리 실행
mysql> SHOW STATUS LIKE 'Created_tmp%';
-- Created_tmp_tables: 내부 임시 테이블 생성 횟수
-- Created_tmp_disk_tables: 디스크 임시 테이블 생성 횟수
```

#### 10.3.12.30 Using where

MySQL 엔진 레이어에서 별도의 가공을 해서 **필터링(여과) 작업을 처리한 경우**에만 표시된다. 각 스토리지 엔진에서 레코드를 읽어 MySQL 엔진에 넘겼지만, MySQL 엔진이 그 중 일부를 필터링해서 버렸다는 의미다.

```sql
mysql> EXPLAIN
SELECT *
FROM employees
WHERE emp_no BETWEEN 10001 AND 10100
AND gender='F';
-- rows: 100 (스토리지 엔진이 100건 반환)
-- filtered: 37% (MySQL 엔진이 100건 중 37건만 남기고 63건 버림)
-- Extra: Using where
```

"Using where"는 Extra 칼럼에 가장 흔히 표시되는 내용이다. MySQL 8.0에서는 `filtered` 칼럼이 함께 표시되므로 성능상 이슈가 있는지 쉽게 파악할 수 있다.

#### 10.3.12.31 Zero limit

쿼리의 결괏값의 **메타데이터(칼럼 수, 각 칼럼 타입 등)만 필요한 경우** `LIMIT 0`을 사용하면, MySQL 옵티마이저가 실제 테이블의 레코드는 전혀 읽지 않고 결괏값의 메타 정보만 반환한다. 이때 표시된다.

```sql
mysql> EXPLAIN SELECT * FROM employees LIMIT 0;
-- Extra: Zero limit
```

# 들어가기 전: 개념정리
#### 트랜잭션
+ 작업의 완전성(원자성), 정합성을 보장해주는 기능
+ 논리적인 작업 셋을 모두 완벽하게 처리하거나, 처리하지 못할 경우 원 상태로 복구
#### 잠금
+ 동시성을 제어하기 위한 기능
+ 여러 커넥션에서 동시에 동일한 자원을 요청 시, 한 시점에는 하나의 커넥션만 변경할 수 있도록 제어해줌
#### 격리 수준
+ 하나의 트랜잭션 내 또는 여러 트랜잭션간의 작업 내용을 어떻게 공유하고 차단할 것인지 결정하는 레벨
# 5.1 트랜잭션

## 1. 트랜잭션이란?

트랜잭션(Transaction)은 하나의 논리적인 작업 단위를 의미한다.  
여러 개의 SQL 문이 하나의 작업으로 묶여 **모두 성공하거나, 모두 실패해야 하는 단위**다.

### ✔ 트랜잭션의 목적

- 작업의 **원자성(Atomicity)** 보장
- 데이터의 **정합성(Consistency)** 유지
- 부분 업데이트(Partial Update) 방지
- 장애 발생 시 안전한 복구

---

## 2. MyISAM vs InnoDB 트랜잭션 차이

| 구분 | MyISAM | InnoDB |
|------|--------|--------|
| 트랜잭션 지원 | ❌ 지원 안함 | ✅ 지원 |
| 롤백 가능 여부 | 불가능 | 가능 |
| 장애 발생 시 | 일부만 반영 가능 | 전체 롤백 |
| 데이터 정합성 | 애플리케이션 책임 | 엔진이 보장 |

---

## 3. MySQL에서의 트랜잭션 동작

### Commit

- 트랜잭션 내에서 수행한 변경 내용을 최종 확정
- COMMIT 이전까지는 변경 내용이 완전히 확정된 상태가 아님
- COMMIT 시점에 영구 반영

### Rollback

- 트랜잭션 시작 이전 상태로 되돌림
- 마지막 COMMIT 완료 시점으로 복구

---

## 4. MyISAM과 InnoDB의 실제 동작 차이

### 테스트 환경

```sql
CREATE TABLE tab_myisam (
  fdpk INT NOT NULL,
  PRIMARY KEY (fdpk)
) ENGINE=MyISAM;

CREATE TABLE tab_innodb (
  fdpk INT NOT NULL,
  PRIMARY KEY (fdpk)
) ENGINE=InnoDB;
```

```sql
SET autocommit=ON;

INSERT INTO tab_myisam VALUES (1),(2),(3);
INSERT INTO tab_innodb VALUES (1),(2),(3);
```

중복 키 삽입 시도:

```sql
INSERT INTO tab_myisam VALUES (1),(2),(3);
INSERT INTO tab_innodb VALUES (1),(2),(3);
```

### 결과 차이

#### MyISAM
- 1, 2는 저장됨
- 3에서 PRIMARY KEY 충돌 발생
- 이미 저장된 1, 2는 유지됨
- **부분 반영 발생**

#### InnoDB
- 3에서 에러 발생
- 전체 INSERT 문이 취소됨
- 아무 데이터도 저장되지 않음
- **원자성 보장**

## 5. 트랜잭션이 없는 경우의 문제점

예시:

```pseudo
INSERT INTO tab_a ...

IF 성공:
    INSERT INTO tab_b ...
    IF 실패:
        DELETE FROM tab_a ...
```

문제점:

- 예외 처리 코드 복잡
- 롤백 로직을 직접 구현해야 함
- 예외 케이스 누락 가능성
- 유지보수 어려움
- 데이터 불일치 위험 증가

---

## 6. 트랜잭션을 사용하는 경우

```sql
START TRANSACTION;

INSERT INTO tab_a ...
INSERT INTO tab_b ...

COMMIT;
```

예외 발생 시:

```sql
ROLLBACK;
```

장점:

- 비즈니스 로직 단순화
- 예외 처리 명확
- 데이터 정합성 보장
- 코드 안정성 향상

---

## 7. 트랜잭션 설계 시 주의사항

### 1) 트랜잭션은 최소 범위로 사용

트랜잭션은 필요한 부분에만 적용해야 한다.

잘못된 예:

1. 사용자 로그인 확인
2. 파일 업로드
3. 이메일 전송
4. 외부 API 호출
5. DB 저장
6. COMMIT

→ 네트워크 작업, 파일 작업은 트랜잭션에 포함하면 안 된다.

---

### 2) 외부 작업은 트랜잭션 밖에서 수행

- 파일 업로드
- 이메일 전송
- FTP 전송
- 외부 API 호출

이 작업들은 DB 트랜잭션과 무관하므로 포함시키면 안 된다.

---

### 3) 트랜잭션은 짧게 유지

이유:

- 잠금(Lock) 유지 시간 증가
- 동시성 저하
- 데드락 위험 증가
- 커넥션 점유 시간 증가

---

## 8. 올바른 트랜잭션 처리 예시

```pseudo
1. 사용자 입력 검증
2. 파일 업로드 처리
3. START TRANSACTION
4. DB 데이터 저장
5. COMMIT
6. 이메일 전송
```

핵심 원칙:

> 트랜잭션은 "DB 변경 작업"에만 적용한다.

---

## 9. 정리

트랜잭션은 단순한 기능이 아니라  
DBMS가 제공하는 가장 중요한 안정성 장치다.

- MyISAM은 부분 반영 가능
- InnoDB는 원자성 보장
- 트랜잭션은 짧게, 최소 범위로
- 외부 작업은 트랜잭션 밖에서

트랜잭션을 올바르게 설계하는 것은  
단순히 SQL 문법을 아는 문제가 아니라  
시스템 안정성을 설계하는 문제다.

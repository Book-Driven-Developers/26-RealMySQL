# 5.2 MySQL 엔진의 잠금

MySQL에서 사용하는 잠금은 크게 두 계층으로 나뉜다.

1. MySQL 엔진 레벨 잠금
2. 스토리지 엔진(InnoDB 등) 레벨 잠금

5.2에서는 MySQL 엔진 자체에서 제공하는 잠금을 다룬다.  
이는 스토리지 엔진과 무관하게 동작하는 잠금이다.

---

## 5.2.1 글로벌 락 (Global Lock)

글로벌 락은 MySQL 서버 전체에 영향을 미치는 잠금이다.

### 획득 방법

```sql
FLUSH TABLES WITH READ LOCK;
```

이 명령이 실행되면:

- 현재 열려 있는 모든 테이블을 닫고
- 모든 테이블에 읽기 잠금을 설정한다.
> InnoDB는 트랜잭션을 지원하기 때문에 백업할 때 모든 쓰기 작업을 멈출 필요는 없다.
> 
> why? 
> 
> InnoDB는 MVCC + Undo 로그 덕분에
백업 도중에도 일관된 스냅샷을 만들 수 있다.
> 
> 즉,
백업 중에 INSERT/UPDATE가 일어나도
트랜잭션 기반으로 “백업 시작 시점의 데이터”를 읽을 수 있다
그래서 굳이 모든 작업을 멈출 필요는 없다.

### 특징

- 다른 세션에서 데이터 변경 불가
- DML, 대부분의 DDL 실행 불가
- SELECT는 가능

즉, 서버 전체가 "읽기 전용" 상태가 된다.

### 사용 목적

- 전체 데이터 백업 시 일관성 확보
- 복제(Replication) 환경에서 스냅샷 생성

### 문제점

- 서버 전체 쓰기 작업 중단
- 트래픽이 많은 서비스에서는 사용 어려움
- 잠금 유지 시간 동안 서비스 영향 큼

---

### MySQL 8.0의 개선: LOCK INSTANCE FOR BACKUP

기존 글로벌 락의 단점을 보완하기 위해 도입됨.

```sql
LOCK INSTANCE FOR BACKUP;
-- 백업 수행
UNLOCK INSTANCE;
```

특징:

- DML은 허용
- 파일 생성/삭제/변경과 같은 위험 작업 차단
- 백업 도구와 함께 사용

글로벌 락보다 서비스 영향이 적다.

---

## 5.2.2 테이블 락 (Table Lock)

테이블 단위로 설정되는 잠금이다.

### 명시적 획득

```sql
LOCK TABLES table_name READ;
LOCK TABLES table_name WRITE;
```

### 해제

```sql
UNLOCK TABLES;
```

### 특징

- MyISAM에서 주로 사용
- InnoDB에서도 사용 가능
- WRITE 락 → 다른 세션 읽기/쓰기 모두 불가
- READ 락 → 다른 세션 쓰기 불가

---

### 묵시적 테이블 락

MyISAM에서는 UPDATE, DELETE 시 자동으로 테이블 락이 설정된다.

반면 InnoDB는:

- 대부분 레코드 단위 잠금 사용
- 테이블 락은 DDL 시 주로 사용

---

## 5.2.3 네임드 락 (Named Lock)

임의의 문자열에 대해 잠금을 설정할 수 있다.

### 획득

```sql
SELECT GET_LOCK('mylock', 10);
```

- 'mylock'이라는 이름의 잠금 요청
- 10초 대기 후 실패 시 NULL 반환

### 해제

```sql
SELECT RELEASE_LOCK('mylock');
```

### 확인

```sql
SELECT IS_FREE_LOCK('mylock');
```

### 특징

- 테이블과 무관
- 사용자 정의 잠금
- 서버 단위 잠금
- 동일 이름에 대해 한 세션만 획득 가능

### 활용 예

- 배치 프로그램 중복 실행 방지
- 특정 작업을 단일 세션만 수행하도록 제어

---

## 5.2.4 메타데이터 락 (Metadata Lock, MDL)

테이블 구조 변경과 관련된 잠금이다.

테이블의 이름, 구조, 인덱스 등 메타데이터를 보호하기 위해 자동으로 설정된다.

예:

```sql
RENAME TABLE tab_a TO tab_b;
```

이 명령 실행 시:

- tab_a와 tab_b에 메타데이터 락 설정
- 다른 세션에서 해당 테이블 접근 대기

---

### 메타데이터 락이 발생하는 경우

- RENAME TABLE
- ALTER TABLE
- DROP TABLE
- TRUNCATE TABLE

### 특징

- 명시적으로 획득하지 않음
- MySQL이 자동 관리
- DDL과 DML 간 충돌 가능

---

### 주의할 점

트랜잭션이 열려 있는 상태에서 DDL을 수행하면:

- 해당 테이블에 메타데이터 락이 유지됨
- 다른 세션에서 쿼리 대기
- 서비스 장애처럼 보일 수 있음

운영 환경에서 ALTER TABLE 수행 시  
예상치 못한 대기 현상이 발생하는 이유가 바로 MDL 때문이다.

---

## 5.2 정리

MySQL 엔진 레벨 잠금은 다음과 같이 구분된다.

| 종류 | 범위 | 사용 목적 |
|------|------|------------|
| 글로벌 락 | 서버 전체 | 백업, 복제 |
| 테이블 락 | 테이블 단위 | MyISAM, 명시적 제어 |
| 네임드 락 | 사용자 정의 | 배치 제어 |
| 메타데이터 락 | 테이블 구조 | DDL 보호 |

핵심 정리:

- 글로벌 락은 서비스 전체에 영향
- 테이블 락은 MyISAM 중심
- 네임드 락은 애플리케이션 제어용
- 메타데이터 락은 DDL 시 자동 발생

MySQL 잠금은 단순히 동시성 제어가 아니라  
서버 레벨 안정성과 운영 전략에 직접적인 영향을 미친다.

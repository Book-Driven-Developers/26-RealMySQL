# 딥다이브 — 격리 수준 실무 선택 기준

> 실무에서 트랜잭션 격리 수준을 고민할 때, READ COMMITTED와 REPEATABLE READ 중 어떤 기준으로 선택하는가?
> 그리고 READ UNCOMMITTED, SERIALIZABLE은 왜 실무에서 거의 선택받지 못하는가?

---

## READ UNCOMMITTED와 SERIALIZABLE은 왜 선택받지 못하는가

### READ UNCOMMITTED — Dirty Read

커밋되지 않은 데이터를 읽는다는 것이 실제로 얼마나 위험한지 다음 시나리오로 확인할 수 있다.

```
[트랜잭션 A] 계좌 잔액: 100만원 → 10만원으로 변경 중...
[트랜잭션 B] 잔액 조회 → 10만원 읽음 → 대출 한도 계산
[트랜잭션 A] ROLLBACK → 실제 잔액은 여전히 100만원
```

B는 존재한 적도 없는 데이터를 기반으로 비즈니스 로직을 실행해버린 것이다.
금융, 재고처럼 정합성이 중요한 도메인에서는 치명적인 버그로 이어진다.

### SERIALIZABLE — SELECT에도 공유 잠금

단순히 "느리다"를 넘어서, **SELECT 쿼리조차 공유 잠금을 획득**해야 한다.

```
[트랜잭션 A] SELECT * FROM orders → 공유 잠금 획득
[트랜잭션 B] INSERT INTO orders → A가 끝날 때까지 대기...
[트랜잭션 C] SELECT * FROM orders → A, B가 끝날 때까지 대기...
```

동시 요청이 많은 웹 서비스에서 커넥션이 줄줄이 대기 상태로 쌓이면서 장애로 이어진다.
실질적으로 동시성을 포기하는 수준이다.

> InnoDB는 갭 락과 넥스트 키 락 덕분에 REPEATABLE READ에서도 PHANTOM READ가 발생하지 않기 때문에, SERIALIZABLE을 사용할 이유가 더욱 없다.

---

## READ COMMITTED vs REPEATABLE READ — 가장 큰 차이

| | READ COMMITTED | REPEATABLE READ |
|---|---|---|
| Read View 생성 시점 | **쿼리(SQL문)마다** 새로 생성 | **트랜잭션 시작 시** 한 번만 생성 |
| 같은 쿼리를 두 번 실행하면 | 결과가 달라질 수 있음 | 항상 같은 결과 |
| 대표 부정합 | Non-Repeatable Read | (일반 SELECT에서는 거의 없음) |

### 시나리오로 확인하기

```sql
-- [트랜잭션 A] 시작
BEGIN;
SELECT balance FROM account WHERE id = 1;  -- ① 100만원 조회

    -- 이 사이에 [트랜잭션 B]가 실행되고 COMMIT
    -- UPDATE account SET balance = 50만원 WHERE id = 1;

SELECT balance FROM account WHERE id = 1;  -- ② 두 번째 조회
COMMIT;
```

- **READ COMMITTED** → ②에서 **50만원** (B가 커밋했으니 최신값 반영)
- **REPEATABLE READ** → ②에서 **100만원** (A 시작 시점 스냅샷 유지)

READ COMMITTED에서 발생하는 이 현상이 바로 **Non-Repeatable Read** 이다.

---

## MVCC Read View의 동작 원리 — 단순 번호 비교가 아니다

REPEATABLE READ의 스냅샷은 "내 TRX 번호보다 작으면 무조건 보인다"가 아니다.
트랜잭션 시작 시 **Read View** 를 생성하며, 이 안에는 다음 정보가 담긴다.

```
[B의 Read View (TRX #10 시작 시점)]
- 현재 활성(미커밋) 트랜잭션 목록: [9, 10, ...]
- 이 목록에 있는 트랜잭션의 변경은 이후 커밋되어도 보이지 않는다
```

### 핵심은 "A가 언제 커밋했느냐"

**Case 1. A(TRX #9)가 B 시작 전에 이미 커밋한 경우**

```
TRX #9 (A):  BEGIN → UPDATE → COMMIT
                                       ↓
TRX #10 (B):                        BEGIN → SELECT → SELECT
```

B의 Read View 생성 시점에 TRX #9는 이미 커밋 완료 → 활성 목록에 없음 → **"Toto"가 보인다**

---

**Case 2. A(TRX #9)가 B 시작 이후에 커밋한 경우**

```
TRX #9 (A):  BEGIN → UPDATE → (미커밋)... → COMMIT
TRX #10 (B):          BEGIN → SELECT → SELECT
```

B의 Read View 생성 시점에 TRX #9는 **활성 상태** → 활성 목록에 포함
→ A가 이후에 커밋해도 B는 끝까지 undo log에서 **"Lara"** 를 읽는다

> **결론**: Read View 생성 시점에 활성 상태였던 트랜잭션의 변경은, 이후 커밋되어도 보이지 않는다.
> "B가 undo에서 한 번, record에서 한 번 읽는" 상황은 REPEATABLE READ에서 발생하지 않는다.

## 실무 선택 기준

> **"트랜잭션 안에서 데이터 일관성이 필요한가"** vs **"항상 최신 데이터가 필요한가"**

| 상황 | 격리 수준 | 이유 |
|---|---|---|
| 정산 배치 (조회 → 건별 합산 → 최종 합계) | **REPEATABLE READ** | 트랜잭션 내내 동일한 스냅샷 기준으로 계산해야 함 |
| SNS 좋아요 수 실시간 표시 | **READ COMMITTED** | 항상 최신 커밋값이 필요, 높은 동시성 요구 |

### REPEATABLE READ가 필요한 신호
- 트랜잭션 안에서 **조회 → 가공 → 다시 조회** 흐름이 있을 때
- 중간에 다른 트랜잭션이 커밋해도 내 계산 기준이 흔들리면 안 될 때
- 예: 정산 배치, 재무 계산, 긴 트랜잭션

### READ COMMITTED가 맞는 신호
- 트랜잭션이 짧고, **항상 가장 최신의 커밋된 값**이 중요할 때
- 같은 쿼리를 트랜잭션 안에서 두 번 실행할 일이 없을 때
- 예: 단건 조회/수정, 실시간 피드, 높은 동시성이 요구되는 OLTP

# 5.3 딥다이브 - 인덱스와 잠금

> **주제**: InnoDB에서 클러스터링 인덱스와 보조 인덱스의 잠금 방식 차이, 갭 락(Gap Lock)의 존재 이유, 그리고 UNIQUE 여부에 따른 잠금 전략의 차이를 탐구한다.

---

## 클러스터링 인덱스와 보조 인덱스의 구조

### 클러스터링 인덱스 (Clustered Index)

InnoDB의 클러스터링 인덱스는 B-Tree의 **리프 노드(Leaf Node)** 에 실제 레코드 데이터가 통째로 저장된다.
즉, 클러스터링 인덱스 자체가 곧 테이블 데이터다.

```
클러스터링 인덱스 B-Tree
         [PK: emp_no]
        /      |      \
    [100]    [200]    [300]
      ↓        ↓        ↓
  실제 레코드  실제 레코드  실제 레코드  ← 리프 노드 = 데이터 그 자체
 (name, age, ...)
```

### 보조 인덱스 (Secondary Index)

보조 인덱스의 리프 노드에는 **인덱스 컬럼 값 + PK 참조값**이 담겨 있다.
실제 레코드를 찾으려면 이 PK를 통해 다시 클러스터링 인덱스를 탐색하는 **2단계 탐색**이 발생한다.

```
보조 인덱스 B-Tree
      [first_name]
      /    |    \
[Georgi] [Kim] [Lee]
    ↓
  (first_name='Georgi', PK=10)  ← 리프 노드 = 인덱스값 + PK 참조값
  (first_name='Georgi', PK=20)
  ...
         ↓ PK로 다시 클러스터링 인덱스 탐색
       실제 레코드
```

## 클러스터링 인덱스 vs 보조 인덱스의 잠금 차이

InnoDB의 잠금은 레코드 자체가 아닌 **인덱스를 잠그는 방식**으로 동작한다.
이때 어떤 인덱스를 사용하느냐에 따라 잠금 전략이 달라진다.

### PK (클러스터링 인덱스)로 조회 시

```sql
UPDATE employees SET hire_date=NOW() WHERE emp_no=100001;
```

PK는 **유일(Unique)** 하므로 정확히 1건만 특정된다.
→ **레코드 락(Record Lock)** 만 사용, 갭 락(Gap Lock) 없음

```
클러스터링 인덱스에서 emp_no=100001 직접 탐색
→ 해당 레코드 1건에만 레코드 락
→ 갭 락 없음
```

### 보조 인덱스로 조회 시

```sql
-- ix_firstname: KEY (first_name) -- 단일 보조 인덱스
UPDATE employees SET hire_date=NOW() WHERE first_name='Georgi' AND last_name='Klassen';
```

`last_name`은 인덱스에 없으므로, 인덱스로 필터링 가능한 조건은 `first_name='Georgi'`뿐이다.
→ `first_name='Georgi'`에 해당하는 **253건 모두 잠금** 대상이 된다.

```
1단계: 보조 인덱스(ix_firstname) 탐색
       first_name='Georgi' → 253건 매칭
       → 253건 모두 넥스트 키 락(레코드 락 + 갭 락) 적용

2단계: 각각의 PK로 클러스터링 인덱스 접근
       → 253건의 실제 레코드에도 레코드 락
```

보조 인덱스 조회 시 잠금은 **두 군데**에 걸린다.
- 보조 인덱스 항목: 넥스트 키 락 (레코드 락 + 갭 락)
- 클러스터링 인덱스 (실제 레코드): 레코드 락

## 갭 락이 필요한 이유 - 팬텀 리드(Phantom Read)

### 팬텀 리드란?

같은 트랜잭션 안에서 **같은 조건으로 두 번 읽었는데**, 처음엔 없던 레코드가 나타나는 현상이다.

```
트랜잭션 A                              트랜잭션 B
─────────────────────────────────────────────────────
BEGIN;
SELECT * FROM employees
WHERE first_name='Georgi';
→ 253건 조회됨

                                        BEGIN;
                                        INSERT INTO employees
                                        (first_name, ...) VALUES ('Georgi', ...);
                                        COMMIT;

SELECT * FROM employees
WHERE first_name='Georgi';
→ 254건 조회됨  ← 유령(Phantom)처럼 1건이 나타남
```

### 레코드 락만으로는 팬텀 리드를 막을 수 없다

레코드 락은 **이미 존재하는 레코드**에만 걸린다.
기존 레코드를 수정·삭제하는 것은 막을 수 있지만, **새로운 레코드가 끼어드는 것**은 막지 못한다.

```
[Georgi, PK=10] ← 레코드 락
[Georgi, PK=20] ← 레코드 락
[Georgi, PK=30] ← 레코드 락
      ↑
   253건 모두 레코드 락을 걸어도...

[Georgi, PK=99] ← 새로운 INSERT는 막을 수 없음!
```

### 갭 락이 해결하는 방식

갭 락은 레코드 **사이의 공간(gap)** 을 잠가 새로운 레코드의 INSERT를 차단한다.

```
보조 인덱스 (first_name 정렬 순서)

...[Georgi, PK=10]...[Georgi, PK=20]...[Georgi, PK=30]...
          ↑                   ↑                  ↑
       갭 락               갭 락              갭 락
  (이 사이에 INSERT 불가)
```

트랜잭션 B가 `first_name='Georgi'`인 레코드를 INSERT하려 하면,
인덱스 상 Georgi 항목들 사이 어딘가에 삽입되어야 하는데 → **갭 락에 막혀 대기**하게 된다.

## 복합 인덱스에서의 잠금 동작

보조 인덱스가 `(first_name, last_name)` 복합 인덱스로 설계되어 있을 때 잠금은 어떻게 달라질까?

### 단일 인덱스 vs 복합 인덱스 구조 비교

```
단일 인덱스 (first_name만)          복합 인덱스 (first_name, last_name)

[Georgi, PK=10]                    [Georgi, Bader,   PK=10]
[Georgi, PK=20]      →             [Georgi, Klassen, PK=99]  ← 1건 특정 가능
[Georgi, PK=30]                    [Georgi, Smith,   PK=20]
... (253건)
```

복합 인덱스에서는 `WHERE first_name='Georgi' AND last_name='Klassen'` 조건을 **두 컬럼 모두 사용**해서 탐색하므로, 인덱스에서 매칭되는 건은 1건이다.
→ 잠기는 레코드도 **1건**으로 줄어든다.

### UNIQUE 여부에 따른 갭 락 차이

하지만 갭 락 유무는 **인덱스가 유일성을 보장하는가**에 따라 달라진다.

#### 일반 복합 인덱스 `KEY (first_name, last_name)`

인덱스가 고유함을 보장하지 않으므로, `(Georgi, Klassen)` 조합을 가진 직원이 **또 존재할 수 있다.**
InnoDB는 새 레코드가 끼어들 가능성이 있다고 판단해 → **갭 락 사용**

#### UNIQUE 복합 인덱스 `UNIQUE KEY (first_name, last_name)`

`(Georgi, Klassen)` 조합은 테이블에 오직 1건만 존재 가능하다.
동일한 조합의 INSERT 시도는 MySQL이 중복 에러로 차단하므로, 팬텀 리드 가능성이 없다.
→ **레코드 락만 사용, 갭 락 없음** (PK와 동일한 방식)

### 잠금 방식 전체 비교

| 인덱스 종류 | 매칭 건수 | 레코드 락 | 갭 락 |
|---|---|---|---|
| 단일 인덱스 `KEY (first_name)` | 253건 | 253건 | 있음 |
| 복합 인덱스 `KEY (first_name, last_name)` | 1건 | 1건 | **있음** |
| 복합 인덱스 `UNIQUE KEY (first_name, last_name)` | 1건 | 1건 | **없음** |
| PK (emp_no) | 1건 | 1건 | 없음 |

**갭 락 유무의 기준은 컬럼 개수가 아니라, 해당 인덱스가 유일성을 보장하느냐**이다.
InnoDB 입장에서는 "이 인덱스로 찾은 결과에 새 레코드가 끼어들 가능성이 있는가?"를 기준으로 판단한다.

---

## 핵심 요약

- InnoDB의 잠금은 레코드가 아닌 **인덱스를 잠그는 방식**으로 동작한다.
- **보조 인덱스 조회** 시 잠금은 두 군데에 걸린다: 보조 인덱스 항목 + 클러스터링 인덱스(실제 레코드)
- **갭 락**은 "아직 존재하지 않는 레코드"의 INSERT를 막아 팬텀 리드를 방지한다.
- **PK 또는 UNIQUE 인덱스**의 정확한 단건 조회: 레코드 락만 사용 (갭 락 없음)
- **non-UNIQUE 보조 인덱스**: 항상 넥스트 키 락 (레코드 락 + 갭 락) 사용
- 인덱스 설계가 잘못되면 불필요하게 많은 레코드가 잠겨 **동시성이 크게 저하**될 수 있다.

# 8-8 딥다이브 - 클러스터링 인덱스와 세컨더리 인덱스 탐색 과정

> 클러스터링 인덱스와 세컨더리 인덱스는 논리적 개념인가, 실제 디스크에 존재하는 물리적 구조인가?  
> B-Tree 탐색 과정과 디스크 I/O를 연결하여 각 인덱스의 동작 방식을 깊이 이해한다.

---

## 1. 클러스터링 인덱스 = 실제 디스크에 기록된 물리적 데이터

클러스터링 인덱스는 단순히 논리적인 개념이 아니다. **실제 디스크에 기록된 물리적 저장 구조**이다.

InnoDB의 B-Tree 구조에서 핵심은 **리프 노드**에 있다.

| 인덱스 종류 | 리프 노드 내용 |
|------------|--------------|
| 일반 인덱스 | 실제 레코드를 가리키는 **포인터(주소)** |
| 클러스터링 인덱스 | **실제 레코드 그 자체** |

즉, 클러스터링 인덱스 B-Tree 자체가 테이블의 실 데이터다.

```
클러스터링 인덱스 B-Tree

        [루트 노드]
       /           \
  [브랜치]        [브랜치]
   /    \           /    \
[리프]  [리프]  [리프]  [리프]
  ↓
실제 레코드 데이터 (실물)
```

> Java의 `HashMap`처럼 "키로 접근하는 별도의 자료구조"가 아니라,
> 아예 **데이터 자체가 PK 순서로 정렬되어 `.ibd` 파일에 기록**되어 있는 것이다.

| 스토리지 엔진 | 저장 형태 |
|-------------|----------|
| MyISAM | 데이터 파일 따로, 인덱스 파일 따로 |
| InnoDB | 클러스터링 인덱스 B-Tree 안에 데이터가 들어있음 |

---

## 2. 세컨더리 인덱스도 물리적으로 디스크에 존재한다

세컨더리 인덱스 역시 논리적인 구조가 아니다. **같은 `.ibd` 파일 안에 별도의 B-Tree 구조물로 물리적으로 기록**되어 있다.

```
[ .ibd 파일 ]
│
├── 클러스터링 인덱스 B-Tree  ← 실제 레코드가 리프에 존재
│       리프 노드: { PK=1, name="홍길동", age=30, ... }
│
└── 세컨더리 인덱스 B-Tree    ← 별도로 기록된 인덱스 구조
        리프 노드: { name="홍길동", PK=1 }  ← 포인터 = PK값
```

둘 다 디스크에 쓰여있지만, 세컨더리 인덱스는 **"실제 레코드를 찾기 위해 별도로 만들어진 구조물"** 이라는 점이 다르다.

---

## 3. B-Tree 탐색 과정과 디스크 I/O

### 전제: InnoDB의 I/O 단위는 "페이지(Page)"

B-Tree의 노드 하나 = **페이지 하나 (기본 16KB)**

디스크에서 데이터를 읽을 때 레코드 단 하나만 읽는 게 아니라, **페이지 단위로 통째로 Buffer Pool(RAM)에 올린다.**

```
디스크                          Buffer Pool (RAM)
[ Page 1: 루트 ]   →  읽기  →  [ Page 1 캐싱 ]
[ Page 3: 브랜치 ] →  읽기  →  [ Page 3 캐싱 ]
[ Page 7: 리프 ]   →  읽기  →  [ Page 7 캐싱 ]
```

---

### 클러스터링 인덱스 탐색

```sql
SELECT * FROM member WHERE id = 5;  -- id = PK
```

**Step 1. 루트 페이지 접근**
```
Buffer Pool에 루트 페이지 없음
→ 디스크에서 루트 페이지 읽기  (디스크 I/O 1회)
→ Buffer Pool에 캐싱
→ "id=5는 Page3에 있다" 확인
```

**Step 2. 브랜치 페이지 접근**
```
Buffer Pool에 Page3 없음
→ 디스크에서 Page3 읽기  (디스크 I/O 2회)
→ Buffer Pool에 캐싱
→ "id=5는 Page7(리프)에 있다" 확인
```

**Step 3. 리프 페이지 접근**
```
Buffer Pool에 Page7 없음
→ 디스크에서 Page7 읽기  (디스크 I/O 3회)
→ 리프 노드에 실제 레코드가 있음
→ { id=5, name="홍길동", age=30, ... } 반환
```

```
총 디스크 I/O: 루트 + 브랜치 + 리프 = 트리 높이만큼
결과: 레코드 바로 획득
```

---

### 세컨더리 인덱스 탐색 (북마크 룩업 발생)

```sql
SELECT * FROM member WHERE name = '홍길동';  -- name = 세컨더리 인덱스
```

#### Phase 1 - 세컨더리 인덱스 B-Tree 탐색

```
루트 페이지 읽기  (디스크 I/O 1회)
→ 브랜치 페이지 읽기  (디스크 I/O 2회)
→ 리프 페이지 읽기  (디스크 I/O 3회)

리프 노드 내용: { name="홍길동", PK=5 }
                              ↑
                   실제 레코드가 아닌 PK값만 있음
```

#### Phase 2 - 클러스터링 인덱스 재탐색 (북마크 룩업)

```
PK=5 를 가지고 클러스터링 인덱스를 처음부터 다시 탐색

루트 페이지 읽기  (디스크 I/O 4회)
→ 브랜치 페이지 읽기  (디스크 I/O 5회)
→ 리프 페이지 읽기  (디스크 I/O 6회)

→ { id=5, name="홍길동", age=30, ... } 반환
```

```
총 디스크 I/O: 세컨더리(3회) + 클러스터링(3회) = 최대 6회
```

---

### 커버링 인덱스 - 재탐색 없이 끝내기

```sql
SELECT name FROM member WHERE name = '홍길동';
```

```
세컨더리 인덱스 리프 노드: { name="홍길동", PK=5 }
                              ↑
                   SELECT 대상인 name이 여기 이미 있음

→ 클러스터링 인덱스 재탐색 불필요
총 디스크 I/O: 3회만으로 끝
```

**커버링 인덱스(Covering Index)** 란, 쿼리에 필요한 모든 컬럼이 인덱스 안에 포함되어 클러스터링 인덱스 재탐색 없이 처리되는 것을 말한다.

---

## 정리

| 케이스 | 탐색 B-Tree | 디스크 I/O |
|--------|------------|-----------|
| PK로 조회 | 클러스터링 1번 | 트리 높이만큼 |
| 세컨더리로 `SELECT *` | 세컨더리 + 클러스터링 2번 | 트리 높이 × 2 |
| 커버링 인덱스 | 세컨더리 1번 | 트리 높이만큼 |

> 정말 필요한 컬럼에만 생성하고, 가능하면 부분적 유니크 인덱스로 범위를 좁히자.
MySQL에서 동시 INSERT가 많은 테이블은 중복 검사 때문에 병목 유발.
>

## 쓰기 작업마다 발생하는 중복 검사 오버헤드

- 일반 인덱스는 값을 삽입할 때마다 B+Tree에 추가하면 그만이지만, 유니크 인덱스는 삽입과 수정 시마다 기존 값 전체를 조회해 중복 여부를 확인해야 합니다.
- 유니크인덱스의 목적은 데이터 일관성 보장이지 성능 향상이 아니다.

## MySQL InnoDB: 갭 락(Gap Lock)으로 인한 동시성 저하

유니크 인덱스가 있는 테이블에 동시 INSERT가 발생하면 InnoDB는 중복 방지를 위해 갭락을 획득한다.

→ 병목의 원인

| 읽기 유형 | 유니크 인덱스 + 단일 조건 | 유니크 인덱스 + 범위 조건 | 일반 인덱스 |
| --- | --- | --- | --- |
| 일반 SELECT | 락 없음 | 락 없음 | 락 없음 |
| SELECT FOR UPDATE | 레코드 락만(갭 없음) | 갭 락 발생 | 갭 락 발생 |
| SELECT FOR SHARE | S-LOCK | 갭 락 발생 | 갭 락 발생 |

도서관의 책 한 권을 생각하면 된다.

- Shared Lock: 여러 사람이 동시에 책을 읽을 수 있음. 단, 누군가 읽는 중엔 책에 필기 불가
- Exclusive Lock: 한 사람이 책에 필기 중. 다른 사람은 읽는 것도, 필기하는 것도 전부 대기

### JPA에서 S-LOCK 사용법

S-Lock → `PESSIMISTIC_READ`

```java
@Lock(LockModeType.PESSIMISTIC_READ)
@Query("SELECT u FROM User u WHERE u.id = :id")
Optional<User> findByIdWithSLock(@Param("id") Long id);
```

*X-Lock → `PESSIMISTIC_WRITE`

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
Optional<User> findById(Long id);
```

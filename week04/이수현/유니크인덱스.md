# 8.9 유니크 인덱스 (Unique Index)

유니크 인덱스는 **컬럼 값의 중복을 허용하지 않는 인덱스**이다.  
일반 세컨더리 인덱스와 구조적으로 큰 차이는 없지만, **중복 여부 검증 로직이 추가된다는 점**에서 성능 특성이 달라질 수 있다.

MySQL에서는 **NULL 값은 여러 개 허용**되며,  
프라이머리 키는 기본적으로 NULL을 허용하지 않는 **유니크 인덱스의 특별한 형태**라고 볼 수 있다.

<summary><strong>8.9.1 유니크 인덱스와 일반 세컨더리 인덱스 비교</strong></summary>

## 1. 인덱스 읽기 (SELECT)

많은 사람들이 **유니크 인덱스가 더 빠를 것이라고 생각하지만 실제로는 그렇지 않다.**

왜냐하면

- 유니크 인덱스도 **B-Tree 구조**를 사용한다.
- 일반 인덱스와 **탐색 방식이 동일**하다.
- 단지 **결과 레코드가 1개일 가능성이 높을 뿐**이다.

즉

일반 인덱스 검색

1. B-Tree 탐색
2. 조건에 맞는 여러 레코드 반환 가능

유니크 인덱스 검색

1. B-Tree 탐색
2. 조건에 맞는 레코드 1개 반환

하지만 실제 실행 과정에서는

- B-Tree 탐색 비용
- 디스크 I/O
- 버퍼 풀 접근

등이 동일하기 때문에

👉 **유니크 인덱스가 특별히 빠르다고 보기는 어렵다.**

예를 들어

일반 인덱스

- 1번째 레코드 읽기 0.1초
- 2번째 레코드 읽기 0.2초

이라면

유니크 인덱스는

- 1개의 레코드만 읽기 때문에 0.1초 정도에 가까운 성능

정도로 나타날 수 있지만

이는 **유니크 인덱스 구조 때문이 아니라 반환 레코드 수 차이 때문**이다.

---

## 2. 인덱스 쓰기 (INSERT / UPDATE)

유니크 인덱스는 **쓰기 작업에서 일반 인덱스보다 비용이 더 높다.**

그 이유는 다음과 같다.

### 이유 1 — 중복 검사 과정

데이터를 삽입할 때

1. 인덱스에서 동일 값 존재 여부 확인
2. 존재하지 않으면 삽입

즉 **추가적인 검색 과정이 필요하다.**

---

### 이유 2 — Change Buffer 사용 제한

InnoDB에는 **Change Buffer**라는 기능이 있다.

이 기능은

세컨더리 인덱스 변경 작업을  
즉시 디스크에 반영하지 않고  
버퍼에 저장했다가 나중에 처리

하는 방식이다.

하지만

👉 **유니크 인덱스는 Change Buffer를 사용할 수 없다.**

왜냐하면

중복 여부를 **즉시 검증해야 하기 때문**이다.

따라서

일반 인덱스

- Change Buffer 사용 가능
- 쓰기 작업 지연 가능

유니크 인덱스

- Change Buffer 사용 불가
- 즉시 디스크 반영 필요

결과적으로

👉 **쓰기 작업에서는 일반 인덱스보다 느릴 가능성이 높다.**

<details>
<summary><strong>8.9.2 유니크 인덱스 사용 시 주의사항</strong></summary>

유니크 인덱스는 **데이터 무결성을 보장하기 위해 사용**한다.

하지만 다음과 같은 경우에는 **불필요한 성능 저하를 만들 수 있다.**

---

## 1. 꼭 필요한 경우에만 사용

유니크 인덱스는

데이터 중복이 **절대 허용되면 안 되는 컬럼**

에만 사용하는 것이 좋다.

예시

- 이메일
- 회원ID
- 주민번호
- 로그인ID

---

## 2. 유니크 인덱스 + 일반 인덱스 중복 생성 금지

다음과 같은 경우를 종종 볼 수 있다.

```sql
CREATE TABLE tb_unique (
    id INT NOT NULL,
    nick_name VARCHAR(100),
    PRIMARY KEY (id),
    UNIQUE INDEX ux_nickname (nick_name),
    INDEX ix_nickname (nick_name)
);
```

하지만 MySQL에서는

유니크 인덱스 자체가 **일반 인덱스 역할을 수행한다.**

따라서

UNIQUE INDEX + 일반 INDEX

를 동시에 생성할 필요가 없다.

즉

UNIQUE INDEX 하나만 있으면 충분하다.

---

## 3. 불필요한 유니크 인덱스는 성능 저하

유니크 인덱스를 많이 만들면

- INSERT 성능 저하
- UPDATE 성능 저하
- 디스크 사용량 증가

문제가 발생할 수 있다.

따라서

데이터 무결성이 반드시 필요한 컬럼에만  
유니크 인덱스를 생성하는 것이 바람직하다.

</details>

# 8.10 외래키 (Foreign Key)

MySQL에서 외래키는 **InnoDB 스토리지 엔진에서만 지원**된다.  
외래키가 설정되면 부모 테이블과 자식 테이블 간의 **참조 무결성(Referential Integrity)** 을 보장하게 된다.

외래키가 설정되면 InnoDB는 내부적으로 다음과 같은 동작을 수행한다.

- 부모 테이블의 레코드가 변경되거나 삭제될 때 **참조 무결성 검증**
- 자식 테이블의 레코드가 추가될 때 **부모 레코드 존재 여부 확인**

이 과정에서 **잠금(lock) 대기 상황이 발생할 수 있다.**

---

<details>
<summary><strong>8.10.1 자식 테이블의 변경이 대기하는 경우</strong></summary>

다음과 같은 상황을 가정한다.

부모 테이블

```
tb_parent
```

자식 테이블

```
tb_child
```

자식 테이블은 부모 테이블의 id를 외래키로 참조한다.

예시 테이블 구조

```sql
CREATE TABLE tb_parent (
    id INT NOT NULL,
    fd VARCHAR(100),
    PRIMARY KEY (id)
) ENGINE=InnoDB;

CREATE TABLE tb_child (
    id INT NOT NULL,
    pid INT,
    fd VARCHAR(100),
    PRIMARY KEY (id),
    KEY ix_parentid (pid),
    CONSTRAINT child_ibfk_1
        FOREIGN KEY (pid) REFERENCES tb_parent (id)
        ON DELETE CASCADE
) ENGINE=InnoDB;
```

초기 데이터

```sql
INSERT INTO tb_parent VALUES (1,'parent-1'),(2,'parent-2');
INSERT INTO tb_child VALUES (100,1,'child-100');
```

---

### 트랜잭션 상황

|작업|커넥션1|커넥션2|
|---|---|---|
|1|BEGIN| |
|2|UPDATE tb_parent SET fd='changed-2' WHERE id=2;| |
|3| |BEGIN|
|4| |UPDATE tb_child SET pid=2 WHERE id=100;|
|5|ROLLBACK| |
|6| |Query OK|

---

### 동작 설명

1. 커넥션1이 먼저 트랜잭션을 시작하고  
부모 테이블(tb_parent)의 id=2 레코드를 UPDATE한다.

이때 해당 레코드에 **잠금(lock)** 이 걸린다.

2. 커넥션2에서 자식 테이블(tb_child)의 pid 값을 2로 변경하려고 한다.

하지만

```
pid = 2
```

는 부모 테이블의 **id=2 레코드를 참조해야 한다.**

따라서 InnoDB는 다음 검사를 수행한다.

```
부모 테이블에 id=2 레코드가 존재하는가
```

하지만 해당 레코드는 **이미 커넥션1이 잠금을 보유하고 있다.**

그래서

👉 커넥션2의 UPDATE는 **부모 테이블 잠금이 해제될 때까지 대기**한다.

---

### 핵심 포인트

외래키가 존재하면

```
자식 테이블 UPDATE
→ 부모 레코드 존재 여부 검사
→ 부모 레코드 잠금 확인
```

과정이 발생한다.

즉

👉 **자식 테이블의 변경 작업도 부모 테이블 잠금에 영향을 받는다.**

<details>
<summary><strong>8.10.2 부모 테이블의 변경 작업이 대기하는 경우</strong></summary>

이번에는 반대 상황이다.

즉

```
자식 테이블이 먼저 잠금을 잡고
부모 테이블이 변경되는 경우
```

---

### 트랜잭션 상황

|작업|커넥션1|커넥션2|
|---|---|---|
|1|BEGIN| |
|2|UPDATE tb_child SET fd='changed-100' WHERE id=100;| |
|3| |BEGIN|
|4| |DELETE FROM tb_parent WHERE id=1;|
|5|ROLLBACK| |
|6| |Query OK|

---

### 동작 설명

1. 커넥션1이 자식 테이블(tb_child)의 레코드를 UPDATE한다.

이때 해당 자식 레코드에 **잠금이 발생한다.**

2. 커넥션2에서 부모 테이블(tb_parent)의 레코드를 삭제하려고 한다.

하지만

```
tb_child.pid = 1
```

인 레코드가 존재한다.

또한 외래키는

```
ON DELETE CASCADE
```

로 설정되어 있다.

따라서 부모 레코드가 삭제되면

👉 **자식 레코드도 같이 삭제되어야 한다.**

하지만 현재 자식 레코드는 **커넥션1이 잠금 상태**이다.

그래서

👉 커넥션2의 DELETE는 **자식 레코드 잠금이 해제될 때까지 대기**한다.

---

### 핵심 포인트

외래키가 존재하면

```
부모 레코드 삭제
→ 자식 레코드 존재 여부 검사
→ 자식 레코드 잠금 확인
```

이 과정이 필요하다.

따라서

👉 **부모 테이블 작업도 자식 테이블의 잠금 상태에 영향을 받는다.**

---

### 외래키가 성능에 영향을 주는 이유

외래키는 단순한 참조 제약 조건이 아니라

다음과 같은 내부 동작을 수행한다.

```
1. 참조 무결성 검사
2. 부모 레코드 존재 확인
3. 자식 레코드 존재 확인
4. 잠금 확인
```

이 과정 때문에

- 잠금 대기(lock wait)
- 트랜잭션 지연
- 동시성 저하

문제가 발생할 수 있다.

따라서 대규모 트래픽 환경에서는

👉 **외래키 사용 여부를 신중히 결정해야 한다.**

# 8.8 클러스터링 인덱스 (InnoDB)

클러스터링 인덱스(Clustered Index)는 **테이블의 레코드(데이터)가 인덱스 키 순서대로 “정렬된 상태로” 저장**되는 구조를 말한다.  
MySQL에서는 **InnoDB 스토리지 엔진에서만** 클러스터링 인덱스를 지원한다.

---

<details>
<summary><strong>8.8.1 클러스터링 인덱스</strong></summary>

## 핵심 개념
- InnoDB에서 **클러스터링 인덱스의 키 = 기본키(Primary Key)** 이다.
- “기본키 값”을 기준으로 레코드의 **저장 위치(물리적/논리적 배치)** 가 결정된다.
- 결과적으로 InnoDB 테이블은 흔히 **“클러스터링 테이블”** 이라고 부른다.

## “프라이머리 키로 정렬되어 저장”이 의미하는 것
- 테이블의 데이터(레코드)는 **프라이머리 키 순서**에 맞춰 페이지(리프 페이지)에 저장된다.
- 따라서 프라이머리 키로 범위 조회를 하면, **리프 페이지를 순차적으로 읽기** 쉬워진다.

## 프라이머리 키 변경의 영향(중요)
- 클러스터링 테이블에서 **프라이머리 키 값이 변경되면**,  
  해당 레코드는 “원래 위치”에서 빠지고 “새 키 위치”로 **이동**해야 한다.
- 즉, PK 변경은 단순 UPDATE가 아니라 **레코드 재배치(사실상 삭제+삽입에 가까운 비용)** 를 유발한다.
- 그래서 InnoDB에서는 **PK를 신중히 선택**해야 한다.

## 용어/관찰 포인트
- “클러스터링 인덱스”는 인덱스 자체가 특별하다기보다,
  **테이블이 PK를 기준으로 인덱스 구조(B-Tree)에 ‘붙어서’ 저장되는 방식**을 의미한다.
- 책의 그림(구조도)에서는 “루트/브랜치/리프” 형태는 일반 B-Tree 인덱스와 유사하지만,
  **리프 노드가 곧 데이터 페이지(레코드 전체를 포함)** 라는 점이 핵심이다.

<details>
<summary><strong>8.8.2 세컨더리 인덱스에 미치는 영향</strong></summary>

## 왜 InnoDB의 세컨더리 인덱스가 달라지는가?
클러스터링 테이블(InnoDB)에서는 레코드가 PK 순으로 저장되므로,
세컨더리 인덱스가 “레코드 위치(주소)”를 직접 가리키는 방식이 아니라  
**“PK 값”을 들고 있다가 PK로 다시 찾아가는 방식**을 쓴다.

## InnoDB vs MyISAM 비교(핵심 차이)
- **MyISAM**
    - 세컨더리 인덱스가 레코드를 찾을 때 **레코드의 물리 주소(ROWID 개념)** 를 저장해 바로 접근하는 방식에 가깝다.
- **InnoDB**
    - 세컨더리 인덱스 리프에는
        - (세컨더리 키) + **(프라이머리 키 값)** 이 함께 저장된다.
    - 따라서 세컨더리 인덱스로 후보를 찾은 뒤,
      **프라이머리 키로 클러스터링 인덱스를 한 번 더 탐색**해서 레코드를 가져온다.  
      (즉, 접근 단계가 1번 더 생길 수 있다.)

## 예시 흐름(책의 employees / first_name 검색 맥락)
- 조건: `first_name`에 인덱스가 있고, `WHERE first_name='Aamer'` 같은 조회를 수행
- MyISAM: `ix_firstname`에서 바로 레코드 주소를 얻어 레코드 접근
- InnoDB: `ix_firstname`에서 **PK(emp_no)** 를 얻고 → 다시 PK로 클러스터링 테이블에서 레코드 접근

## 실무적 의미
- InnoDB에서 세컨더리 인덱스 조회는 종종 **“세컨더리 인덱스 탐색 + PK 재탐색”** 비용이 포함된다.
- 다만, 선택 컬럼이 인덱스에 모두 포함되는 **커버링 인덱스** 상황이면
  PK 재탐색을 줄이거나 피할 여지가 생긴다(쿼리/인덱스 설계에 따라).

<details>
<summary><strong>8.8.3 클러스터링 인덱스의 장점과 단점</strong></summary>

## 장점
- **빠른 읽기(SELECT)에 유리**
    - PK 기반 조회/범위 검색에서, 데이터가 PK 순서로 붙어 있어 **연속 접근**이 쉬움.
- **모든 세컨더리 인덱스가 PK를 포함**
    - 특정 조회 패턴에서는 커버링이 되기 쉬워져, 추가 랜덤 I/O를 줄이는 방향으로 설계 가능.

## 단점
- **쓰기(INSERT/UPDATE/DELETE)에 불리**
    - PK 순서 유지를 위해 페이지 분할(Page Split), 이동 비용이 발생할 수 있음.
- **세컨더리 인덱스가 커짐**
    - 세컨더리 인덱스 리프에 PK가 함께 저장되므로,
    - PK가 길면 길수록 **모든 세컨더리 인덱스가 함께 비대해짐**.
- **PK 변경이 특히 비쌈**
    - PK 변경은 레코드 재배치가 필요해서 비용이 매우 큼(“DELETE + INSERT” 유사).

## 결론 요약
- 클러스터링 인덱스의 “이점”은 주로 **읽기 성능(특히 PK 기반)** 에서,
- “대가”는 주로 **쓰기 비용 + 인덱스 크기 증가**에서 발생한다.

<summary><strong>8.8.4 클러스터링 테이블 사용 시 주의사항</strong></summary>

## 8.8.4.1 클러스터링 인덱스 키(=PK)의 크기
InnoDB에서는 세컨더리 인덱스가 PK를 포함하므로,
**PK가 커지면 “모든 세컨더리 인덱스”가 함께 커진다.** 

책의 비교 예시(요지):
- 세컨더리 인덱스가 5개일 때,
    - PK 10바이트 → 세컨더리 인덱스 키가 (10B * 5) 만큼 더해져 상대적으로 작음
    - PK 50바이트 → 세컨더리 인덱스 키가 (50B * 5) 만큼 늘어 급격히 커짐
- 그 결과, “100만 건” 같은 규모에서 인덱스 저장 공간이 **수십 MB vs 수백 MB**로 벌어질 수 있음을 보여준다.

팁:
- PK는 가능하면 **짧고(Compact)**, **불변(Stable)**, **단조 증가(Insert-friendly)** 한 값이 유리하다.

---

## 8.8.4.2 PK는 AUTO_INCREMENT보다 “업무적인 컬럼”이 더 좋은 경우
- 클러스터링은 PK 순서대로 저장되므로,
  “업무적으로 자주 범위 검색/정렬/조인이 되는 키”가 있다면  
  AUTO_INCREMENT보다 그 업무 키를 PK로 두는 것이 **조회에서 유리**할 수 있다.
- 다만, 업무 키가 길거나 변경 가능성이 있다면(예: 문자열 코드 변경),
  오히려 전체 구조 비용이 커질 수 있으니 트레이드오프를 판단한다.

---

## 8.8.4.3 프라이머리 키는 반드시 명시할 것
- InnoDB에서 PK가 없으면 InnoDB가 대체 키를 선택/생성하는 규칙이 있다.
    - (1) PK가 없으면 후보를 찾고(유니크 인덱스 등),
    - (2) 적절한 게 없으면 내부적으로 **숨겨진 키(일련번호/Row ID 성격)** 를 만들어 클러스터링 키로 사용한다.
- 하지만 내부 생성 키는 사용자가 명시적으로 활용할 수 없고, 설계/튜닝/운영 관점에서 불리하므로
  **PK는 명시적으로 생성**하는 것이 권장된다.

---

## 8.8.4.4 AUTO_INCREMENT 값을 “인조 식별자”로 사용할 때
- 업무 키가 길거나(복합키), 변경 가능성이 높다면,
  PK를 업무 키로 두기보다 **AUTO_INCREMENT 같은 짧은 인조 키를 PK로 두는 전략**이 현실적이다.
- 이 경우:
    - PK는 짧고 단조 증가 → INSERT에 유리(페이지 분할/랜덤 삽입 위험 감소)
    - 업무 키는 별도의 **UNIQUE 인덱스**로 강제(중복 방지)
- 즉, “클러스터링(저장 구조) 최적화”와 “업무 제약(유니크/검색)”을 분리하는 방식이다.

</details>

# 9.4 쿼리 힌트 (Query Hint)

MySQL은 버전이 업그레이드되면서 통계 정보와 비용 기반 옵티마이저(Cost-Based Optimizer)가 계속 개선되었다.  
하지만 모든 상황에서 옵티마이저가 **항상 최적의 실행 계획을 선택한다고 보장할 수는 없다.**

특히 다음과 같은 경우에는 옵티마이저의 판단이 항상 올바르지 않을 수 있다.

- 테이블의 통계 정보가 실제 데이터 분포와 다른 경우
- 조인 순서가 잘못 선택되는 경우
- 특정 인덱스를 사용해야 성능이 좋은 경우
- 옵티마이저가 잘못된 비용 계산을 수행하는 경우

이러한 상황에서 **DBA 또는 개발자가 실행 계획에 직접 영향을 주기 위해 사용하는 것이 "쿼리 힌트(Query Hint)"** 이다.

쿼리 힌트는 **옵티마이저에게 특정 실행 전략을 유도하거나 제한하는 역할**을 한다.

MySQL에서 사용할 수 있는 힌트는 크게 다음 두 가지로 나눌 수 있다.

- **인덱스 힌트 (Index Hint)**
- **옵티마이저 힌트 (Optimizer Hint)**

일반적으로 오래된 MySQL 버전에서 사용되던 힌트는 대부분 **인덱스 힌트**이며,  
MySQL 5.6 이후에는 **옵티마이저 힌트가 새롭게 추가되어 더 다양한 실행 전략을 제어할 수 있다.**

---

<details>
<summary><strong>9.4.1 인덱스 힌트 (Index Hint)</strong></summary>

인덱스 힌트는 **MySQL 옵티마이저가 어떤 인덱스를 사용할지 결정하는 과정에 영향을 주는 힌트**이다.

대표적인 인덱스 힌트는 다음과 같다.

- `STRAIGHT_JOIN`
- `USE INDEX`
- `FORCE INDEX`
- `IGNORE INDEX`

이러한 힌트는 **SELECT 뿐 아니라 UPDATE, DELETE 문에서도 사용할 수 있다.**

하지만 인덱스 힌트는 MySQL에 특화된 문법이기 때문에  
다른 RDBMS에서는 사용되지 않는 경우가 많다.

따라서 **특정 MySQL 환경에서만 사용해야 하며, 과도한 사용은 권장되지 않는다.**

---

## 9.4.1.1 STRAIGHT_JOIN

`STRAIGHT_JOIN` 힌트는 **조인 순서를 옵티마이저가 아닌 사용자가 직접 지정하도록 만드는 힌트**이다.

일반적으로 MySQL 옵티마이저는 다음 기준으로 조인 순서를 결정한다.

- 테이블의 통계 정보
- 인덱스 존재 여부
- 예상 레코드 수
- 비용 계산

하지만 이러한 판단이 잘못될 수 있기 때문에  
**특정 상황에서는 사용자가 직접 조인 순서를 지정하는 것이 더 좋은 성능을 낼 수 있다.**

예시:

```sql
SELECT STRAIGHT_JOIN
    e.first_name,
    e.last_name,
    d.dept_name
FROM employees e, dept_emp de, departments d
WHERE e.emp_no = de.emp_no
AND de.dept_no = d.dept_no;
```

이 힌트를 사용하면 **FROM 절에 작성된 순서대로 조인이 수행된다.**

즉

```
employees → dept_emp → departments
```

순서로 조인이 진행된다.

### STRAIGHT_JOIN 사용이 유리한 경우

다음과 같은 경우 STRAIGHT_JOIN 사용을 고려할 수 있다.

1️⃣ **옵티마이저가 잘못된 드라이빙 테이블을 선택하는 경우**

드라이빙 테이블(Driver Table)은  
조인 실행 시 가장 먼저 접근하는 테이블이다.

레코드 수가 많은 테이블이 드라이빙 테이블이 되면  
전체 쿼리 성능이 크게 저하될 수 있다.

---

2️⃣ **WHERE 조건으로 레코드 수가 크게 줄어드는 테이블이 있는 경우**

예를 들어

```
employees (300,000 rows)
WHERE emp_no = 10001
```

같이 **조건으로 단 1건만 조회되는 경우**  
해당 테이블을 먼저 조회하는 것이 훨씬 효율적이다.

---

### 조인 순서 힌트 (Optimizer Join Hint)

STRAIGHT_JOIN과 유사한 기능을 하는 옵티마이저 힌트도 존재한다.

대표적으로 다음이 있다.

- `JOIN_FIXED_ORDER`
- `JOIN_ORDER`
- `JOIN_PREFIX`
- `JOIN_SUFFIX`

이 힌트들은 **STRAIGHT_JOIN보다 더 세밀하게 조인 순서를 제어할 수 있다.**

---

## 9.4.1.2 USE INDEX / FORCE INDEX / IGNORE INDEX

이 힌트들은 **특정 테이블에서 사용할 인덱스를 직접 지정할 때 사용한다.**

인덱스 힌트는 다음과 같은 형식을 가진다.

```
USE INDEX(index_name)
FORCE INDEX(index_name)
IGNORE INDEX(index_name)
```

각 힌트의 의미는 다음과 같다.

---

### USE INDEX

옵티마이저에게 **특정 인덱스를 사용하는 것이 좋다는 힌트**를 제공한다.

```sql
SELECT *
FROM employees USE INDEX(primary)
WHERE emp_no = 10001;
```

하지만 옵티마이저는 필요에 따라 이 힌트를 **무시할 수도 있다.**

즉 **권장 수준의 힌트**이다.

---

### FORCE INDEX

옵티마이저에게 **반드시 해당 인덱스를 사용하도록 강제하는 힌트**이다.

```sql
SELECT *
FROM employees FORCE INDEX(primary)
WHERE emp_no = 10001;
```

옵티마이저가 테이블 풀 스캔을 선택하려는 경우에도  
**강제로 인덱스를 사용하도록 만든다.**

---

### IGNORE INDEX

특정 인덱스를 **사용하지 못하도록 제한하는 힌트**이다.

```sql
SELECT *
FROM employees IGNORE INDEX(ix_firstname)
WHERE first_name = 'Georgi';
```

옵티마이저가 해당 인덱스를 사용하려고 하더라도  
이 힌트를 통해 **다른 인덱스를 사용하도록 유도할 수 있다.**

---

### 인덱스 힌트의 적용 범위

인덱스 힌트는 다음과 같이 **사용 목적에 따라 범위를 지정할 수 있다.**

- `FOR JOIN`
- `FOR ORDER BY`
- `FOR GROUP BY`

예시:

```sql
SELECT *
FROM employees
USE INDEX FOR ORDER BY (idx_firstname)
ORDER BY first_name;
```

---

## 9.4.1.3 SQL_CALC_FOUND_ROWS

`SQL_CALC_FOUND_ROWS` 는 **LIMIT 절이 사용된 쿼리에서 전체 레코드 수를 계산하기 위한 힌트**이다.

예시:

```sql
SELECT SQL_CALC_FOUND_ROWS *
FROM employees
LIMIT 5;
```

이 쿼리는 다음과 같은 특징을 가진다.

- LIMIT 조건에 맞는 결과만 반환
- 하지만 전체 레코드 수도 계산

전체 레코드 수는 다음 쿼리로 확인할 수 있다.

```sql
SELECT FOUND_ROWS();
```

---

### 문제점

이 방식은 **성능 문제가 발생할 수 있다.**

왜냐하면

```
LIMIT 20
```

같이 일부 데이터만 필요하더라도

```
전체 레코드 수 계산
```

을 위해 **전체 테이블을 스캔해야 할 수도 있기 때문이다.**

---

### 권장 방식

대부분의 경우 다음 방식이 더 효율적이다.

```sql
SELECT COUNT(*)
FROM employees
WHERE first_name='Georgi';

SELECT *
FROM employees
WHERE first_name='Georgi'
LIMIT 20;
```

즉

**COUNT 쿼리와 실제 조회 쿼리를 분리하는 것이 더 좋은 성능을 낸다.**

<summary><strong>9.4.2 옵티마이저 힌트 (Optimizer Hint)</strong></summary>

MySQL 8.0 버전에서는 다양한 **옵티마이저 힌트(Optimizer Hint)** 를 사용할 수 있다.  
옵티마이저 힌트는 **쿼리 실행 계획에 직접 영향을 주기 위해 사용하는 힌트**이며, 인덱스 힌트보다 더 다양한 실행 전략을 제어할 수 있다.

옵티마이저 힌트는 **영향 범위에 따라 다음 4가지로 분류할 수 있다.**

1. **인덱스 수준 힌트**
2. **테이블 수준 힌트**
3. **쿼리 블록 수준 힌트**
4. **글로벌 힌트**

MySQL 옵티마이저 힌트는 다음과 같은 형태로 사용한다.

```sql
SELECT /*+ 힌트 */ ...
```

또는

```sql
SELECT /*+ 힌트(테이블명) */ ...
```

---

<details>
<summary><strong>9.4.2.1 옵티마이저 힌트 종류</strong></summary>

MySQL 옵티마이저 힌트는 **영향 범위에 따라 다음과 같이 분류된다.**

### 1️⃣ 인덱스 수준 힌트

특정 인덱스 사용 여부를 제어한다.

대표 힌트

- `INDEX`
- `NO_INDEX`
- `GROUP_INDEX`
- `NO_GROUP_INDEX`
- `JOIN_INDEX`
- `NO_JOIN_INDEX`
- `ORDER_INDEX`
- `NO_ORDER_INDEX`

예시

```sql
SELECT /*+ INDEX(employees ix_firstname) */ *
FROM employees
WHERE first_name='Matt';
```

또는

```sql
SELECT /*+ NO_INDEX(employees ix_firstname) */ *
FROM employees
WHERE first_name='Matt';
```

이 힌트들은 **특정 인덱스 사용 여부를 직접 제어할 수 있다.**

---

### 2️⃣ 테이블 수준 힌트

특정 테이블의 접근 전략을 제어한다.

대표 예시

- `BNL`
- `NO_BNL`
- `HASH_JOIN`
- `NO_HASH_JOIN`

---

### 3️⃣ 쿼리 블록 수준 힌트

쿼리 블록 단위로 실행 전략을 제어한다.

대표 예시

- `SEMIJOIN`
- `SUBQUERY`
- `MERGE`

---

### 4️⃣ 글로벌 힌트

쿼리 전체에 영향을 준다.

대표 예시

- `MAX_EXECUTION_TIME`
- `RESOURCE_GROUP`
- `SET_VAR`

---

옵티마이저 힌트는 **쿼리 블록 단위로도 적용할 수 있다.**

예시

```sql
SELECT /*+ JOIN_ORDER(e, subq1) */
COUNT(*)
FROM employees e
WHERE e.emp_no IN (
    SELECT /*+ QB_NAME(subq1) */ s.emp_no
    FROM salaries s
);
```

</details>

<details>
<summary><strong>9.4.2.2 MAX_EXECUTION_TIME</strong></summary>

`MAX_EXECUTION_TIME` 힌트는 **쿼리 실행 시간의 최대 제한을 설정하는 힌트**이다.

이 힌트를 사용하면 지정된 시간 이상 실행되는 쿼리는 자동으로 중단된다.

예시

```sql
SELECT /*+ MAX_EXECUTION_TIME(100) */ *
FROM employees
ORDER BY last_name
LIMIT 1;
```

이 쿼리는 **최대 실행 시간을 100ms로 제한한다.**

만약 실행 시간이 이를 초과하면 다음과 같은 오류가 발생한다.

```
ERROR 3024 (HY000):
Query execution was interrupted
```

이 힌트는 **쿼리 실행 계획에는 영향을 주지 않고 실행 시간만 제한한다.**

</details>

---

<details>
<summary><strong>9.4.2.3 SET_VAR</strong></summary>

`SET_VAR` 힌트는 **쿼리 실행 시 특정 시스템 변수 값을 변경하는 힌트**이다.

예를 들어 **옵티마이저 설정을 쿼리 단위로 변경할 수 있다.**

예시

```sql
SELECT /*+ SET_VAR(optimizer_switch='index_merge_intersection=off') */ *
FROM employees
WHERE first_name='Georgi'
AND emp_no BETWEEN 1000 AND 20000;
```

이 힌트는 다음과 같은 용도로 사용된다.

- 옵티마이저 기능 활성화 / 비활성화
- 조인 버퍼 크기 조정
- 특정 실행 전략 테스트

하지만 **모든 시스템 변수를 변경할 수 있는 것은 아니다.**

</details>

<details>
<summary><strong>9.4.2.4 SEMIJOIN & NO_SEMIJOIN</strong></summary>

세미 조인(Semi Join)은 **IN 서브쿼리를 최적화하기 위한 전략**이다.

예시

```sql
SELECT *
FROM departments d
WHERE d.dept_no IN (
    SELECT de.dept_no
    FROM dept_emp de
);
```

MySQL 옵티마이저는 다음과 같은 **세미 조인 전략**을 사용할 수 있다.

| 전략 | 설명 |
|---|---|
| Duplicate Weed-out | 중복 제거 방식 |
| FirstMatch | 첫 번째 매칭만 사용 |
| Loose Scan | 인덱스 기반 스캔 |
| Materialization | 서브쿼리 결과를 임시 테이블로 생성 |

이 전략을 힌트로 제어할 수 있다.

예시

```sql
SELECT *
FROM departments d
WHERE d.dept_no IN (
    SELECT /*+ SEMIJOIN(FIRSTMATCH) */ de.dept_no
    FROM dept_emp de
);
```

또는

```sql
SELECT /*+ NO_SEMIJOIN */ ...
```

</details>

---

<details>
<summary><strong>9.4.2.5 SUBQUERY</strong></summary>

서브쿼리 최적화 전략을 제어하는 힌트이다.

대표 전략은 다음 두 가지이다.

| 전략 | 힌트 |
|---|---|
| IN → EXISTS 변환 | `SUBQUERY(INTOEXISTS)` |
| Materialization | `SUBQUERY(MATERIALIZATION)` |

예시

```sql
SELECT *
FROM departments d
WHERE d.dept_no IN (
    SELECT /*+ SUBQUERY(MATERIALIZATION) */ de.dept_no
    FROM dept_emp de
);
```

이 힌트는 **서브쿼리 실행 방식을 직접 지정할 때 사용된다.**

</details>

<details>
<summary><strong>9.4.2.6 BNL / HASH_JOIN</strong></summary>

조인 알고리즘을 제어하는 힌트이다.

### BNL (Block Nested Loop)

블록 단위로 데이터를 읽어 조인을 수행한다.

```sql
SELECT /*+ BNL(e, de) */
FROM employees e
JOIN dept_emp de
ON e.emp_no = de.emp_no;
```

---

### HASH_JOIN

해시 조인을 사용하도록 유도한다.

```sql
SELECT /*+ HASH_JOIN(e, de) */
FROM employees e
JOIN dept_emp de
ON e.emp_no = de.emp_no;
```

MySQL 8.0 이후에는 **Hash Join이 기본적으로 사용될 수 있다.**

</details>

---

<details>
<summary><strong>9.4.2.7 JOIN_ORDER / JOIN_FIXED_ORDER / JOIN_PREFIX / JOIN_SUFFIX</strong></summary>

이 힌트들은 **조인 순서를 직접 제어하기 위한 힌트**이다.

기존에는 `STRAIGHT_JOIN` 으로만 조인 순서를 제어할 수 있었지만  
옵티마이저 힌트를 통해 더 정밀한 제어가 가능해졌다.

---

### JOIN_FIXED_ORDER

FROM 절 순서를 그대로 유지한다.

```sql
SELECT /*+ JOIN_FIXED_ORDER */
FROM employees e
JOIN dept_emp de ON e.emp_no = de.emp_no
JOIN departments d ON d.dept_no = de.dept_no;
```

---

### JOIN_ORDER

특정 조인 순서를 지정한다.

```sql
SELECT /*+ JOIN_ORDER(e, de, d) */
...
```

---

### JOIN_PREFIX

특정 테이블을 **조인의 시작 테이블로 지정**

```sql
SELECT /*+ JOIN_PREFIX(e) */
...
```

---

### JOIN_SUFFIX

특정 테이블을 **조인의 마지막 테이블로 지정**

```sql
SELECT /*+ JOIN_SUFFIX(d) */
...
```

이 힌트들을 통해 **옵티마이저의 조인 순서 선택을 직접 제어할 수 있다.**

</details>

<details>
<summary><strong>9.4.2.8 Loose Index Scan</strong></summary>

### Loose Index Scan

Loose Index Scan은 **GROUP BY 또는 DISTINCT 처리 시 불필요한 레코드 스캔을 줄이기 위해 인덱스의 일부만 건너뛰며 읽는 방식**이다.

일반적인 인덱스 스캔은 조건에 해당하는 모든 레코드를 읽지만, Loose Index Scan은 **각 그룹의 첫 번째 값만 읽고 다음 그룹으로 이동**한다.

#### 특징

- GROUP BY / DISTINCT 쿼리 최적화에 사용
- 불필요한 레코드 접근 최소화
- 인덱스를 활용하여 그룹의 대표 값만 조회

#### 예시

```sql
SELECT DISTINCT user_id
FROM orders;
```

이 경우 인덱스가 `(user_id)`로 존재하면  
각 user_id 그룹의 첫 레코드만 읽고 다음 값으로 이동할 수 있다.

#### 장점

- 디스크 I/O 감소
- GROUP BY 성능 향상
- 대용량 데이터에서 효과적

#### 제약 조건

Loose Index Scan이 사용되려면

- GROUP BY 컬럼이 인덱스 선두 컬럼이어야 함
- WHERE 조건이 인덱스를 활용 가능해야 함

</details>

<details>
<summary><strong>9.4.2.9 Skip Scan</strong></summary>

### Skip Scan

Skip Scan은 **복합 인덱스에서 선두 컬럼을 조건으로 사용하지 않아도 인덱스를 활용할 수 있도록 하는 방식**이다.

예를 들어 인덱스가 다음과 같이 존재한다고 가정한다.

```
INDEX idx (A, B)
```

일반적으로 MySQL은 **A 조건이 없으면 인덱스를 사용하지 못한다.**

하지만 Skip Scan은 다음처럼 동작한다.

```
A=값1 AND B=조건
A=값2 AND B=조건
A=값3 AND B=조건
```

즉 **A의 가능한 값을 순차적으로 탐색하면서 B 조건을 적용**한다.

#### 특징

- 복합 인덱스에서 선두 컬럼 없이도 인덱스 사용 가능
- MySQL 8.0 이후 지원

#### 장점

- 인덱스 활용 가능성 증가
- Full Table Scan 방지

#### 단점

- 선두 컬럼의 distinct 값이 많으면 비효율적
- 내부적으로 여러 번 Range Scan 수행

</details>

<details>
<summary><strong>9.4.2.10 Index Condition Pushdown (ICP)</strong></summary>

### Index Condition Pushdown (ICP)

Index Condition Pushdown은 **스토리지 엔진 단계에서 WHERE 조건을 미리 적용하는 최적화 기법**이다.

일반적인 인덱스 검색 과정

```
1️⃣ 인덱스 검색
2️⃣ 레코드 읽기
3️⃣ WHERE 조건 필터링
```

ICP가 적용되면

```
1️⃣ 인덱스 검색
2️⃣ 인덱스 단계에서 WHERE 조건 일부 필터링
3️⃣ 필요한 레코드만 읽기
```

즉 **불필요한 레코드 접근을 줄인다.**

#### 예시

```sql
SELECT *
FROM employees
WHERE last_name='Kim'
AND first_name LIKE 'J%';
```

인덱스 `(last_name, first_name)`이 존재하면

- last_name = 'Kim' → 인덱스 검색
- first_name LIKE 'J%' → 인덱스 단계에서 필터링

#### 장점

- 디스크 접근 감소
- CPU 연산 감소

#### 적용 대상

- Secondary Index
- Range Scan

</details>

<details>
<summary><strong>9.4.2.11 Multi-Range Read (MRR)</strong></summary>

### Multi-Range Read (MRR)

MRR은 **랜덤 I/O를 줄이기 위해 레코드 접근 순서를 정렬하여 읽는 방식**이다.

일반적인 인덱스 접근 방식

```
Index → Table → Index → Table → Index → Table
```

→ 랜덤 I/O 발생

MRR 방식

```
1️⃣ 인덱스로 PK 목록 수집
2️⃣ PK 정렬
3️⃣ 테이블 순차 접근
```

즉 **랜덤 I/O를 순차 I/O로 변환한다.**

#### 동작 과정

```
Index Scan
     ↓
PK 목록 수집
     ↓
PK 정렬
     ↓
Table Sequential Read
```

#### 장점

- 랜덤 I/O 감소
- 대용량 데이터에서 성능 향상

#### 사용되는 경우

- Range Scan
- Join 처리

</details>

---

<details>
<summary><strong>9.4.2.12 Batched Key Access (BKA)</strong></summary>

### Batched Key Access (BKA)

BKA는 **JOIN 수행 시 MRR을 활용하여 성능을 개선하는 방식**이다.

일반적인 Nested Loop Join

```
for each row in table A:
    index lookup in table B
```

문제

- 반복적인 랜덤 I/O 발생

BKA 방식

```
1️⃣ A 테이블에서 여러 row 수집
2️⃣ B 테이블 lookup key batch 생성
3️⃣ MRR로 정렬
4️⃣ 순차 접근
```

#### 구조

```
Table A → Key Buffer
             ↓
        Key Batch 생성
             ↓
          MRR 실행
             ↓
        Table B 접근
```

#### 장점

- JOIN 랜덤 I/O 감소
- 대규모 JOIN 성능 향상

#### 활성화 방법

```sql
SET optimizer_switch='batched_key_access=on';
```

또는

```
optimizer_switch
```

옵션으로 제어 가능

</details>

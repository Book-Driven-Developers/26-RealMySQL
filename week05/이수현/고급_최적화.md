# 9.3 고급 최적화

MySQL 옵티마이저는 실행 계획을 수립할 때 **통계 정보 + 옵티마이저 옵션**을 활용하여  
최적의 실행 계획을 선택한다.

MySQL 5.5 이후부터 **optimizer_switch** 시스템 변수를 통해  
여러 최적화 알고리즘을 **활성화 / 비활성화**할 수 있다.

---

<details>
<summary>9.3.1 옵티마이저 스위치 옵션</summary>

옵티마이저 스위치는 `optimizer_switch` 시스템 변수로 제어된다.

```sql
SHOW VARIABLES LIKE 'optimizer_switch';
```

옵션 변경 방법

```sql
-- 서버 전체
SET GLOBAL optimizer_switch='index_merge=on';

-- 현재 세션
SET SESSION optimizer_switch='index_merge=on';
```

또는 **쿼리 힌트로 설정 가능**

```sql
SELECT /*+ SET_VAR(optimizer_switch='index_merge=off') */ *
FROM employees;
```

<details>
<summary>9.3.1.1 MRR과 BKA (mrr & batched_key_access)</summary>

### Multi Range Read (MRR)

MRR은 **디스크 접근을 최적화하기 위한 기술**이다.

기존 방식

```
Nested Loop Join
→ 드라이빙 테이블 한 행
→ 드리븐 테이블 랜덤 접근
```

문제점

- 랜덤 I/O 증가
- 디스크 접근 비효율

MRR 방식

```
1️⃣ 드라이빙 테이블의 키를 버퍼에 저장  
2️⃣ 키를 정렬  
3️⃣ 드리븐 테이블을 순차적으로 읽음
```

효과

- 랜덤 I/O 감소
- 디스크 접근 효율 증가

---

### Batched Key Access (BKA)

BKA는 **MRR 기반 Join 방식**이다.

동작 과정

1️⃣ 조인 키들을 버퍼에 저장  
2️⃣ 정렬 수행  
3️⃣ 한 번에 조인 수행

즉

```
Nested Loop Join
→ row by row join

BKA Join
→ batch 기반 join
```

옵션

```
mrr
batched_key_access
```

</details>

<details>
<summary>9.3.1.2 블록 네스티드 루프 조인 (block nested loop)</summary>

MySQL의 기본 조인 방식은 **Nested Loop Join**이다.

동작 방식

```
for row1 in tableA
    for row2 in tableB
        if 조건 만족
            결과 반환
```

문제점

- 드리븐 테이블 반복 스캔

---

### Block Nested Loop Join

조인 버퍼를 활용하여 효율을 높인다.

동작

```
1️⃣ 드라이빙 테이블 row들을 join buffer에 저장
2️⃣ 드리븐 테이블을 한 번만 스캔
3️⃣ buffer와 비교
```

EXPLAIN에서 확인

```
Using join buffer (block nested loop)
```

특징

- 드리븐 테이블 반복 스캔 감소
- join_buffer_size 영향 받음

</details>

<summary>9.3.1.3 인덱스 컨디션 푸시다운 (Index Condition Pushdown)</summary>

MySQL 5.6부터 도입된 기능이다.

목적

```
스토리지 엔진에서
필터링을 먼저 수행
```

기존 방식

```
1️⃣ 인덱스 검색
2️⃣ 테이블 row 읽기
3️⃣ WHERE 조건 검사
```

ICP 방식

```
1️⃣ 인덱스에서 조건 일부 필터링
2️⃣ 필요한 row만 테이블 접근
```

즉

```
Index filtering → Table access
```

---

### 예시

```sql
SELECT *
FROM employees
WHERE last_name='Acton'
AND first_name LIKE 'sal%';
```

EXPLAIN 결과

```
Using index condition
```

효과

- 불필요한 테이블 row 접근 감소
- I/O 감소
- 성능 향상

</details>

<summary>9.3.1.4 인덱스 확장 (use_index_extensions)</summary>

InnoDB는 **세컨더리 인덱스에 자동으로 PK를 포함한다.**

예시

```sql
PRIMARY KEY (dept_no, emp_no)

INDEX ix_fromdate (from_date)
```

실제로는

```
(from_date, dept_no, emp_no)
```

형태로 동작한다.

이를 **Index Extension**이라 한다.

옵티마이저는 이를 활용해

- 추가 인덱스 생성 없이
- 더 많은 조건을 인덱스로 처리할 수 있다.

옵션

```
use_index_extensions
```

</details>

<details>
<summary>9.3.1.5 인덱스 머지 (index merge)</summary>

일반적으로 MySQL은 **하나의 테이블에서 하나의 인덱스만 사용**한다.

하지만 index merge는

```
여러 인덱스를 동시에 사용
```

하여 결과를 합친다.

대표 알고리즘

```
index_merge_intersection
index_merge_union
index_merge_sort_union
```

사용 예

```
WHERE col1 = ?
AND col2 = ?
```

각 조건에 인덱스가 있을 때

```
index1 결과 ∩ index2 결과
```

</details>

<details>
<summary>9.3.1.6 인덱스 머지 교집합 (index merge intersection)</summary>

여러 인덱스 결과의 **교집합**을 사용하는 방식이다.

예시

```sql
SELECT *
FROM employees
WHERE first_name='Georgi'
AND emp_no BETWEEN 10000 AND 20000;
```

각 인덱스 결과

```
first_name index → 253 rows
PK range → 10000 rows
```

옵티마이저는

```
두 인덱스 결과 교집합
```

을 사용한다.

EXPLAIN 결과

```
Using intersect(first_name,PRIMARY)
```

즉

```
index1 결과
∩
index2 결과
```

특징

- 조건별 인덱스 활용
- 데이터 접근 최소화

옵션

```
index_merge_intersection
```

비활성화 예

```sql
SET SESSION optimizer_switch='index_merge_intersection=off';
```

</details>

<details>
<summary>9.3.1.7 인덱스 머지 - 합집합 (index_merge_union)</summary>

## 9.3.1.7 인덱스 머지 - 합집합 (index_merge_union)

`index_merge_union`은 여러 인덱스 검색 결과를 **합집합(Union)** 형태로 결합하는 최적화 방법이다.

### 동작 방식

예를 들어 다음과 같은 쿼리가 있다고 가정하자.

```sql
SELECT *
FROM employees
WHERE first_name='Matt'
   OR hire_date='1987-03-31';
```

위 쿼리는 다음 두 인덱스를 사용할 수 있다.

- `ix_firstname`
- `ix_hiredate`

MySQL 옵티마이저는 각각의 인덱스를 사용해 결과를 검색한 뒤 **두 결과를 합집합으로 병합**한다.

EXPLAIN 결과 예

```
Using union(ix_firstname, ix_hiredate)
```

### 특징

- OR 조건에서 자주 사용됨
- 여러 인덱스를 동시에 활용 가능
- 내부적으로 **중복 제거 과정이 포함됨**

MySQL은 합집합을 만들 때 **우선순위 큐(priority queue)** 구조를 사용하여  
각 인덱스 결과를 병합하면서 중복을 제거한다.

</details>

---

<details>
<summary>9.3.1.8 인덱스 머지 - 정렬 후 합집합 (index_merge_sort_union)</summary>

## 9.3.1.8 인덱스 머지 - 정렬 후 합집합 (index_merge_sort_union)

`index_merge_sort_union`은 `index_merge_union`과 유사하지만  
**결과를 정렬한 후 합집합을 수행하는 방식**이다.

### 사용되는 경우

두 인덱스 검색 결과가 **정렬되어 있지 않은 경우** 사용된다.

예

```sql
SELECT *
FROM employees
WHERE first_name='Matt'
   OR hire_date BETWEEN '1987-03-01' AND '1987-03-31';
```

EXPLAIN 결과

```
Using sort_union(ix_firstname, ix_hiredate)
```

### 특징

- 먼저 각 인덱스 결과를 가져옴
- 결과를 **정렬**
- 이후 **중복 제거 및 병합**

즉

```
Index Scan
 → Sort
 → Merge
```

</details>

<summary>9.3.1.9 세미 조인 (Semijoin)</summary>

## 9.3.1.9 세미 조인 (Semijoin)

세미 조인은 **다른 테이블과 실제 조인을 수행하지 않고  
조건을 만족하는 레코드가 존재하는지만 확인하는 방식**이다.

대표적인 예

```sql
SELECT *
FROM employees e
WHERE e.emp_no IN (
    SELECT de.emp_no
    FROM dept_emp de
    WHERE de.from_date='1995-01-01'
);
```

### 세미 조인 최적화 방법

MySQL은 다음과 같은 세미 조인 전략을 제공한다.

- Table Pull-out
- Duplicate Weed-out
- FirstMatch
- LooseScan
- Materialization

세미 조인 최적화가 적용되면

```
서브쿼리 → 조인 형태
```

로 변환되어 실행된다.

</details>

<details>
<summary>9.3.1.10 테이블 풀아웃 (Table Pull-out)</summary>

## 9.3.1.10 테이블 풀아웃 (Table Pull-out)

`Table Pull-out`은 **서브쿼리 내부의 테이블을 밖으로 끌어내어  
조인 형태로 변환하는 최적화 기법**이다.

예

```sql
SELECT *
FROM employees e
WHERE e.emp_no IN (
    SELECT de.emp_no
    FROM dept_emp de
    WHERE de.dept_no='d009'
);
```

옵티마이저는 이를 다음과 같이 변환할 수 있다.

```
employees
JOIN dept_emp
```

### 특징

- 서브쿼리를 JOIN으로 변환
- 인덱스를 더 효율적으로 사용 가능
- 성능 향상

실행 계획에서 직접 표시되지는 않으며  
`SHOW WARNINGS`로 변환된 쿼리를 확인할 수 있다.

</details>

<summary>9.3.1.11 퍼스트 매치 (FirstMatch)</summary>

## 9.3.1.11 퍼스트 매치 (FirstMatch)

`FirstMatch`는 세미 조인 최적화 중 하나로  
**조건을 만족하는 첫 번째 레코드를 찾으면 추가 탐색을 중단하는 방식**이다.

예

```sql
SELECT *
FROM employees e
WHERE e.first_name='Matt'
AND e.emp_no IN (
    SELECT t.emp_no
    FROM titles t
    WHERE t.from_date BETWEEN '1995-01-01' AND '1995-01-30'
);
```

EXPLAIN 결과

```
FirstMatch(e)
```

### 특징

- 조건을 만족하는 레코드 발견 시 탐색 중단
- 불필요한 검색 감소
- EXISTS와 유사한 동작

### 제한 사항

- 서브쿼리에 GROUP BY 사용 시 적용 불가

</details>

<summary>9.3.1.12 루스 스캔 (LooseScan)</summary>

## 9.3.1.12 루스 스캔 (LooseScan)

LooseScan은 **GROUP BY 최적화 기법을 활용한 세미 조인 방식**이다.

예

```sql
SELECT *
FROM departments d
WHERE d.dept_no IN (
    SELECT de.dept_no
    FROM dept_emp de
);
```

EXPLAIN 결과

```
Using index; LooseScan
```

### 특징

LooseScan은

```
인덱스 스캔
 → 중복 그룹 건너뜀
```

형태로 동작한다.

즉

```
Index Skip Scan
```

과 유사한 방식이다.

### 장점

- 중복된 레코드를 효율적으로 제거
- 불필요한 데이터 접근 감소

옵티마이저 옵션

```
loosescan
```

</details>

<details>
<summary>9.3.1.13 구체화 (Materialization)</summary>

## 9.3.1.13 구체화 (Materialization)

Materialization은 **서브쿼리 결과를 임시 테이블로 생성하여  
이를 기반으로 쿼리를 수행하는 방식**이다.

예

```sql
SELECT *
FROM employees e
WHERE e.emp_no IN (
    SELECT de.emp_no
    FROM dept_emp de
    WHERE de.from_date='1995-01-01'
);
```

옵티마이저는 다음과 같이 동작할 수 있다.

```
1️⃣ 서브쿼리 실행
2️⃣ 결과를 임시 테이블 생성
3️⃣ 외부 쿼리에서 사용
```

EXPLAIN 결과

```
MATERIALIZED
```

### 특징

- 서브쿼리를 한 번만 실행
- 결과를 캐싱
- 반복 실행 비용 감소

### 제한

Materialization은 다음 조건에서 사용된다.

- IN(subquery)
- 서브쿼리가 상관 서브쿼리가 아닌 경우

옵티마이저 옵션

```
materialization
```

</details>

<details>
<summary><strong>9.3.1.14 중복 제거 (Duplicate Weedout)</strong></summary>

### 개념

Duplicate Weedout은 **세미 조인(Semi Join) 최적화 전략 중 하나**로,  
서브쿼리를 **INNER JOIN 방식으로 실행한 뒤 중복을 제거하는 방식**이다.

일반적으로 `IN` 또는 `EXISTS` 형태의 서브쿼리를 처리할 때 사용된다.

예시 쿼리

```sql
SELECT *
FROM employees e
WHERE e.emp_no IN (
    SELECT s.emp_no
    FROM salaries s
    WHERE s.salary > 150000
);
```

위 쿼리는 `salary > 150000` 조건을 만족하는 직원 정보를 조회한다.

하지만 `salaries` 테이블에는 **한 직원에 대해 여러 레코드가 존재할 수 있기 때문에**
JOIN 수행 시 `emp_no`가 **중복될 가능성**이 있다.

그래서 MySQL 옵티마이저는 다음과 같은 전략을 사용한다.

---

### 처리 과정

Duplicate Weedout 알고리즘은 다음 절차로 수행된다.

1️⃣ salaries 테이블에서 조건을 만족하는 레코드를 조회  
2️⃣ employees 테이블과 조인 수행  
3️⃣ 조인 결과를 **임시 테이블에 저장**  
4️⃣ emp_no 기준으로 **중복 레코드 제거**  
5️⃣ 최종 결과 반환

즉 내부적으로는 다음과 유사한 형태로 실행된다.

```sql
SELECT e.*
FROM employees e
JOIN salaries s ON s.emp_no = e.emp_no
WHERE s.salary > 150000
GROUP BY e.emp_no;
```

---

### 실행 계획 특징

EXPLAIN 결과의 `Extra` 컬럼에서 다음 문구를 확인할 수 있다.

```
Start temporary
End temporary
```

이 구간이 **중복 제거를 위한 임시 테이블 처리 과정**이다.

---

### 특징

Duplicate Weedout 전략의 특징

- 내부적으로 **INNER JOIN 방식으로 처리**
- 중복 제거를 위해 **임시 테이블 사용**
- 서브쿼리 대신 **JOIN 기반 실행 계획 생성**

---

### 장점

- 서브쿼리 실행 비용 감소
- JOIN 기반 실행으로 성능 개선 가능

---

### 제약

다음과 같은 경우에는 사용되지 않을 수 있다.

- 서브쿼리에 GROUP BY 존재
- 집계 함수 사용
- 서브쿼리 구조가 복잡한 경우

</details>

<summary><strong>9.3.1.15 컨디션 팬아웃 필터 (condition_fanout_filter)</strong></summary>

### 개념

`condition_fanout_filter`는 **조인 실행 계획의 정확도를 높이기 위한 옵티마이저 기능**이다.

조인 시 **각 테이블에서 필터링되는 레코드 수를 더 정확하게 예측**하여  
더 좋은 실행 계획을 선택하도록 도와준다.

---

### 예제 쿼리

```sql
SELECT *
FROM employees e
INNER JOIN salaries s ON s.emp_no = e.emp_no
WHERE e.first_name='Matt'
AND e.hire_date BETWEEN '1985-11-21' AND '1986-11-21';
```

---

### 기본 실행 계획

옵티마이저는 보통 다음 순서로 처리한다.

1️⃣ employees 테이블에서 `first_name='Matt'` 조건으로 검색  
2️⃣ 해당 결과를 기반으로 salaries 테이블 조인

문제는 **WHERE 조건이 여러 개 존재할 때**  
각 조건이 얼마나 레코드를 줄일지 **정확히 계산하기 어렵다**는 점이다.

---

### filtered 컬럼

EXPLAIN 결과에는 다음 값이 나타난다.

| 컬럼 | 의미 |
|-----|------|
| rows | 예상 레코드 수 |
| filtered | 조건 적용 후 남는 비율 |

예시

```
rows = 233
filtered = 23.20
```

이는 다음 의미다.

```
233 * 23.2% ≈ 약 54건
```

즉 최종적으로 약 54개의 레코드가 남을 것으로 예측한다.

---

### condition_fanout_filter 동작 방식

MySQL 8.0에서는 다음 정보를 이용해 더 정확한 예측을 수행한다.

- 인덱스 통계 정보
- 히스토그램
- 컬럼 분포

이를 통해 **각 조건이 결과 집합을 얼마나 줄이는지 계산한다.**

---

### 설정

옵션 활성화

```sql
SET optimizer_switch='condition_fanout_filter=on';
```

비활성화

```sql
SET optimizer_switch='condition_fanout_filter=off';
```

---

### 특징

- 조인 순서 결정 정확도 향상
- 실행 계획 품질 개선
- 복잡한 WHERE 조건에서 효과적

</details>

<summary><strong>9.3.1.16 파생 테이블 병합 (Derived Merge)</strong></summary>

### 개념

파생 테이블(Derived Table)은 `FROM` 절의 서브쿼리로 생성되는 **임시 테이블**이다.

예시

```sql
SELECT *
FROM (
    SELECT *
    FROM employees
    WHERE first_name='Matt'
) derived_table
WHERE derived_table.hire_date='1986-04-03';
```

기존 MySQL은 다음과 같이 처리했다.

1️⃣ 서브쿼리 실행  
2️⃣ 임시 테이블 생성  
3️⃣ 외부 쿼리 실행

이 과정은 **임시 테이블 생성 비용 + 데이터 복사 비용**이 발생한다.

---

### Derived Merge 최적화

MySQL 옵티마이저는 가능한 경우  
**파생 테이블을 외부 쿼리와 병합한다.**

즉 다음처럼 변환된다.

```sql
SELECT *
FROM employees
WHERE first_name='Matt'
AND hire_date='1986-04-03';
```

이렇게 하면

- 임시 테이블 생성 없음
- 데이터 복사 없음
- 쿼리 성능 향상

---

### Derived Merge가 불가능한 경우

다음 조건에서는 병합이 불가능하다.

- 집계 함수 사용
- DISTINCT 사용
- GROUP BY
- LIMIT
- UNION
- 윈도우 함수

이 경우 파생 테이블을 실제로 생성한다.

</details>

<summary><strong>9.3.1.17 인비저블 인덱스 (use_invisible_indexes)</strong></summary>

### 개념

---

### 목적

- 인덱스 제거 전 성능 영향 테스트
- 인덱스 사용 여부 실험

### 인덱스 숨기기

```sql
ALTER TABLE employees
```

```sql
ALTER TABLE employees

```

### 옵티마이저 설정

Invisible 인덱스를 사용하도록 설정

```sql
SET optimizer_switch='use_invisible_indexes=on';
```

기본값은 OFF이다.

---

### 특징

- 실제 인덱스는 존재
- 옵티마이저만 무시
- 성능 테스트에 유용

<summary><strong>9.3.1.18 스킵 스캔 (Skip Scan)</strong></summary>

### 개념

Skip Scan은 **복합 인덱스의 선행 컬럼을 사용하지 않아도 인덱스를 사용할 수 있도록 하는 최적화**이다.

예를 들어 다음 인덱스가 있다고 가정한다.

```sql
INDEX ix_gender_birthdate (gender, birth_date)
```

---

### 일반적인 인덱스 사용 조건

복합 인덱스는 **선행 컬럼부터 조건이 있어야 사용 가능**하다.

가능한 쿼리

```sql
SELECT *
FROM employees
WHERE gender='M'
AND birth_date >= '1965-02-01';
```

하지만 다음 쿼리는 인덱스를 사용할 수 없다.

```sql
SELECT *
FROM employees
WHERE birth_date >= '1965-02-01';
```

---

### Skip Scan 최적화

MySQL 8.0은 내부적으로 다음과 같이 처리한다.

```
gender='M' 조건 검색
gender='F' 조건 검색
```

즉 **선행 컬럼의 가능한 값을 순회하면서 인덱스를 스캔**한다.

그래서 `birth_date` 조건만 있어도 인덱스를 사용할 수 있다.

---

### 설정

활성화

```sql
SET optimizer_switch='skip_scan=on';
```

비활성화

```sql
SET optimizer_switch='skip_scan=off';
```

---

### 힌트 사용

Skip Scan 강제 사용

```sql
SELECT /*+ SKIP_SCAN(employees) */ COUNT(*)
FROM employees
WHERE birth_date >= '1965-02-01';
```

Skip Scan 비활성화

```sql
SELECT /*+ NO_SKIP_SCAN(employees) */ COUNT(*)
FROM employees
WHERE birth_date >= '1965-02-01';
```

---

### 특징

- 복합 인덱스 활용도 증가
- 선행 컬럼 조건이 없어도 인덱스 사용 가능
- 특정 상황에서 성능 개선

</details>

<summary><strong>9.3.1.19 해시 조인 (Hash Join)</strong></summary>

## 개념

Hash Join은 **조인 키 값을 해시 테이블로 구성하여 두 테이블을 조인하는 알고리즘**이다.

MySQL 8.0.18부터 도입되었으며, **조인 조건에 인덱스가 없을 때 성능을 크게 개선할 수 있는 방식**이다.

기존 MySQL은 대부분 **Nested Loop Join**을 사용했는데,  
이 방식은 다음과 같은 문제가 있다.

- 조인 대상 테이블을 여러 번 반복 스캔
- 데이터가 많을 경우 성능 급격히 저하

그래서 MySQL 8.0에서는 **Hash Join을 추가로 지원**하게 되었다.

---

## Nested Loop Join vs Hash Join

### Nested Loop Join

```
for each row in table A
    search matching row in table B
```

특징

- 인덱스 존재 시 매우 빠름
- 인덱스 없으면 성능 저하

---

### Hash Join

```
1️⃣ Build Phase
2️⃣ Probe Phase
```

두 단계로 동작한다.

---

## Build Phase

작은 테이블을 기준으로 **해시 테이블을 생성**한다.

```
hash(key) → bucket
```

예시

```
dept_emp 테이블
emp_no 기준 해시 테이블 생성
```

이 테이블을 **Build Table**이라고 한다.

---

## Probe Phase

다른 테이블을 스캔하면서

```
hash(join_key)
```

값을 계산하여 **해시 테이블에서 매칭되는 레코드 검색**한다.

이를 **Probe Table**이라고 한다.

---

## 실행 예시

```sql
SELECT *
FROM employees e
INNER JOIN dept_emp de
ON de.emp_no = e.emp_no
AND de.from_date = e.hire_date;
```

EXPLAIN 결과

```
Using join buffer (hash join)
```

Extra 컬럼에 나타난다.

---

## Hash Join 내부 구조

해시 조인은 다음 구조로 동작한다.

```
Build table → Hash table 생성
Probe table → Hash lookup
Result → 조인 결과 반환
```

---

## 메모리 문제

해시 테이블은 **join_buffer_size 메모리 영역**을 사용한다.

기본값

```
256KB
```

테이블이 매우 크면 메모리에 모두 저장할 수 없다.

이 경우 **디스크 기반 Hash Join**이 수행된다.

---

## Grace Hash Join

메모리가 부족하면 다음 방식이 사용된다.

1️⃣ 데이터를 여러 **partition**으로 나눔  
2️⃣ 각 partition을 디스크에 저장  
3️⃣ partition 단위로 Hash Join 수행

이 방식은 **Grace Hash Join**이라고 한다.

---

## Hash Join 특징

장점

- 인덱스 없어도 빠른 조인 가능
- 대량 데이터 조인에 유리

단점

- 메모리 사용량 큼
- 작은 테이블에서는 Nested Loop가 더 빠를 수 있음

---

## MySQL Hash Join 특징

MySQL은 내부적으로 **xxHash64 해시 함수**를 사용한다.

이는

- 충돌 적음
- 계산 속도 빠름

이라는 특징이 있다.

</details>

<details>
<summary><strong>9.3.1.20 인덱스 정렬 선호 (prefer_ordering_index)</strong></summary>

## 개념

MySQL 옵티마이저는 `ORDER BY` 또는 `GROUP BY`가 있는 경우  
**정렬 비용을 줄이기 위해 정렬된 인덱스를 우선적으로 선택**할 수 있다.

이를 제어하는 옵션이

```
prefer_ordering_index
```

이다.

---

## 예시 쿼리

```sql
SELECT *
FROM employees
WHERE hire_date BETWEEN '1985-01-01' AND '1985-02-01'
ORDER BY emp_no;
```

이 쿼리는 다음 두 가지 실행 계획이 가능하다.

---

### 실행 계획 1

```
hire_date 인덱스 사용
→ 결과 조회
→ 정렬 수행
```

장점

- WHERE 조건 빠름

단점

- 정렬 비용 발생

---

### 실행 계획 2

```
PRIMARY KEY(emp_no) 순서로 스캔
→ WHERE 조건 필터링
```

장점

- ORDER BY 정렬 필요 없음

단점

- 불필요한 레코드 많이 읽음

---

## 옵티마이저 판단

옵티마이저는 다음을 비교한다.

```
정렬 비용
vs
추가 레코드 읽기 비용
```

상황에 따라 두 번째 방식이 더 효율적일 수 있다.

---

## prefer_ordering_index 옵션

이 옵션이 활성화되면

```
ORDER BY 정렬을 줄일 수 있는 인덱스
```

를 **우선적으로 선택**한다.

---

## 설정

활성화

```sql
SET optimizer_switch='prefer_ordering_index=on';
```

비활성화

```sql
SET optimizer_switch='prefer_ordering_index=off';
```

---

## 주의점

MySQL 8.0 이전에는

```
ORDER BY 인덱스 우선 선택
```

때문에 오히려 **비효율적인 실행 계획이 선택되는 문제**가 있었다.

그래서 최신 버전에서는 이 옵션을 통해  
**옵티마이저 동작을 제어할 수 있게 되었다.**

---

## 특징

- ORDER BY 성능 개선 가능
- 불필요한 filesort 감소
- 하지만 경우에 따라 성능 악화 가능

</details>

<summary><strong>9.3.2 조인 최적화 알고리즘</strong></summary>

## 개요

MySQL 옵티마이저는 **조인 쿼리의 실행 계획을 결정하기 위해 두 가지 알고리즘**을 사용한다.

1️⃣ Exhaustive 검색 알고리즘  
2️⃣ Greedy 검색 알고리즘

조인 쿼리는 **테이블 조인 순서(join order)** 에 따라 성능이 크게 달라진다.

예시

```sql
SELECT *
FROM t1, t2, t3, t4
WHERE ...
```

이 경우 다음과 같은 조인 순서가 가능하다.

```
t1 → t2 → t3 → t4
t1 → t3 → t2 → t4
t2 → t1 → t4 → t3
...
```

즉 **모든 가능한 조합 중에서 가장 비용이 낮은 실행 계획을 선택해야 한다.**

---

# 1️⃣ Exhaustive 검색 알고리즘

## 개념

Exhaustive 알고리즘은 **가능한 모든 조인 순서를 탐색하여 최적의 실행 계획을 찾는 방식**이다.

즉

```
모든 조합 탐색
→ 비용 계산
→ 최적 실행 계획 선택
```

---

## 예시

테이블이 4개라면 가능한 조인 순서는

```
4! = 24가지
```

20개 테이블이면

```
20! ≈ 2.4 × 10^18
```

즉 **모든 조합 탐색이 현실적으로 불가능해진다.**

그래서 테이블 수가 많아지면 실행 계획 수립 시간이 급격히 증가한다.

---

## 특징

장점

- 항상 **최적 실행 계획 보장**

단점

- 테이블 수가 많아지면 **계산 비용 폭증**

---

## MySQL 사용 범위

MySQL은

```
조인 테이블 수가 적을 때
```

Exhaustive 알고리즘을 사용한다.

---

# 2️⃣ Greedy 검색 알고리즘

## 개념

Greedy 알고리즘은 **Exhaustive 알고리즘의 시간 복잡도를 줄이기 위해 도입된 방식**이다.

전체 조합을 탐색하지 않고 **현재 단계에서 가장 좋은 선택을 계속 선택하는 방식**이다.

---

## 동작 과정

예를 들어 테이블이 4개라면

```
t1, t2, t3, t4
```

다음 단계로 실행 계획을 만든다.

### 1단계

가능한 첫 테이블 선택

```
t1
t2
t3
t4
```

가장 비용이 낮은 테이블 선택

---

### 2단계

선택된 테이블에 대해 다음 조인 후보 계산

예

```
(t1,t2)
(t1,t3)
(t1,t4)
```

가장 비용이 낮은 조합 선택

---

### 3단계

다음 테이블 추가

```
(t1,t2,t3)
(t1,t2,t4)
```

최소 비용 선택

---

### 4단계

마지막 테이블 추가

```
(t1,t2,t3,t4)
```

최종 실행 계획 생성

---

## 특징

장점

- 실행 계획 생성 속도 빠름
- 테이블 수 많아도 처리 가능

단점

- 항상 최적 실행 계획 보장하지 않음

---

# optimizer 관련 시스템 변수

MySQL에서는 조인 최적화 동작을 제어하기 위해 다음 시스템 변수를 제공한다.

---

## optimizer_search_depth

Greedy 알고리즘에서 **탐색 깊이**를 설정한다.

예시

```
optimizer_search_depth = 4
```

의미

```
최대 4개 테이블 조합까지 탐색
```

설정 예시

```sql
SET optimizer_search_depth = 5;
```

---

## optimizer_prune_level

조인 순서를 탐색할 때 **불필요한 실행 계획을 제거하는 휴리스틱(Heuristic)** 기능이다.

값

```
0 = pruning 없음
1 = pruning 사용
```

기본값

```
1
```

설정 예시

```sql
SET optimizer_prune_level = 1;
```

---

# optimizer_search_depth 실험

책에서는 다음 실험을 수행한다.

```
30개 테이블 조인 쿼리
```

테이블 구조

```
tab01 ~ tab30
```

그리고 다음 변수를 변경하며 실행 계획 생성 시간을 측정했다.

```
optimizer_search_depth = 1 ~ 62
```

---

## 결과

실험 결과

```
depth 증가 → 실행 계획 수립 시간 증가
```

하지만 일정 깊이 이상에서는 **성능 개선 효과가 거의 없다.**

그래서 MySQL 8.0에서는

```
optimizer_search_depth
```

의 영향이 과거보다 크게 줄어들었다.

---

# MySQL 8.0의 변화

MySQL 8.0에서는

- 조인 최적화 알고리즘 개선
- 통계 정보 개선
- 히스토그램 도입

등의 이유로

```
optimizer_search_depth
optimizer_prune_level
```

같은 변수의 튜닝 필요성이 크게 줄어들었다.

---

# 핵심 정리

| 알고리즘 | 특징 |
|---|---|
| Exhaustive | 모든 조합 탐색, 최적 보장 |
| Greedy | 단계별 최적 선택, 속도 빠름 |

---

## 핵심 포인트

- 조인 성능은 **테이블 조인 순서**에 크게 영향을 받는다.
- MySQL은 **Exhaustive + Greedy 알고리즘을 함께 사용**한다.
- 시스템 변수로 조인 탐색 깊이를 조절할 수 있다.

</details>

# 9.2 기본 데이터 처리

MySQL 서버를 포함한 대부분의 RDBMS는 데이터를 **정렬하거나 그룹핑하는 등 기본적인 데이터 처리 기능**을 제공한다.  
이러한 처리는 DBMS 내부 알고리즘에 의해 수행되며, MySQL 옵티마이저는 다양한 처리 방식 중에서 **가장 효율적인 방법을 선택**한다.

---

<details>
<summary><strong>9.2.1 풀 테이블 스캔과 풀 인덱스 스캔</strong></summary>

## 풀 테이블 스캔 (Full Table Scan)

풀 테이블 스캔은 **인덱스를 사용하지 않고 테이블의 데이터를 처음부터 끝까지 읽는 방식**이다.

즉, 테이블에 저장된 모든 레코드를 순차적으로 읽으면서 조건을 검사한다.

### 특징

- 인덱스를 사용하지 않음
- 테이블 전체를 순차적으로 스캔
- 많은 디스크 I/O 발생 가능
- 데이터 양이 많을 경우 성능 저하 가능

### 옵티마이저가 풀 테이블 스캔을 선택하는 경우

MySQL 옵티마이저는 다음과 같은 상황에서 풀 테이블 스캔을 선택할 수 있다.

1️⃣ 테이블의 레코드 수가 매우 적은 경우

- 인덱스를 사용하는 것보다
- **테이블 전체를 읽는 것이 더 빠른 경우**

2️⃣ WHERE 조건에 사용할 적절한 인덱스가 없는 경우

3️⃣ 인덱스를 사용하더라도 조회되는 레코드가 매우 많은 경우

- 인덱스 → 테이블 접근이 반복되기 때문에
- 차라리 **테이블 전체를 읽는 것이 더 빠를 수 있음**

---

## 풀 인덱스 스캔 (Full Index Scan)

풀 인덱스 스캔은 **인덱스의 처음부터 끝까지 순차적으로 읽는 방식**이다.

테이블 전체를 읽는 대신 **인덱스 전체를 스캔**한다.

### 특징

- 인덱스를 순차적으로 읽음
- 테이블보다 인덱스 크기가 작기 때문에  
  **디스크 I/O가 더 적을 수 있음**

### 예시

```sql
SELECT COUNT(*) FROM employees;
```

이 쿼리는 특정 조건 없이 **레코드 개수만 계산**한다.

이 경우 옵티마이저는 다음 두 가지 방법 중 선택할 수 있다.

1️⃣ 풀 테이블 스캔  
2️⃣ 풀 인덱스 스캔

일반적으로 인덱스가 테이블보다 작기 때문에  
**풀 인덱스 스캔이 더 효율적일 가능성이 높다.**

---

## InnoDB의 Read Ahead

InnoDB 스토리지 엔진은 풀 테이블 스캔 성능을 향상시키기 위해  
**Read Ahead 기능**을 사용한다.

### Read Ahead란

필요한 데이터 페이지를 읽기 전에  
**앞으로 필요할 것으로 예상되는 페이지를 미리 읽어오는 기능**

### 동작 방식

1️⃣ 특정 페이지 접근 발생  
2️⃣ InnoDB가 이후에 필요한 페이지를 예측  
3️⃣ 여러 페이지를 한 번에 미리 읽어 버퍼 풀에 적재

이를 통해

- 디스크 I/O 감소
- 순차 접근 성능 향상

을 얻을 수 있다.

### 설정 파라미터

```sql
innodb_read_ahead_threshold
```

이 값은 **Read Ahead가 시작되는 기준**을 설정한다.

---

## 정리

| 구분 | 설명 |
|---|---|
| 풀 테이블 스캔 | 테이블 데이터를 처음부터 끝까지 읽음 |
| 풀 인덱스 스캔 | 인덱스를 처음부터 끝까지 읽음 |
| Read Ahead | InnoDB가 필요한 페이지를 미리 읽어오는 기능 |

</details>

<summary><strong>9.2.2 병렬 처리</strong></summary>

## 병렬 처리 (Parallel Processing)

MySQL 8.0부터는 **하나의 쿼리를 여러 스레드로 병렬 처리**할 수 있는 기능이 추가되었다.

주의할 점은

> 여러 개의 쿼리를 동시에 처리하는 것이 아니라  
> **하나의 쿼리를 여러 스레드로 나누어 처리하는 것**이다.

---

## 병렬 처리 적용 조건

현재 MySQL에서 병렬 처리는 다음과 같은 조건에서 적용된다.

- WHERE 조건이 단순한 경우
- 테이블 전체 스캔이 필요한 경우
- 대량 데이터 처리 쿼리

예를 들어 다음과 같은 쿼리

```sql
SELECT COUNT(*) FROM salaries;
```

이 경우 테이블 전체 레코드를 읽어야 하기 때문에  
**병렬 처리를 통해 성능을 향상시킬 수 있다.**

---

## 병렬 처리 설정

MySQL에서는 다음 시스템 변수를 통해 병렬 처리 스레드 수를 설정할 수 있다.

```sql
innodb_parallel_read_threads
```

### 예시

```sql
SET SESSION innodb_parallel_read_threads=1;
SELECT COUNT(*) FROM salaries;
```

```sql
SET SESSION innodb_parallel_read_threads=2;
SELECT COUNT(*) FROM salaries;
```

```sql
SET SESSION innodb_parallel_read_threads=4;
SELECT COUNT(*) FROM salaries;
```

```sql
SET SESSION innodb_parallel_read_threads=8;
SELECT COUNT(*) FROM salaries;
```

스레드 수가 증가할수록  
**쿼리 처리 시간이 감소하는 것을 확인할 수 있다.**

---

## 병렬 처리의 한계

현재 MySQL 병렬 처리는 다음과 같은 제한이 있다.

- 모든 쿼리에 적용되지 않음
- 복잡한 WHERE 조건에서는 사용되지 않음
- 주로 **Full Table Scan 기반 쿼리에서만 적용**

즉, 병렬 처리는 **대량 데이터 분석 쿼리에서 효과적**이다.

---

## 정리

| 항목 | 설명 |
|---|---|
| 병렬 처리 | 하나의 쿼리를 여러 스레드로 나누어 처리 |
| 적용 버전 | MySQL 8.0 |
| 설정 변수 | innodb_parallel_read_threads |
| 적용 대상 | 대량 데이터 스캔 쿼리 |

</details>

<summary><strong>9.2.3 ORDER BY 처리 (Using filesort)</strong></summary>

# 9.2.3 ORDER BY 처리

대부분의 SELECT 쿼리는 **정렬(ORDER BY)** 기능을 사용한다.  
MySQL은 정렬을 수행할 때 다음 두 가지 방법 중 하나를 사용한다.

1️⃣ 인덱스를 이용한 정렬  
2️⃣ Filesort를 이용한 정렬

실행 계획(EXPLAIN)의 Extra 컬럼에 다음과 같은 메시지가 나타난다면  
MySQL이 **Filesort 방식으로 정렬을 수행**한다는 의미다.

```
Using filesort
```

---

# Filesort란?

Filesort는 **MySQL이 별도의 정렬 작업을 수행하는 방식**이다.

이름 때문에 디스크 정렬처럼 보이지만 실제 의미는 다음과 같다.

> 인덱스를 이용하지 못할 때 MySQL이 자체 정렬 알고리즘을 수행하는 것

정렬 과정은

1️⃣ 정렬 대상 레코드를 메모리에 적재  
2️⃣ 메모리에서 정렬 수행  
3️⃣ 메모리가 부족하면 디스크 사용

---

# 9.2.3.1 소트 버퍼 (Sort Buffer)

MySQL은 정렬을 수행할 때 **Sort Buffer**라는 메모리를 사용한다.

정렬 대상 데이터를 이 버퍼에 저장한 뒤 정렬을 수행한다.

관련 시스템 변수

```
sort_buffer_size
```

### 특징

- 정렬 작업마다 할당됨
- 쿼리 실행이 끝나면 해제됨
- 너무 크게 설정하면 메모리 낭비 가능

### 주의

대량 정렬 시

- 메모리 부족
- 디스크 임시 파일 사용

이 발생할 수 있다.

---

# 9.2.3.2 정렬 알고리즘

MySQL의 filesort 정렬 방식은 크게 **두 가지 알고리즘**이 존재한다.

---

## 1️⃣ 싱글 패스 정렬 (Single-pass)

정렬 대상 레코드를 **한 번만 읽어 정렬하는 방식**

### 특징

- SELECT 대상 컬럼을 함께 버퍼에 저장
- 추가 테이블 접근 필요 없음
- I/O 감소

### 동작 과정

1️⃣ 테이블 스캔  
2️⃣ 정렬 대상 + SELECT 컬럼을 sort buffer 저장  
3️⃣ 메모리 정렬  
4️⃣ 결과 반환

### 장점

- 테이블을 한 번만 읽음
- 디스크 접근 최소화

### 단점

- 메모리 사용량 증가

---

## 2️⃣ 투 패스 정렬 (Two-pass)

예전 MySQL에서 사용하던 방식

### 특징

- 정렬 키 + rowid만 저장
- 정렬 후 다시 테이블 접근

### 동작 과정

1️⃣ 테이블 스캔
2️⃣ 정렬 키 + rowid 저장
3️⃣ 정렬 수행
4️⃣ rowid 기반으로 다시 테이블 조회

### 단점

- 테이블을 두 번 읽어야 함
- 디스크 I/O 증가

현재 MySQL에서는 **가능하면 싱글 패스 정렬을 사용**한다.

---

# 9.2.3.3 정렬 처리 방법

ORDER BY 정렬은 다음 세 가지 방식으로 수행된다.

| 정렬 방법 | 특징 |
|---|---|
| 인덱스 정렬 | 가장 빠름 |
| 조인의 드라이빙 테이블 정렬 | 조인 전에 정렬 |
| 임시 테이블 정렬 | Using temporary + filesort |

---

## 1️⃣ 인덱스를 이용한 정렬

가장 이상적인 정렬 방식이다.

조건

- ORDER BY 컬럼이 인덱스에 존재
- 인덱스 순서와 ORDER BY 방향이 동일

예시

```
SELECT *
FROM employees
WHERE emp_no BETWEEN 10002 AND 10020
ORDER BY emp_no;
```

이 경우

- emp_no 인덱스 사용
- 별도의 정렬 필요 없음

---

## 2️⃣ 조인의 드라이빙 테이블만 정렬

조인 쿼리에서 사용하는 방식이다.

특징

- 조인 전에 드라이빙 테이블을 정렬
- 이후 조인 수행

즉

```
ORDER BY 컬럼이 드라이빙 테이블에 있을 때
```

사용 가능하다.

---

## 3️⃣ 임시 테이블을 이용한 정렬

다음 경우 발생한다.

- ORDER BY 컬럼이 조인 결과에 존재
- 인덱스를 사용할 수 없음

이 경우 MySQL은

1️⃣ 조인 수행
2️⃣ 결과를 임시 테이블 저장
3️⃣ filesort 수행

실행 계획

```
Using temporary; Using filesort
```

---

# 9.2.3.4 스트리밍 방식 vs 버퍼링 방식

MySQL의 쿼리 처리 방식은 크게 두 가지로 나뉜다.

---

## 1️⃣ 스트리밍 방식 (Streaming)

데이터를 읽는 즉시 클라이언트에 전달한다.

### 특징

- 메모리 사용량 적음
- 빠른 응답 시작

예

```
SELECT * FROM table WHERE ...
```

조건을 만족하는 레코드를 찾는 즉시 반환한다.

---

## 2️⃣ 버퍼링 방식 (Buffering)

결과를 모두 생성한 뒤 반환한다.

### 특징

- ORDER BY
- GROUP BY
- DISTINCT

같은 작업에서 사용된다.

이 경우 MySQL은

1️⃣ 모든 결과 생성
2️⃣ 정렬 / 그룹 처리
3️⃣ 결과 반환

을 수행한다.

---

# 정렬 처리 성능 비교

정렬 처리 성능은 다음 순서로 좋다.

```
인덱스 정렬
   ↓
조인 드라이빙 테이블 정렬
   ↓
임시 테이블 + filesort
```

가능하다면 항상

> **ORDER BY 컬럼에 인덱스를 생성하는 것이 가장 좋다**

---

# 정렬 관련 상태 변수

MySQL은 정렬 작업 관련 통계를 제공한다.

```
SHOW STATUS LIKE 'Sort%';
```

대표 변수

| 변수 | 의미 |
|---|---|
| Sort_scan | 테이블 스캔 기반 정렬 |
| Sort_range | 인덱스 범위 정렬 |
| Sort_rows | 정렬된 레코드 수 |
| Sort_merge_passes | 디스크 merge 발생 횟수 |

이 값들을 통해

- 정렬 성능
- filesort 발생 여부

를 확인할 수 있다.

---

# 핵심 정리

ORDER BY 처리 방식

```
1️⃣ 인덱스 정렬 (가장 빠름)
2️⃣ 조인 드라이빙 테이블 정렬
3️⃣ 임시 테이블 + filesort (가장 느림)
```

Filesort 특징

- 인덱스 사용 불가 시 수행
- sort buffer 사용
- 싱글 패스 / 투 패스 알고리즘 존재

가능하다면

> ORDER BY 컬럼에 인덱스를 생성하는 것이 가장 좋은 성능 최적화 방법이다.

</details>

# 9.2.4 GROUP BY 처리

GROUP BY는 ORDER BY와 달리 **스트리밍 방식으로 처리할 수 없는 연산**이다.  
그 이유는 동일 그룹에 속하는 모든 레코드를 모아 **집계(Aggregation)** 해야 하기 때문이다.

MySQL에서 GROUP BY 처리는 크게 다음 세 가지 방식으로 수행된다.

```
1️⃣ 인덱스 스캔을 이용한 GROUP BY
2️⃣ 루스 인덱스 스캔을 이용한 GROUP BY
3️⃣ 임시 테이블을 이용한 GROUP BY
```

---

# 9.2.4.1 인덱스 스캔을 이용한 GROUP BY

GROUP BY 컬럼이 인덱스의 **왼쪽 Prefix**와 일치하면  
MySQL은 인덱스를 순서대로 읽으면서 그룹핑을 수행할 수 있다.

예시

```sql
EXPLAIN
SELECT emp_no
FROM salaries
WHERE from_date='1985-03-01'
GROUP BY emp_no;
```

실행 계획

```
Using where; Using index for group-by
```

### 특징

- 인덱스를 순서대로 읽으면서 그룹 생성
- 별도의 정렬 또는 임시 테이블 필요 없음
- 가장 효율적인 GROUP BY 방식

---

# 9.2.4.2 루스 인덱스 스캔을 이용한 GROUP BY

루스 인덱스 스캔(Loose Index Scan)은  
**각 그룹의 첫 번째 레코드만 읽어 그룹을 생성하는 방식**이다.

즉, 인덱스를 순차적으로 모두 읽는 것이 아니라  
**그룹 경계만 찾아서 점프하면서 읽는다.**

### 특징

- 매우 적은 I/O
- GROUP BY 성능 크게 향상

하지만 다음 조건을 만족해야 한다.

### 사용 조건

- GROUP BY 컬럼이 인덱스에 존재
- 집계 함수가 제한됨

예

```
MIN()
MAX()
```

사용 가능

하지만 다음은 사용 불가

```
SUM()
COUNT()
AVG()
```

---

# 루스 인덱스 스캔 사용 예

가능한 쿼리

```sql
SELECT col1, MIN(col2)
FROM table
GROUP BY col1;
```

불가능한 쿼리

```sql
SELECT col1, SUM(col2)
FROM table
GROUP BY col1;
```

---

# 9.2.4.3 임시 테이블을 이용한 GROUP BY

다음 상황에서는 임시 테이블이 사용된다.

- GROUP BY 컬럼에 인덱스 없음
- GROUP BY 컬럼이 조인 결과에 존재
- 집계 함수가 복잡

예

```sql
EXPLAIN
SELECT e.last_name, AVG(s.salary)
FROM employees e, salaries s
WHERE s.emp_no=e.emp_no
GROUP BY e.last_name;
```

실행 계획

```
Using temporary
```

### 처리 과정

```
1️⃣ 조인 수행
2️⃣ 결과를 임시 테이블 저장
3️⃣ GROUP BY 수행
4️⃣ 집계 계산
```

### 특징

- 디스크 사용 가능
- 성능 저하 가능

---

# MySQL 8.0 GROUP BY 변경 사항

MySQL 8.0 이전

```
GROUP BY → 자동 정렬
```

MySQL 8.0 이후

```
GROUP BY → 정렬 보장하지 않음
```

따라서 정렬이 필요하면

```
ORDER BY
```

를 명시해야 한다.

---

# GROUP BY 처리 방식 성능 비교

```
인덱스 스캔
   ↓
루스 인덱스 스캔
   ↓
임시 테이블
```

가능하면

> GROUP BY 컬럼에 인덱스를 생성하는 것이 가장 좋다.

# 9.2.5 DISTINCT 처리

DISTINCT는 SELECT 결과에서 **중복된 레코드를 제거**하는 기능이다.

예

```sql
SELECT DISTINCT emp_no
FROM salaries;
```

DISTINCT 처리는 크게 두 가지 방식으로 나뉜다.

```
1️⃣ SELECT DISTINCT
2️⃣ 집합 함수와 함께 사용된 DISTINCT
```

---

# 9.2.5.1 SELECT DISTINCT

SELECT DISTINCT는 **GROUP BY와 매우 유사한 방식으로 처리된다.**

예

```sql
SELECT DISTINCT first_name, last_name
FROM employees;
```

이 쿼리는

```
(first_name, last_name)
```

조합이 유니크한 레코드만 반환한다.

### 특징

- DISTINCT는 특정 컬럼이 아니라  
  **SELECT 전체 컬럼 조합에 대해 적용**

예

```
DISTINCT first_name
```

이 아니라

```
DISTINCT (first_name, last_name)
```

조합 기준으로 중복 제거

---

# DISTINCT와 GROUP BY 비교

다음 두 쿼리는 동일한 결과를 반환한다.

```sql
SELECT DISTINCT emp_no FROM salaries;
```

```sql
SELECT emp_no FROM salaries GROUP BY emp_no;
```

---

# DISTINCT 실행 방식

DISTINCT 처리는 다음 방식 중 하나로 수행된다.

```
1️⃣ 인덱스 사용
2️⃣ 임시 테이블 사용
```

가능하다면 MySQL은 **인덱스를 이용해 DISTINCT를 처리**한다.

---

# 9.2.5.2 집합 함수와 함께 사용된 DISTINCT

DISTINCT는 집합 함수에서도 사용 가능하다.

예

```sql
SELECT COUNT(DISTINCT salary)
FROM employees;
```

이 경우

```
salary 값의 중복 제거 후 COUNT
```

수행된다.

---

# DISTINCT 집합 함수 처리 방식

MySQL은 다음 과정을 수행한다.

```
1️⃣ DISTINCT 값 추출
2️⃣ 임시 테이블 저장
3️⃣ 집합 함수 계산
```

예

```sql
SELECT COUNT(DISTINCT s.salary)
FROM employees e, salaries s
WHERE e.emp_no=s.emp_no;
```

실행 계획

```
Using index
```

또는

```
Using temporary
```

---

# DISTINCT 사용 시 주의점

다음은 비효율적인 쿼리이다.

```sql
SELECT DISTINCT first_name, last_name
FROM employees;
```

하지만 실제로는

```
first_name만 유니크
```

필요한 경우가 많다.

이 경우 DISTINCT 대신

```
GROUP BY
```

또는

```
인덱스 활용
```

이 더 효율적일 수 있다.

---

# 핵심 정리

DISTINCT 처리 방식

```
1️⃣ 인덱스 기반 DISTINCT
2️⃣ 임시 테이블 기반 DISTINCT
```

집합 함수 DISTINCT

```
COUNT(DISTINCT col)
MIN(DISTINCT col)
MAX(DISTINCT col)
```

가능하면

> DISTINCT 컬럼에 인덱스를 생성하는 것이 가장 좋다.

## 9.2.6 내부 임시 테이블 활용

MySQL 옵티마이저는 **쿼리 실행 과정에서 정렬(SORT)이나 그룹화(GROUP BY), 중간 결과 저장**이 필요할 때  
자동으로 **내부 임시 테이블(Internal Temporary Table)**을 생성하여 사용한다.

이 내부 임시 테이블은 사용자가 직접 생성하는 `CREATE TEMPORARY TABLE`과는 다르며  
**쿼리 처리 과정에서 MySQL 엔진이 자동으로 생성하고 제거한다.**

---

## 1. 내부 임시 테이블의 특징

내부 임시 테이블은 다음과 같은 특징을 가진다.

- 쿼리 실행 중 **중간 결과 저장을 위해 자동 생성**
- 쿼리 종료 시 **자동으로 삭제**
- 일반 사용자 세션에서는 직접 접근 불가
- **메모리 기반 또는 디스크 기반**으로 생성됨

또한 내부 임시 테이블은 **다른 세션이나 쿼리에서 접근할 수 없다.**

---

# 9.2.6.1 메모리 임시 테이블과 디스크 임시 테이블

MySQL은 내부 임시 테이블을 다음 두 가지 방식으로 생성한다.

### 1️⃣ 메모리 임시 테이블

메모리에 생성되는 임시 테이블이며 다음 특징을 가진다.

- 빠른 처리 속도
- 메모리 크기 제한 존재
- TempTable 또는 MEMORY 스토리지 엔진 사용

MySQL 8.0 이후에는 기본적으로 **TempTable 스토리지 엔진**이 사용된다.

---

### 2️⃣ 디스크 임시 테이블

다음과 같은 경우 **디스크 기반 임시 테이블**이 생성된다.

- 임시 테이블 크기가 **메모리 한도를 초과**
- TEXT / BLOB 컬럼 사용
- 정렬 대상 데이터가 매우 큰 경우

디스크 기반 임시 테이블은 일반적으로 다음 엔진을 사용한다.

- InnoDB

---

### 내부 임시 테이블 관련 시스템 변수

| 변수 | 설명 |
|---|---|
| `tmp_table_size` | 메모리 임시 테이블 최대 크기 |
| `max_heap_table_size` | MEMORY 테이블 최대 크기 |
| `temptable_max_ram` | TempTable 엔진 메모리 제한 |
| `internal_tmp_disk_storage_engine` | 디스크 임시 테이블 스토리지 엔진 |

---

# 9.2.6.2 임시 테이블이 필요한 쿼리

다음과 같은 경우 MySQL은 내부 임시 테이블을 생성할 수 있다.

### 1️⃣ ORDER BY + GROUP BY 조합

```sql
SELECT col1, COUNT(*)
FROM table
GROUP BY col1
ORDER BY COUNT(*);
```

GROUP BY 결과를 정렬하기 위해 임시 테이블이 생성될 수 있다.

---

### 2️⃣ DISTINCT 사용

```sql
SELECT DISTINCT col1
FROM table;
```

중복 제거를 위해 임시 테이블이 사용될 수 있다.

---

### 3️⃣ UNION / UNION DISTINCT

```sql
SELECT col1 FROM t1
UNION
SELECT col1 FROM t2;
```

UNION 결과를 합치기 위해 **UNION RESULT 임시 테이블**이 생성된다.

---

### 4️⃣ 파생 테이블 (DERIVED)

```sql
SELECT *
FROM (
    SELECT col1, COUNT(*)
    FROM table
    GROUP BY col1
) AS derived_table;
```

서브쿼리 결과를 저장하기 위해 임시 테이블이 생성된다.

---

# 9.2.6.3 임시 테이블이 디스크에 생성되는 경우

내부 임시 테이블은 기본적으로 **메모리에 생성**되지만 다음 조건에서는 **디스크에 생성**된다.

### 대표적인 조건

- BLOB / TEXT 컬럼 사용
- GROUP BY 결과 크기가 큰 경우
- UNION 결과가 큰 경우
- 메모리 임시 테이블 크기 초과

또한 다음 시스템 변수 설정에 영향을 받는다.

- `tmp_table_size`
- `max_heap_table_size`
- `temptable_max_ram`

---

# 9.2.6.4 임시 테이블 관련 상태 변수

MySQL은 내부 임시 테이블 생성 여부를 확인할 수 있는 **상태 변수(status variable)**를 제공한다.

### 확인 방법

```sql
SHOW SESSION STATUS LIKE 'Created_tmp%';
```

출력 예시

| 변수 | 의미 |
|---|---|
| Created_tmp_tables | 생성된 임시 테이블 수 |
| Created_tmp_disk_tables | 디스크 기반 임시 테이블 수 |

---

### 테스트 예시

```sql
FLUSH STATUS;

SELECT first_name, last_name
FROM employees
GROUP BY first_name, last_name;

SHOW SESSION STATUS LIKE 'Created_tmp%';
```

결과 예시

| Variable_name | Value |
|---|---|
| Created_tmp_tables | 1 |
| Created_tmp_disk_tables | 0 |

---

# 정리

MySQL의 내부 임시 테이블은 다음과 같은 역할을 한다.

1️⃣ 쿼리 실행 중 **중간 결과 저장**

2️⃣ **GROUP BY / ORDER BY / DISTINCT / UNION 처리**

3️⃣ 메모리 → 디스크 자동 전환

4️⃣ 상태 변수로 **생성 여부 확인 가능**

따라서 **임시 테이블이 많이 생성되는 쿼리는 성능 저하의 원인이 될 수 있으며**,  
쿼리 튜닝 시 다음 사항을 확인해야 한다.

- 불필요한 `GROUP BY`
- `DISTINCT` 사용 여부
- `ORDER BY` 최적화
- 인덱스 활용 가능 여부

</details>

# 09. 옵티마이저와 힌트

MySQL 서버는 동일한 SQL이라도 **내부적으로 다양한 실행 방법**을 선택할 수 있다.  
즉, 동일한 결과를 반환하더라도 **어떤 순서로 테이블을 읽고 어떤 인덱스를 사용할지**에 따라 성능이 크게 달라질 수 있다.

이러한 실행 방법을 결정하는 역할을 하는 것이 바로 **옵티마이저(Optimizer)** 이다.

옵티마이저는 테이블의 통계 정보, 인덱스 구조, 조건식 등을 분석하여  
**가장 비용이 낮은 실행 계획(Execution Plan)** 을 선택한다.

MySQL에서는 `EXPLAIN` 명령어를 통해 옵티마이저가 선택한 실행 계획을 확인할 수 있다.

---

# 9.1 개요

옵티마이저는 DBMS에서 **가장 복잡하면서도 중요한 구성 요소** 중 하나다.

쿼리 실행 계획을 이해하지 못하면 다음과 같은 문제가 발생한다.

- 느린 쿼리의 원인을 파악하기 어렵다
- 잘못된 인덱스 설계
- 불필요한 테이블 스캔 발생
- 조인 순서로 인한 성능 저하

따라서 **실행 계획을 이해하는 것은 성능 튜닝의 핵심 요소**이다.

---

# 9.1.1 쿼리 실행 절차

MySQL에서 SQL이 실행되는 과정은 크게 **3단계**로 나눌 수 있다.

## 1️⃣ SQL 파싱 (Parsing)

사용자가 전달한 SQL 문장을 MySQL 서버가 이해할 수 있는 구조로 변환한다.

이 단계에서 수행되는 작업

- SQL 문법 검사
- SQL 파스 트리(SQL Parse Tree) 생성

MySQL 서버는 SQL 문자열 자체를 직접 실행하지 않고  
**파스 트리 구조를 기반으로 실행**한다.

---

## 2️⃣ 최적화 (Optimization)

옵티마이저가 실행 계획을 생성하는 단계이다.

옵티마이저는 다음과 같은 요소를 고려한다.

### 테이블 접근 방법 결정

- 테이블 풀 스캔
- 인덱스 스캔
- 인덱스 범위 스캔

### 조인 순서 결정

여러 테이블을 조인하는 경우

- 어떤 테이블을 먼저 읽을지
- 어떤 인덱스를 사용할지

를 결정한다.

### 인덱스 선택

각 테이블에서 사용할 수 있는 인덱스 중  
**가장 효율적인 인덱스**를 선택한다.

### 임시 테이블 사용 여부

정렬이나 그룹핑이 필요한 경우

- 임시 테이블 생성 여부
- 임시 테이블 저장 위치

를 결정한다.

---

## 3️⃣ 실행 (Execution)

옵티마이저가 생성한 실행 계획을 기반으로 실제 데이터를 읽는다.

이 단계에서는

- 스토리지 엔진(InnoDB 등)에서 레코드 조회
- 조인 수행
- 정렬 및 그룹 처리

가 진행된다.

즉,

```
SQL 입력
   ↓
파싱 (Parser)
   ↓
최적화 (Optimizer)
   ↓
실행 (Execution Engine + Storage Engine)
```

---

# 9.1.2 옵티마이저의 종류

옵티마이저는 크게 두 가지 방식으로 나눌 수 있다.

## Rule-Based Optimizer (RBO)

규칙 기반 옵티마이저

특징

- 정해진 규칙에 따라 실행 계획 선택
- 통계 정보를 사용하지 않음

예시

- 항상 인덱스를 우선 사용
- 항상 특정 순서로 조인

문제점

- 데이터 분포를 고려하지 않음
- 실제 성능과 맞지 않는 실행 계획 선택 가능

현재 대부분의 DBMS에서는 사용되지 않는다.

---

## Cost-Based Optimizer (CBO)

비용 기반 옵티마이저

특징

- 여러 실행 계획을 생성
- 각 계획의 **비용(cost)** 을 계산
- 가장 비용이 낮은 계획 선택

비용 계산에 사용되는 정보

- 테이블 레코드 수
- 인덱스 선택도(selectivity)
- 데이터 분포
- 디스크 I/O 비용

현재 대부분의 RDBMS (MySQL 포함)는  
**Cost-Based Optimizer(CBO)** 를 사용한다.

---

# 핵심 정리

### SQL 실행 흐름

```
1. Parsing
   - SQL 문법 검사
   - Parse Tree 생성

2. Optimization
   - 실행 계획 생성
   - 조인 순서 결정
   - 인덱스 선택
   - 접근 방법 결정

3. Execution
   - 스토리지 엔진에서 데이터 조회
   - 결과 반환
```

---

### 옵티마이저 종류

| 구분 | 특징 |
|-----|-----|
| Rule-Based Optimizer | 규칙 기반 실행 계획 |
| Cost-Based Optimizer | 비용 기반 실행 계획 |

현재 대부분의 DBMS는 **Cost-Based Optimizer**를 사용한다.

---

# 정리

옵티마이저는

- SQL 실행 계획을 생성하고
- 가장 비용이 낮은 실행 방법을 선택하며
- 데이터 접근 방법, 인덱스 사용, 조인 순서를 결정한다.

따라서 **옵티마이저의 동작을 이해하는 것은 SQL 성능 튜닝의 핵심이다.**

# 10-2 실행 계획 (Execution Plan)

MySQL에서 쿼리 성능을 이해하고 개선하려면  
👉 반드시 실행 계획(EXPLAIN)을 읽을 수 있어야 한다.

---

## 🔥 핵심 개념

- 실행 계획 = 옵티마이저의 판단 결과
- "어떻게 데이터를 읽을 것인가"를 보여줌
- 인덱스 사용 여부, 조인 순서, 비용까지 확인 가능

---

## 실행 계획 확인 방법

- EXPLAIN
- DESC (동일 기능)

---

## 10-2-1 실행 계획 출력 포맷

MySQL 8.0부터는 실행 계획을 다양한 형태로 출력 가능

---

## 1. 기본 테이블 형식

EXPLAIN  
SELECT ...;

---

### 특징

- 가장 기본적인 형태
- 컬럼 기반 출력
- 빠르게 확인 가능

---

### 주요 컬럼

- id → 실행 단위
- select_type → 쿼리 유형
- table → 접근 테이블
- type → 접근 방식 (⭐ 중요)
- key → 사용된 인덱스
- rows → 예상 읽을 레코드 수
- filtered → 조건 필터링 비율
- Extra → 추가 정보

---

## 2. TREE 포맷 (⭐ 추천)

```sql
EXPLAIN FORMAT=TREE  
SELECT ...;
```

---

### 특징

- 실행 흐름을 트리 형태로 표현
- 실제 실행 순서 직관적으로 확인 가능

---

### 예시 구조

Nested loop  
→ Index lookup on e  
→ Index lookup on s

---

👉 읽는 방법

- 위 → 아래
- 들여쓰기 = 실행 순서

---

## 3. JSON 포맷

```sql
EXPLAIN FORMAT=JSON  
SELECT ...;
```

---

### 특징

- 가장 상세한 정보 제공
- 내부 비용 정보 포함

---

### 주요 정보

- query_cost → 전체 비용
- access_type → 접근 방식
- possible_keys → 후보 인덱스
- used_key_parts → 사용된 컬럼

---

## 🔥 포맷 선택 기준

- 빠르게 보기 → TABLE
- 실행 흐름 이해 → TREE ⭐
- 깊은 분석 → JSON

## 10-2-2 EXPLAIN ANALYZE

MySQL 8.0.18부터 제공

👉 실행 계획 + 실제 실행 결과

---

## 🔥 기존 EXPLAIN vs ANALYZE

| 구분 | EXPLAIN | EXPLAIN ANALYZE |
|------|--------|----------------|
| 실행 여부 | X | O |
| 시간 정보 | X | O |
| 정확도 | 추정 | 실제 |

---

## 사용 예

```sql
EXPLAIN ANALYZE  
SELECT ...
```

---

## 출력 정보

각 단계마다:

- actual time → 실제 수행 시간
- rows → 처리된 레코드 수
- loops → 반복 횟수

---

## 📌 핵심 필드 해석

### actual time

예:
actual time=0.007..0.009

- 첫 번째 값 → 첫 row 처리 시간
- 두 번째 값 → 마지막 row 처리 시간

---

### rows

- 해당 단계에서 처리한 레코드 수

---

### loops

- 해당 작업 반복 횟수

👉 중요: rows × loops = 실제 처리량

---

## 🔥 실행 순서 읽는 법

TREE 포맷 기준:

1. 들여쓰기 기준으로 읽기
2. 안쪽(하위)부터 실행
3. 위로 올라오며 결과 결합

---

## 📌 실제 실행 흐름 예시

1. 인덱스로 employees 조회
2. PK로 salaries 조회
3. 조건 필터 적용
4. 조인 수행
5. 임시 테이블 생성
6. 결과 반환

---

## 🔥 핵심 인사이트

👉 EXPLAIN = 예상  
👉 ANALYZE = 실제

👉 둘의 차이가 크면 문제 있음

---

## ⚠️ 주의

- 실제 쿼리 실행됨
- 운영 DB에서는 주의

## 10-2-3 실행 계획 분석

실행 계획을 보는 목적:

👉 "이 쿼리가 왜 느린지 찾기"

---

## 🔥 가장 중요한 컬럼 (면접 ⭐)

### 1. type (접근 방식)

성능 순서:

- system / const → 최고
- eq_ref → 매우 좋음
- ref → 좋음
- range → 괜찮음
- index → 풀 인덱스 스캔
- ALL → 풀 테이블 스캔 (❌)

---

👉 핵심:

"ALL 나오면 무조건 의심"

---

### 2. key

- 실제 사용된 인덱스

👉 NULL이면 인덱스 안 씀

---

### 3. rows

- 예상 읽을 데이터 수

👉 많을수록 성능 안 좋음

---

### 4. filtered

- WHERE 조건 필터 비율

---

## 🔥 실행 순서 이해

- id 값이 클수록 먼저 실행
- 결과는 아래 → 위로 결합

---

## 📌 id 컬럼 의미

- SELECT 단위 식별자
- 서브쿼리 존재 시 여러 개 생성

---

## 🔥 실행 순서 규칙

- 같은 레벨 → 위에서 아래
- 다른 레벨 → 안쪽 먼저 실행

---

## 📌 조인 순서 판단

- 먼저 읽는 테이블이 중요
- 작은 데이터 → 먼저 읽기

---

## 🔥 성능 분석 체크리스트

1. type이 ALL인가?
2. key가 NULL인가?
3. rows가 비정상적으로 큰가?
4. filtered가 너무 낮은가?
5. 조인 순서가 비효율적인가?

---

## 🔥 실무 최적화 전략

### 1. 인덱스 추가

- WHERE
- JOIN
- ORDER BY

---

### 2. 조건 변경

- 함수 제거
- 컬럼 변환 제거

---

### 3. 데이터 분포 개선

- 히스토그램 활용

---

### 4. 통계 정보 갱신

ANALYZE TABLE ...

---

## 🔥 핵심 연결

통계 정보 → 선택도 계산

선택도 → 비용 계산

비용 → 실행 계획

---

## 🔥 최종 한 줄 정리

👉 "실행 계획은 옵티마이저의 생각을 보여주는 창이다"

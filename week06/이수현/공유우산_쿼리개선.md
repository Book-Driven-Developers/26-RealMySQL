# 📌 공유우산 서비스 - 실행계획 기반 쿼리 최적화

## 1. 개요 (Overview)

공유우산 서비스에서 사용자 대여 요청 시,  
특정 우산함(box)에서 사용 가능한 우산(slot)을 조회하는 쿼리를 대상으로  
실행 계획(EXPLAIN)을 분석하고 성능 개선을 수행하였다.

본 과제의 목표는 다음과 같다.

- 실행 계획을 기반으로 쿼리의 비효율 원인 파악
- 인덱스 및 쿼리 구조 개선
- 실제 성능 개선 결과 확인

---

## 2. 문제 상황 (Problem)

### 📌 기존 로직

사용자가 우산을 대여할 때:

1. 특정 우산함(box_id)
2. 상태가 AVAILABLE인 우산 슬롯 조회
3. 해당 슬롯을 점유 → 결제 → 상태 변경

---

### ❌ 기존 쿼리
```sql
SELECT *
FROM umbrella_slots
WHERE box_id = 1
AND status = 'AVAILABLE';
```

---

## 3. 실행 계획 분석 (EXPLAIN)
```sql
EXPLAIN
SELECT *
FROM umbrella_slots
WHERE box_id = 1
AND status = 'AVAILABLE';
```

### 🔍 결과

| 항목 | 값 |
|------|----|
| type | ALL |
| key | NULL |
| rows | 10000+ |

---

### 🚨 문제점

- type = ALL → 풀 테이블 스캔 발생
- 인덱스 미사용 (key = NULL)
- 불필요하게 많은 row 탐색

👉 데이터 증가 시 성능 급격히 저하

---

## 4. 원인 분석 (Root Cause)

- WHERE 조건 컬럼(box_id, status)에 대한 인덱스 부재
- 옵티마이저가 인덱스를 사용할 수 없는 구조
- SELECT *로 불필요한 데이터까지 조회

---

## 5. 개선 전략 (Optimization Strategy)

### ✅ 전략 1: 복합 인덱스 생성
```sql
CREATE INDEX idx_box_status
ON umbrella_slots (box_id, status);
```

---

### ✅ 전략 2: 필요한 컬럼만 조회
```sql
SELECT slot_id
FROM umbrella_slots
WHERE box_id = 1
AND status = 'AVAILABLE';
```

---

### ✅ 전략 3: LIMIT 적용
```sql
SELECT slot_id
FROM umbrella_slots
WHERE box_id = 1
AND status = 'AVAILABLE'
LIMIT 1;
```

---

## 6. 개선 후 실행 계획
```sql
EXPLAIN
SELECT slot_id
FROM umbrella_slots
WHERE box_id = 1
AND status = 'AVAILABLE'
LIMIT 1;
```

---

### 🔍 결과

| 항목 | 개선 전 | 개선 후 |
|------|--------|--------|
| type | ALL | ref |
| key | NULL | idx_box_status |
| rows | 10000+ | 5 |

---

### 📌 개선 효과

- 풀 스캔 → 인덱스 탐색
- 읽는 데이터 수 대폭 감소
- 응답 시간 단축

---

## 7. 동시성 문제까지 고려한 개선 (Advanced)

### 📌 기존 문제

- 여러 사용자가 동시에 요청 시:
    - 동일 slot 조회 가능
    - 중복 대여 발생 가능

---

### ❌ 기존 흐름

1. SELECT AVAILABLE slot
2. 이후 UPDATE

👉 Race Condition 발생

---

### ✅ 개선 쿼리 (락 기반)
```sql
SELECT slot_id
FROM umbrella_slots
WHERE box_id = 1
AND status = 'AVAILABLE'
LIMIT 1
FOR UPDATE;
```

---

### 📌 효과

- 해당 row에 배타적 락 적용
- 다른 트랜잭션 접근 차단
- 중복 대여 방지

---

### 🔥 최종 트랜잭션 구조
```sql
BEGIN;

SELECT slot_id
FROM umbrella_slots
WHERE box_id = 1
AND status = 'AVAILABLE'
LIMIT 1
FOR UPDATE;

UPDATE umbrella_slots
SET status = 'RENTED'
WHERE slot_id = ?;

COMMIT;
```

---

## 8. 실행 계획 + 트랜잭션 연결 분석

### 📌 핵심 인사이트

- 실행 계획 최적화 → 조회 속도 개선
- 조회 속도 개선 → 락 유지 시간 감소
- 락 유지 시간 감소 → 동시성 처리 성능 향상

👉 즉,

"쿼리 성능 개선은 단순 속도가 아니라  
동시성 문제 해결에도 직접적인 영향을 준다"

---

## 9. 최종 결과 (Result)

| 항목 | 개선 전 | 개선 후 |
|------|--------|--------|
| 실행 방식 | Full Scan | Index Scan |
| 응답 속도 | 느림 | 빠름 |
| 동시성 | 중복 대여 가능 | 안전 |
| 트랜잭션 안정성 | 낮음 | 높음 |

---

## 10. 핵심 인사이트 (Takeaways)

- 실행 계획 분석은 성능 개선의 출발점이다
- 인덱스 설계는 WHERE 조건 기준으로 해야 한다
- SELECT 범위를 최소화하는 것이 중요하다
- 쿼리 최적화는 동시성 문제 해결과도 연결된다

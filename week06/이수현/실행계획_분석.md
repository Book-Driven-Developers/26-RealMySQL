# 10-3 실행 계획 분석

MySQL에서 실행 계획을 확인하는 것(EXPLAIN)은 시작일 뿐이다.  
👉 진짜 중요한 것은 "해석"이다.

---

## 🔥 핵심 개념

- 실행 계획 = 옵티마이저의 선택 결과
- 실행 계획 분석 = 성능 문제 해결의 출발점

---

## 실행 계획 분석의 목적

- 어떤 인덱스를 사용했는가?
- 어떤 순서로 테이블을 읽는가?
- 왜 이 실행 계획이 선택되었는가?

👉 결국:

"왜 느린지 설명할 수 있어야 한다"

---

## 출력 포맷보다 중요한 것

MySQL 8.0에서는 다양한 포맷 지원

- TABLE
- TREE
- JSON

---

하지만 중요한 건:

👉 **포맷이 아니라 실행 방식 이해**

---

## 🔥 핵심 관점

실행 계획 분석은 다음 3가지 관점에서 봐야 한다:

1. 접근 방법 (Index vs Full Scan)
2. 조인 순서
3. 예상 vs 실제 데이터량

---

## 10-3-1 id 컬럼

id 컬럼은 실행 계획에서  
👉 **SELECT 단위(쿼리 블록)를 구분하는 식별자**

---

## 🔥 핵심 정의

👉 id = "쿼리 실행 단위 번호"

---

## 1. 기본 동작

하나의 SELECT 문 → 하나의 id

---

### 예시

SELECT * FROM employees;

→ id = 1

---

## 2. 조인 시 id

조인은 하나의 SELECT이므로  
👉 같은 id 사용

---

### 예시

SELECT *
FROM employees e  
JOIN salaries s ON e.emp_no = s.emp_no;

---

결과:

- employees → id=1
- salaries → id=1

👉 같은 SELECT이기 때문

---

## 3. 서브쿼리 존재 시

👉 SELECT가 여러 개 → id도 여러 개

---

### 예시
```sql
SELECT *
FROM dept_emp de  
WHERE de.emp_no = (
SELECT e.emp_no  
FROM employees e  
WHERE e.first_name='Georgi'
);
```

---

결과:

- PRIMARY 쿼리 → id=1
- SUBQUERY → id=2

---

## 🔥 중요한 포인트

👉 id는 실행 순서를 의미하지 않는다 ❌

---

## 4. 실행 순서 vs id

많이 하는 오해:

"id 작은 게 먼저 실행된다"

👉 ❌ 틀림

---

## 실제 실행 순서

👉 옵티마이저가 결정

---

### 예시 해석

위 쿼리 실제 실행:

1. employees 서브쿼리 먼저 실행
2. 결과를 사용해서 dept_emp 조회

---

👉 하지만 id는:

- dept_emp → id=1
- employees → id=2

---

## 🔥 핵심 결론

👉 id = 실행 순서 X  
👉 id = 쿼리 블록 구분 O

---

## 5. id 해석 방법

### 기본 규칙

- 같은 id → 같은 SELECT
- 다른 id → 다른 SELECT (서브쿼리)

---

### 해석 순서

1. id로 쿼리 구조 파악
2. select_type으로 역할 파악
3. TREE로 실제 실행 순서 확인

---

## 📌 실무 팁

👉 id만 보고 실행 순서 판단하지 말 것

👉 반드시:

- TREE 포맷
- EXPLAIN ANALYZE

로 실제 흐름 확인

---

## 🔥 핵심 요약

- id 컬럼은 SELECT 단위를 구분하는 값
- 조인은 같은 id
- 서브쿼리는 다른 id
- 실행 순서와는 무관

---

## 🔥 한 줄 정리

👉 "id는 실행 순서가 아니라 쿼리 구조를 보여준다"

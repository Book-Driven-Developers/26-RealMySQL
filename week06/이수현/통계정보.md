# 10. 통계 정보

MySQL에서 실행 계획(Execution Plan)을 제대로 이해하려면,  
옵티마이저가 어떤 근거로 실행 계획을 수립하는지 먼저 이해해야 한다.

옵티마이저는 SQL을 가장 효율적으로 실행하기 위해 다양한 실행 경로를 비교하고,  
그중 비용(Cost)이 가장 낮다고 판단되는 계획을 선택한다.

이때 중요한 기준이 바로 **통계 정보(Statistics)**이다.

---

## 핵심 개념

- 실행 계획의 정확도 = 통계 정보의 정확도
- 통계 정보가 틀리면 실행 계획도 틀린다
- 결국 성능 문제의 원인은 실행 계획보다 **통계 정보일 가능성이 높다**

---

## 왜 중요한가?

통계 정보가 부정확하면:

- 인덱스를 써야 할 쿼리 → 풀 테이블 스캔 선택
- 선택도 잘못 계산 → 비효율적인 실행 순서
- 조인 순서 비효율 발생
- 쿼리 성능 급격히 저하

## 10-1. 통계 정보

MySQL 5.7까지는 **단순한 통계 정보**만으로 실행 계획을 수립했다.

즉,
- 테이블 크기
- 인덱스 크기
- 대략적인 레코드 수

👉 하지만 "데이터 분포"는 알지 못했다.

---

## 문제점

예를 들어:

- 실제 데이터: 1억 건
- 통계 정보: 10건으로 인식

👉 옵티마이저는 완전히 잘못된 실행 계획 선택

---

## MySQL 8.0의 개선

MySQL 8.0부터는 다음 두 가지로 개선됨:

### 1) 기본 통계 정보
- 레코드 수
- 인덱스 크기
- 유니크 값 수

### 2) 히스토그램
- 컬럼 값 분포까지 반영

👉 단순 규모 → 실제 데이터 분포 기반으로 진화

---

## 핵심 요약

- 옵티마이저는 비용 기반으로 동작한다
- 비용 계산의 핵심 = 통계 정보
- 통계 정보가 틀리면 실행 계획도 틀린다

## 10-1-1. 테이블 및 인덱스 통계 정보

옵티마이저가 가장 먼저 사용하는 정보:

- 테이블 레코드 수
- 인덱스 크기
- 유니크 값 개수

---

## 1. MySQL 5.7까지의 한계

- 단순한 통계 정보만 존재
- 데이터 분포 반영 불가능
- 실행 계획 오류 발생 가능

---

## 2. MySQL 5.6 이후 변화 (핵심 ⭐)

👉 통계 정보 **영구 저장(Persistent)** 가능

### 이전 (5.5 이하)
- 메모리에만 존재
- 서버 재시작 시 초기화

### 이후 (5.6+)
- 시스템 테이블에 저장

#### 저장 위치
- mysql.innodb_index_stats
- mysql.innodb_table_stats

👉 재시작해도 유지됨

---

## 3. STATS_PERSISTENT 옵션

테이블 생성 시 설정 가능

```sql
CREATE TABLE tab_test (
fd1 INT,
fd2 VARCHAR(20),
PRIMARY KEY(fd1)
) ENGINE=InnoDB
STATS_PERSISTENT=1;
```

### 옵션 의미

- 0 → 비영구 (메모리)
- 1 → 영구 저장
- DEFAULT → 시스템 설정 따름

👉 보통 기본값 = 1 (ON)

---

## 4. 통계 정보 확인

### 인덱스 통계

```sql
SELECT *
FROM innodb_index_stats
WHERE database_name='employees'
AND table_name='employees';

확인 가능:
- 유니크 값 수
- 인덱스 페이지 수
- 트리 크기
```

---

### 테이블 통계

SELECT *
FROM innodb_table_stats
WHERE database_name='employees'
AND table_name='employees';

확인 가능:
- 전체 레코드 수
- PK 크기
- 인덱스 크기

---

## 5. 자동 갱신 (주의 ⚠️)

다음 상황에서 자동 갱신됨:

- 테이블 오픈
- 대량 데이터 변경
- ANALYZE TABLE 실행
- SHOW TABLE STATUS 실행

---

### 문제점

- 실행 계획이 갑자기 바뀜
- 성능이 예측 불가능
- 장애 원인 분석 어려움

---

## 6. STATS_AUTO_RECALC

자동 갱신 제어 옵션

- 1 → 자동 갱신
- 0 → 수동 (ANALYZE TABLE 필요)
- DEFAULT → 시스템 설정

👉 운영 환경에서는 수동 제어도 고려

---

## 7. 샘플링 기반 통계

InnoDB는 전체 데이터를 보지 않음

👉 일부 페이지만 샘플링

### 관련 변수

- innodb_stats_transient_sample_pages (기본 8)
- innodb_stats_persistent_sample_pages (기본 20)

---

### 특징

- 많이 샘플링 → 정확 ↑
- 적게 샘플링 → 속도 ↑

👉 트레이드오프 존재

---

## 8. ANALYZE TABLE

통계 정보 수동 갱신

```sql
ANALYZE TABLE employees.employees;
```

---

### 언제 사용?

- 대량 데이터 변경 후
- 인덱스 변경 후
- 실행 계획 이상할 때

---

## 9. 핵심 정리 (면접 포인트 ⭐)

- 실행 계획은 통계 정보 기반
- MySQL 5.6부터 영구 통계 도입
- 자동 갱신은 편하지만 위험
- ANALYZE TABLE로 제어 가능
- 샘플링 기반이라 완벽하지 않음

👉 결국 "통계 정보 이해 = 실행 계획 이해"

## 10-1-2. 히스토그램

MySQL 5.7까지의 통계 정보는 매우 제한적이었다.

- 인덱스 유니크 값 개수 정도만 파악 가능
- 컬럼 값의 **실제 분포는 알 수 없음**

👉 즉, 옵티마이저는 "데이터가 균등하다"고 가정

---

## 🔥 왜 히스토그램이 필요한가?

실제 데이터는 절대 균등하지 않다.

예:
- 특정 값에 데이터가 몰림 (편향 데이터)
- 일부 범위에만 값이 존재

👉 기존 방식 문제

- WHERE 조건 선택도 오판
- 실행 계획 비효율
- 인덱스 선택 오류

---

## ✅ 히스토그램의 핵심 역할

👉 컬럼 값의 **분포 정보 저장**

- 값이 얼마나 자주 등장하는지
- 어떤 범위에 데이터가 몰려있는지

👉 옵티마이저가 더 정확한 비용 계산 가능

---

## 1. 히스토그램 생성 및 조회

### 생성

ANALYZE TABLE employees.employees  
UPDATE HISTOGRAM ON gender, hire_date;

---

### 조회
```sql
SELECT *
FROM information_schema.column_statistics  
WHERE schema_name='employees'  
AND table_name='employees';
```

---

## 2. 히스토그램 저장 구조

히스토그램은 JSON 형태로 저장됨

예시 (gender 컬럼):

- buckets (버킷 정보)
- sampling-rate (샘플링 비율)
- histogram-type (타입)
- number-of-buckets (버킷 개수)

---

## 3. 히스토그램 타입 (⭐ 중요)

MySQL 8.0 → 2가지 지원

---

### ① Singleton Histogram

👉 값 하나당 버킷 하나

- value 기반
- 값 종류가 적을 때 사용 (ENUM 등)

특징:
- 정확도 높음
- 값별 분포 그대로 표현

---

### ② Equi-Height Histogram

👉 버킷마다 **레코드 수 동일**

- 범위 기반
- 값 종류 많을 때 사용

특징:
- 범위를 나눠서 관리
- 각 버킷의 데이터 수가 동일

---

## 📊 구조 차이

### Singleton
- 값 단위
- (값, 비율)

### Equi-Height
- 범위 단위
- (시작값, 끝값, 비율, 개수)

---

## 4. 주요 필드 의미

- sampling-rate  
  → 전체 데이터 중 샘플링 비율

- histogram-type  
  → singleton / equi-height

- number-of-buckets  
  → 버킷 개수 (기본 100)

---

## ⚠️ 주의

- 히스토그램은 자동 생성되지 않음
- 반드시 ANALYZE TABLE로 수동 생성

---

## 5. 히스토그램 삭제

```sql
ANALYZE TABLE employees.employees  
DROP HISTOGRAM ON gender, hire_date;
```

---

## 6. 히스토그램 비활성화

### 전체 비활성화

SET GLOBAL optimizer_switch='condition_fanout_filter=off';

---

### 세션 단위

SET SESSION optimizer_switch='condition_fanout_filter=off';

---

### 특정 쿼리만

SELECT /*+ SET_VAR(optimizer_switch='condition_fanout_filter=off') */  
...

---

## 7. 히스토그램의 효과 (핵심 ⭐)

### 기존 (히스토그램 없음)

- 옵티마이저:
  👉 "데이터 균등 분포" 가정

- 결과:
  👉 rows / filtered 부정확

---

### 적용 후

- 실제 데이터 분포 반영
- rows 예측 정확도 증가
- 실행 계획 개선

---

## 📌 실제 예시

조건:
```sql
WHERE first_name='Zita'  
AND birth_date BETWEEN '1950-01-01' AND '1960-01-01'
```
---

### 히스토그램 없음

- filtered ≈ 11.11%
- 완전 틀린 추정

---

### 히스토그램 적용

- filtered ≈ 60.82%

---

### 실제 값

- ≈ 63.84%

👉 훨씬 정확해짐

---

## 🔥 핵심 결론

- 히스토그램 = "데이터 분포 인식"
- 실행 계획 정확도 대폭 향상
- 특히 비균등 데이터에서 효과 큼

## 히스토그램의 용도

히스토그램이 없으면:

👉 옵티마이저는 항상 균등 분포 가정

---

## 문제 상황

- 특정 값에 데이터 집중
- 범위별 데이터 편차 존재

👉 기존 통계로는 절대 예측 불가능

---

## 히스토그램 적용 효과

- 특정 범위의 데이터 밀도 파악 가능
- WHERE 조건 정확도 증가
- 조인 순서 최적화 가능

---

## 🔥 조인 최적화 영향

히스토그램이 있으면:

👉 어떤 테이블을 먼저 읽을지 정확하게 판단

---

### 예시 (핵심)

같은 쿼리인데:

- employees 먼저 읽기 → 빠름
- salaries 먼저 읽기 → 느림

👉 히스토그램 없으면 판단 못함

---

## 결과

- 실행 계획 차이 발생
- 최대 10배 이상 성능 차이 가능

---

## 핵심 정리

- 히스토그램 = 선택도 정확도 개선
- 선택도 정확도 = 실행 계획 품질
- 실행 계획 품질 = 성능

👉 결국 성능 핵심 요소

## 히스토그램 vs 인덱스

둘은 완전히 다른 개념

---

## 인덱스

- 실제 데이터 접근 구조
- 검색 속도 개선

---

## 히스토그램

- 통계 정보
- 실행 계획 결정에 사용

---

## 🔥 중요한 차이

### 인덱스 존재 시

- 옵티마이저는 **Index Dive 사용**
- 실제 B-Tree 샘플링

👉 히스토그램보다 더 정확

---

### 인덱스 없는 경우

👉 히스토그램 사용

---

## 📌 핵심 전략

- 인덱스 있으면 → 히스토그램 필요 없음
- 인덱스 없으면 → 히스토그램 매우 중요

---

## ❗ 실무 포인트

👉 히스토그램은

"인덱스 없는 컬럼의 통계 보완용"

---

## 최종 정리

- 히스토그램 = 컬럼 분포 정보
- 선택도 계산 정확도 향상
- 실행 계획 품질 개선
- 인덱스 없는 컬럼에서 특히 중요

## 10-1-3. 코스트 모델 (Cost Model)

옵티마이저는 여러 실행 계획 중  
👉 **가장 비용(Cost)이 낮은 계획을 선택**한다.

이때 사용하는 기준이 바로 **코스트 모델**이다.

---

## 🔥 코스트 모델이란?

👉 쿼리 실행에 필요한 작업들을  
👉 "비용"으로 수치화한 것

---

## 1. MySQL이 고려하는 작업들

쿼리 실행 시 발생하는 작업:

- 디스크에서 데이터 읽기
- 메모리(InnoDB 버퍼 풀)에서 데이터 읽기
- 인덱스 키 비교
- 레코드 조건 평가
- 임시 테이블 생성
- 임시 테이블 읽기

👉 이 모든 작업에 "비용"이 존재

---

## 2. 코스트 모델의 동작 방식

옵티마이저는:

1. 가능한 실행 계획 생성
2. 각 계획의 비용 계산
3. 가장 비용이 낮은 계획 선택

👉 즉, "정답"이 아니라  
👉 "가장 싸 보이는 것" 선택

---

## 3. MySQL 5.7 이전 vs 이후

### 이전 (5.7 이전)

- 비용이 코드에 하드코딩
- 환경 반영 불가능
- 유연성 부족

---

### 이후 (5.7+ / 8.0)

👉 비용을 테이블로 관리

- server_cost
- engine_cost

👉 DBA가 조정 가능

---

## 4. 코스트 테이블 구조

### server_cost

👉 서버 레벨 작업 비용

- 인덱스 비교
- 레코드 평가
- 임시 테이블 작업

---

### engine_cost

👉 스토리지 엔진 작업 비용

- 디스크 읽기
- 메모리 읽기

---

## 📌 컬럼 구조

공통 컬럼:

- cost_name → 작업 이름
- default_value → 기본 비용
- cost_value → 실제 적용 값
- last_updated → 변경 시점
- comment → 설명

---

추가 컬럼 (engine_cost):

- engine_name → 스토리지 엔진
- device_type → 디스크 타입

---

## 5. 주요 비용 항목 (⭐ 중요)

### 디스크 vs 메모리

- io_block_read_cost → 디스크 읽기 (비쌈)
- memory_block_read_cost → 메모리 읽기 (저렴)

---

### 임시 테이블

- disk_temptable_create_cost
- disk_temptable_row_cost

- memory_temptable_create_cost
- memory_temptable_row_cost

---

### 기타

- key_compare_cost → 인덱스 비교
- row_evaluate_cost → WHERE 조건 평가

---

## 🔥 핵심 개념

👉 디스크 접근 = 비싸다  
👉 메모리 접근 = 싸다

👉 그래서 옵티마이저는  
👉 "메모리 기반 처리"를 선호

---

## 7. 코스트 모델의 한계

⚠️ 중요한 포인트

- 옵티마이저는 실제 비용을 계산하지 않음
- 추정 기반 계산

👉 이유:

- 모든 데이터를 직접 계산하면 너무 느림

---

👉 대신:

- 인덱스 B-Tree 샘플링
- 통계 정보 활용
- 히스토그램 활용

---

## 8. 비용 조정 효과 (핵심 ⭐)

비용 값을 변경하면 실행 계획이 바뀜

---

### 예시

#### key_compare_cost 증가

→ 정렬 회피  
→ 인덱스 사용 증가

---

#### row_evaluate_cost 증가

→ 풀 스캔 회피  
→ 인덱스 사용 증가

---

#### disk_temptable 비용 증가

→ 디스크 임시 테이블 사용 회피

---

#### memory_temptable 비용 증가

→ 메모리 임시 테이블 사용 회피

---

#### io_block_read_cost 증가

→ 디스크 읽기 회피  
→ 인덱스 사용 증가

---

#### memory_block_read_cost 증가

→ 메모리 접근 감소  
→ 다른 실행 계획 선택

---

## 🔥 핵심 인사이트

👉 "비용을 바꾸면 실행 계획이 바뀐다"

- 디스크 > 메모리 비용

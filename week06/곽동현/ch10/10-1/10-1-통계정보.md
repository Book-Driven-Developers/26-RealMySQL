## 10.1 통계 정보

MySQL 서버는 옵티마이저가 최적의 실행 계획을 수립할 수 있도록 통계 정보를 관리한다.
MySQL 5.7까지는 테이블과 인덱스에 대한 개괄적인 정보만으로 실행 계획을 수립했지만,
MySQL 8.0부터는 **히스토그램(Histogram)** 정보가 도입되어 더 정확한 실행 계획 수립이 가능해졌다.

---

### 10.1.1 테이블 및 인덱스 통계 정보

비용 기반 최적화에서 가장 중요한 것은 **통계 정보**이다.
통계 정보가 정확하지 않으면 전혀 엉뚱한 방향으로 쿼리를 실행할 수 있다.

> **예시**: 1억 건의 레코드가 저장된 테이블의 통계 정보가 갱신되지 않아 레코드가 10건 미만인 것처럼 보이면,
> 옵티마이저는 인덱스 레인지 스캔 대신 **풀 테이블 스캔**으로 실행해버릴 수 있다.

---

#### 10.1.1.1 MySQL 서버의 통계 정보

**MySQL 5.5 이전**
- 각 테이블의 통계 정보가 **메모리에만** 관리되었다.
- `SHOW INDEX` 명령으로만 인덱스 칼럼의 분포도를 볼 수 있었다.
- MySQL 서버가 재시작되면 수집된 통계 정보가 **모두 사라졌다.**

**MySQL 5.6 이후 — 영구적(Persistent) 통계 정보**
- InnoDB 스토리지 엔진을 사용하는 테이블의 통계 정보를 **영구적으로** 관리할 수 있게 개선됐다.
- `mysql` 데이터베이스의 `innodb_index_stats`와 `innodb_table_stats` 테이블에 저장된다.

```sql
mysql> SHOW TABLES LIKE '%_stats';
+----------------------------+
| Tables_in_mysql (%_stats)  |
+----------------------------+
| innodb_index_stats         |
| innodb_table_stats         |
+----------------------------+
```

**STATS_PERSISTENT 옵션** (테이블 생성 시 설정 가능)

| 값 | 동작 |
|----|------|
| `0` | MySQL 5.5 이전 방식대로 메모리에만 관리 (innodb_*_stats 테이블에 저장 안 함) |
| `1` | innodb_index_stats와 innodb_table_stats 테이블에 저장 |
| `DEFAULT` | `innodb_stats_persistent` 시스템 변수값으로 결정 (기본값 ON) |

```sql
-- 영구 통계 설정
mysql> CREATE TABLE tab_persistent (fd1 INT PRIMARY KEY, fd2 INT)
       ENGINE=InnoDB STATS_PERSISTENT=1;

-- 단기 통계 설정
mysql> CREATE TABLE tab_transient (fd1 INT PRIMARY KEY, fd2 INT)
       ENGINE=InnoDB STATS_PERSISTENT=0;

-- 기존 테이블에 변경
mysql> ALTER TABLE employees.employees STATS_PERSISTENT=1;
```

**innodb_index_stats 테이블의 주요 칼럼**

| stat_name | 설명 |
|-----------|------|
| `n_diff_pfx%` | 인덱스가 가진 유니크한 값의 개수 |
| `n_leaf_pages` | 인덱스의 리프 노드 페이지 개수 |
| `size` | 인덱스 트리의 전체 페이지 개수 |

**innodb_table_stats 테이블의 주요 칼럼**

| 칼럼명 | 설명 |
|--------|------|
| `n_rows` | 테이블의 전체 레코드 건수 |
| `clustered_index_size` | 프라이머리 키의 크기 (InnoDB 페이지 개수) |
| `sum_of_other_index_sizes` | 프라이머리 키를 제외한 인덱스의 크기 (InnoDB 페이지 개수) |

**통계 정보 자동 갱신 이벤트** (MySQL 5.5 이하 방식)
- 테이블이 새로 오픈되는 경우
- 테이블의 레코드가 대량으로 변경되는 경우 (전체 레코드 중 1/16 정도의 UPDATE/INSERT/DELETE)
- `ANALYZE TABLE` 명령이 실행되는 경우
- `SHOW TABLE STATUS` 또는 `SHOW INDEX FROM` 명령이 실행되는 경우
- InnoDB 모니터가 활성화되는 경우
- `innodb_stats_on_metadata` 시스템 설정이 ON인 상태에서 `SHOW TABLE STATUS` 명령이 실행되는 경우

> 자주 통계 정보가 갱신되면 인덱스 레인지 스캔으로 잘 처리되던 쿼리가 갑자기 풀 테이블 스캔으로 바뀌는 상황이 발생할 수 있다.
> 영구적인 통계 정보를 사용하면 `innodb_stats_auto_recalc=OFF`로 이를 방지할 수 있다.

**STATS_AUTO_RECALC 옵션**

| 값 | 동작 |
|----|------|
| `1` | MySQL 5.5 이전 방식대로 자동 수집 |
| `0` | `ANALYZE TABLE` 명령을 실행할 때만 수집 |
| `DEFAULT` | `innodb_stats_auto_recalc` 시스템 변수값으로 결정 |

**통계 정보 샘플링 페이지 수**

| 시스템 변수 | 기본값 | 설명 |
|------------|--------|------|
| `innodb_stats_transient_sample_pages` | 8 | 자동 통계 수집 시 8개 페이지 샘플링 |
| `innodb_stats_persistent_sample_pages` | 20 | `ANALYZE TABLE` 실행 시 20개 페이지 샘플링 후 영구 저장 |

> 더 정확한 통계 정보를 수집하려면 `innodb_stats_persistent_sample_pages` 값을 높이면 되지만,
> 값이 너무 높으면 통계 정보 수집 시간이 길어지므로 주의한다.

### 10.1.2 히스토그램

MySQL 5.7까지의 통계 정보는 단순히 인덱스된 칼럼의 **유니크한 값의 개수** 정도만 가지고 있었다.
이는 옵티마이저가 최적의 실행 계획을 수립하기에 많이 부족했다.
MySQL 8.0 버전부터 칼럼의 **데이터 분포도**를 참조할 수 있는 히스토그램 정보를 활용할 수 있게 됐다.

---

#### 10.1.2.1 히스토그램 정보 수집 및 삭제

- 히스토그램 정보는 **칼럼 단위**로 관리된다.
- **자동으로 수집되지 않고** `ANALYZE TABLE ... UPDATE HISTOGRAM` 명령으로 수동으로 수집 및 관리된다.
- 수집된 히스토그램 정보는 시스템 딕셔너리에 저장되며, MySQL 서버 시작 시 `information_schema.column_statistics` 테이블로 로드된다.

```sql
-- 히스토그램 수집
mysql> ANALYZE TABLE employees.employees
       UPDATE HISTOGRAM ON gender, hire_date;

-- 히스토그램 조회
mysql> SELECT *
       FROM COLUMN_STATISTICS
       WHERE SCHEMA_NAME='employees'
         AND TABLE_NAME='employees'\G

-- 히스토그램 삭제
mysql> ANALYZE TABLE employees.employees
       DROP HISTOGRAM ON gender, hire_date;
```

**MySQL 8.0에서 지원하는 2종류의 히스토그램 타입**

| 타입 | 별칭 | 설명 | 버킷 정보 |
|------|------|------|-----------|
| **Singleton** (싱글톤 히스토그램) | Value-Based, 도수 분포라고도 불림 | 칼럼값 개별로 레코드 건수를 관리 | 칼럼값 + 발생 빈도의 비율 (2개 값) |
| **Equi-Height** (높이 균형 히스토그램) | Height-Balanced라고도 불림 | 칼럼값의 범위를 균등한 개수로 구분하여 관리 | 범위 시작값, 마지막값, 발생 빈도율, 유니크 값 개수 (4개 값) |

> - 싱글톤 히스토그램은 주로 코드 값처럼 **유니크한 값의 개수가 적은** 경우 사용된다.
> - 높이 균형 히스토그램은 각 버킷에 비슷한 수의 레코드가 포함되도록 범위를 나눈다.
>   그래프의 기울기가 일정한 것을 보면 각 범위가 비슷한 값(레코드의 건수)을 가진다는 것을 알 수 있다.

**히스토그램의 주요 메타 필드 (`HISTOGRAM` 칼럼 내)**

| 필드 | 설명 |
|------|------|
| `sampling-rate` | 히스토그램 정보를 수집하기 위해 스캔한 페이지의 비율 (0.35 = 전체 35% 스캔) |
| `histogram-type` | 히스토그램의 종류 |
| `number-of-buckets-specified` | 생성 시 설정한 버킷의 개수 (기본값 100, 최대 1024) |

> **MySQL 8.0.19 이후**: InnoDB 스토리지 엔진 자체적으로 샘플링 알고리즘을 구현하여,
> 히스토그램 수집 시 풀 스캔이 필요 없게 됐다.

**히스토그램 비활성화 방법**
```sql
-- 글로벌 비활성화
mysql> SET GLOBAL optimizer_switch='condition_fanout_filter=off';

-- 현재 커넥션에서만 비활성화
mysql> SET SESSION optimizer_switch='condition_fanout_filter=off';

-- 현재 쿼리에서만 비활성화
mysql> SELECT /*+ SET_VAR(optimizer_switch='condition_fanout_filter=off') */ *
       FROM ...
```

#### 10.1.2.2 히스토그램의 용도

기존 통계 정보는 테이블의 전체 레코드 건수와 인덱스된 칼럼이 가지는 유니크한 값의 개수만 있었다.
이는 **균등한 분포**를 가정하기 때문에 실제 데이터의 불균등한 분포를 고려하지 못했다.

**히스토그램 사용 전/후 비교 예시**

```sql
-- 히스토그램 없을 때
mysql> EXPLAIN
       SELECT * FROM employees
       WHERE first_name='Zita'
         AND birth_date BETWEEN '1950-01-01' AND '1960-01-01';

-- 결과: filtered = 11.11 (224건 중 24.8명 예측)
```

```sql
-- 히스토그램 수집 후
mysql> ANALYZE TABLE employees
       UPDATE HISTOGRAM ON first_name, birth_date;

mysql> EXPLAIN
       SELECT * FROM employees
       WHERE first_name='Zita'
         AND birth_date BETWEEN '1950-01-01' AND '1960-01-01';

-- 결과: filtered = 60.82 (224건 중 136.2명 예측)
-- 실제 데이터: 63.84% (143명)
```

> 단순 통계 정보만 이용한 경우(11.11%)와 히스토그램을 이용한 경우(60.82%)의 차이가 매우 크다.
> 히스토그램을 이용하면 실제값(63.84%)에 훨씬 근접한 예측이 가능하다.

**조인 최적화에서의 히스토그램 효과**

히스토그램 정보가 있으면 옵티마이저가 조인의 드라이빙 테이블을 더 정확히 결정할 수 있다.

```sql
-- employees를 먼저 읽는 경우: 0.13초
mysql> SELECT /*+ JOIN_ORDER(e, s) */ * FROM salaries s
       INNER JOIN employees e ON e.emp_no=s.emp_no
       AND e.birth_date BETWEEN '1950-01-01' AND '1950-02-01'
       WHERE s.salary BETWEEN 40000 AND 70000;
-- Empty set (0.13 sec)

-- salaries를 먼저 읽는 경우: 1.29초
mysql> SELECT /*+ JOIN_ORDER(s, e) */ * FROM salaries s
       INNER JOIN employees e ON e.emp_no=s.emp_no
       AND e.birth_date BETWEEN '1950-01-01' AND '1950-02-01'
       WHERE s.salary BETWEEN 40000 AND 70000;
-- Empty set (1.29 sec)
```

> 히스토그램이 없으면 옵티마이저는 테이블의 전체 레코드 건수나 크기 등의 단순한 정보만으로 드라이빙 테이블을 결정한다.
> 히스토그램이 있으면 **10배 이상의 성능 차이**를 방지할 수 있다.

#### 10.1.2.3 히스토그램과 인덱스

히스토그램과 인덱스는 완전히 다른 개체이지만, 부족한 통계 정보를 보완한다는 측면에서 공통점이 있다.

- 옵티마이저는 실행 계획 수립 시 사용 가능한 인덱스들로부터 조건에 일치하는 레코드 건수를 파악하기 위해
  실제 인덱스의 **B-Tree를 샘플링**해서 살펴본다. 이를 **"인덱스 다이브(Index Dive)"** 라고 표현한다.

- **MySQL 8.0에서 인덱스된 칼럼의 경우** → 히스토그램보다 실제 **인덱스 다이브**를 통해 직접 수집한 정보를 활용한다.
  인덱스 다이브는 실제 검색 조건의 대상 값에 대한 샘플링을 실행하므로 항상 히스토그램보다 정확한 결과를 기대할 수 있기 때문이다.

- 따라서 **히스토그램은 주로 인덱스되지 않은 칼럼**에 대한 데이터 분포도를 참조하는 용도로 사용된다.

> **주의**: 인덱스 다이브 작업은 어느 정도 비용이 필요하다.
> IN 절에 값이 많이 명시된 경우 실행 계획 수립만으로도 상당한 인덱스 다이브를 실행하고 비용도 커진다.

### 10.1.3 코스트 모델 (Cost Model)

MySQL 서버가 쿼리를 처리하려면 다양한 단위 작업들이 필요하다.

**쿼리 처리 시 필요한 주요 단위 작업**
- 디스크로부터 데이터 페이지 읽기
- 메모리(InnoDB 버퍼 풀)로부터 데이터 페이지 읽기
- 인덱스 키 비교
- 레코드 평가
- 메모리 임시 테이블 작업
- 디스크 임시 테이블 작업

이러한 단위 작업들의 비용을 **코스트 모델(Cost Model)** 이라고 한다.

**버전별 코스트 모델 발전 과정**
- **MySQL 5.7 이전**: 단위 작업의 비용을 MySQL 서버 소스 코드에 상수화해서 사용
- **MySQL 5.7**: DBMS 관리자가 조정할 수 있도록 개선 (단, 히스토그램 등 일부 정보 부족)
- **MySQL 8.0**: 칼럼의 데이터 분포(히스토그램)와 각 인덱스별 메모리에 적재된 페이지 비율이 관리되어 옵티마이저의 실행 계획 수립에 사용되기 시작

**코스트 모델 관련 테이블** (`mysql` DB에 존재)

| 테이블 | 역할 |
|--------|------|
| `server_cost` | 인덱스를 찾고 레코드를 비교하고 임시 테이블 처리에 대한 비용 관리 |
| `engine_cost` | 레코드를 가진 데이터 페이지를 가져오는 데 필요한 비용 관리 |

**두 테이블의 공통 칼럼**

| 칼럼명 | 설명 |
|--------|------|
| `cost_name` | 코스트 모델의 각 단위 작업 |
| `default_value` | 각 단위 작업의 비용 (기본값, MySQL 서버 소스 코드에 설정된 값) |
| `cost_value` | DBMS 관리자가 설정한 값 (NULL이면 default_value 사용) |
| `last_updated` | 단위 작업의 비용이 변경된 시점 |
| `comment` | 비용에 대한 추가 설명 |

`engine_cost` 테이블만의 추가 칼럼: `engine_name`(스토리지 엔진), `device_type`(디스크 타입)

**MySQL 8.0 코스트 모델의 8가지 단위 작업**

| 구분 | cost_name | default_value | 설명 |
|------|-----------|---------------|------|
| `engine_cost` | `io_block_read_cost` | 1.00 | 디스크 데이터 페이지 읽기 |
| `engine_cost` | `memory_block_read_cost` | 0.25 | 메모리 데이터 페이지 읽기 |
| `server_cost` | `disk_temptable_create_cost` | 20.00 | 디스크 임시 테이블 생성 |
| `server_cost` | `disk_temptable_row_cost` | 0.50 | 디스크 임시 테이블의 레코드 읽기 |
| `server_cost` | `key_compare_cost` | 0.05 | 인덱스 키 비교 |
| `server_cost` | `memory_temptable_create_cost` | 1.00 | 메모리 임시 테이블 생성 |
| `server_cost` | `memory_temptable_row_cost` | 0.10 | 메모리 임시 테이블의 레코드 읽기 |
| `server_cost` | `row_evaluate_cost` | 0.10 | 레코드 비교 |

**코스트 값 변경 시 예상되는 결과**

| 비용 항목 | 값을 높이면 |
|-----------|------------|
| `key_compare_cost` | 가능하면 정렬을 수행하지 않는 방향의 실행 계획 선택 가능성 증가 |
| `row_evaluate_cost` | 풀 스캔 쿼리의 비용이 높아지고, 인덱스 레인지 스캔 사용 가능성 증가 |
| `disk_temptable_create_cost`, `disk_temptable_row_cost` | 디스크 임시 테이블을 만들지 않는 방향의 실행 계획 선택 가능성 증가 |
| `memory_temptable_create_cost`, `memory_temptable_row_cost` | 메모리 임시 테이블을 만들지 않는 방향의 실행 계획 선택 가능성 증가 |
| `io_block_read_cost` | InnoDB 버퍼 풀에 데이터 페이지가 많이 적재된 인덱스를 사용하는 실행 계획 선택 가능성 증가 |
| `memory_block_read_cost` | InnoDB 버퍼 풀에 적재된 데이터 페이지가 상대적으로 적어도 그 인덱스를 사용할 가능성 증가 |

**실행 계획의 계산된 비용 확인**

```sql
-- TREE 포맷으로 비용 확인
mysql> EXPLAIN FORMAT=TREE
       SELECT * FROM employees WHERE first_name='Matt' \G

-- EXPLAIN: -> Index lookup on employees using ix_firstname (first_name='Matt')
--                 (cost=256.10 rows=233)

-- JSON 포맷으로 상세 비용 확인
mysql> EXPLAIN FORMAT=JSON
       SELECT * FROM employees WHERE first_name='Matt' \G
-- "cost_info": { "query_cost": "255.08" }
-- "read_cost": "231.78", "eval_cost": "23.30"
```

> **주의**: 코스트 모델의 비용 값을 변경할 수 있다고 해서 꼭 변경해서 사용해야 하는 것은 아니다.
> 서비스에 사용되는 MySQL 서버의 `engine_cost`와 `server_cost` 테이블의 기본값을 함부로 변경하지 않는 것이 좋다.

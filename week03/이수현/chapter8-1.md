# 8.1 디스크 읽기 방식

인덱스를 제대로 이해하려면 먼저 **디스크 I/O 방식**을 이해해야 한다.  
데이터베이스 성능은 결국 얼마나 효율적으로 디스크 I/O를 줄이느냐에 달려 있다.

CPU와 메모리의 성능은 지속적으로 발전해왔지만, 디스크 장치는 상대적으로 발전 속도가 느리다.  
특히 데이터베이스는 대용량 데이터를 저장하기 때문에 디스크 접근이 필연적으로 발생한다.

따라서 DB 성능의 핵심은 다음과 같이 정리할 수 있다.

> **DB 성능 = 디스크 I/O를 얼마나 줄이느냐**

---

## 8.1.1 하드 디스크 드라이브(HDD)와 솔리드 스테이트 드라이브(SSD)

### 1. HDD (Hard Disk Drive)

- 기계식 장치
- 플래터(원판)를 회전시켜 데이터를 읽고 씀
- 물리적인 헤드 이동 필요
- 탐색 시간(Seek Time) + 회전 지연(Rotational Latency) 발생

**특징**
- 랜덤 I/O에 매우 취약
- 순차 I/O는 상대적으로 효율적
- 기계적 구조로 인해 물리적 한계 존재

---

### 2. SSD (Solid State Drive)

- 전자식 장치 (기계적 회전 없음)
- 플래시 메모리 기반
- 헤드 이동 없음 → 접근 속도 매우 빠름

**특징**
- 랜덤 I/O 성능이 HDD보다 월등히 우수
- 지연시간(latency)이 매우 낮음
- 기계적 병목이 없음

---

### 성능 비교 (책의 벤치마크 요약)

책의 그래프에서는 CPU → RAM → SSD → HDD 순으로 처리 속도가 감소한다.

| 장치 | 초당 처리량(상대적) |
|------|--------------------|
| CPU  | 매우 빠름 |
| RAM  | 매우 빠름 |
| SSD  | 빠름 |
| HDD  | 느림 |

벤치마크 결과 예시:

- SSD는 초당 수백 건의 트랜잭션 처리 가능
- HDD는 SSD 대비 현저히 낮은 처리량

👉 그러나 중요한 점은:

> 디스크 성능의 핵심은 **랜덤 I/O 성능 차이**

일반적인 OLTP 환경에서는 작은 데이터를 자주 읽는 작업이 많기 때문에  
SSD의 랜덤 I/O 성능이 DB 성능에 큰 영향을 미친다.

---

### ⚠️ 주의할 점

SSD가 항상 순차 I/O보다 랜덤 I/O가 빠른 것은 아니다.  
SSD에서도 순차 I/O가 여전히 더 높은 스루풋(Throughput)을 보인다.

또한 RAID 구성, 컨트롤러, 캐시 메모리 등에 따라 성능 차이가 발생할 수 있다.

---

# 8.1.2 랜덤 I/O와 순차 I/O

디스크 I/O는 크게 두 가지로 나뉜다.

---

## 1. 순차 I/O (Sequential I/O)

- 연속된 블록을 한 번에 읽는 방식
- 디스크 헤드 이동 최소화
- 대량 데이터 처리에 유리

예:
- 로그 파일 읽기
- 전체 테이블 스캔

---

## 2. 랜덤 I/O (Random I/O)

- 흩어져 있는 데이터를 각각 개별적으로 읽는 방식
- 디스크 헤드 이동이 자주 발생
- 성능 저하의 주요 원인

예:
- 인덱스를 통한 레코드 접근
- 특정 PK 기반 조회

---

### 예시 비교 (책 설명 요약)

#### 순차 I/O
- 3개의 페이지(각 10KB)를 읽을 때
- 한 번의 시스템 콜로 처리 가능

#### 랜덤 I/O
- 동일한 3페이지를 읽더라도
- 각각 별도의 위치 → 3번의 시스템 콜 필요
- 헤드 이동 3번 발생

결과적으로:

> 랜덤 I/O는 순차 I/O보다 훨씬 느리다.

---

## 데이터베이스에서 중요한 점

DBMS는 대부분 **랜덤 I/O 작업이 많다.**

왜냐하면:

- 인덱스는 정렬 구조
- 원하는 레코드를 찾기 위해 트리 탐색 수행
- 그 결과 데이터 페이지를 개별적으로 접근

따라서 DB 성능 튜닝의 본질은 다음과 같다.

> ❗ 랜덤 I/O 횟수를 줄이는 것

---

## 인덱스의 존재 이유

인덱스는 랜덤 I/O를 완전히 제거하지는 못하지만  
**랜덤 I/O의 횟수를 줄이기 위해 존재한다.**

즉,

- 인덱스를 사용하지 않으면 → 전체 테이블 스캔 (많은 순차 I/O)
- 인덱스를 사용하면 → 적은 랜덤 I/O로 필요한 데이터만 접근

결국:

> DB 튜닝 = 쿼리가 얼마나 적은 페이지를 읽도록 만드는가

---

## 정리

| 구분 | 순차 I/O | 랜덤 I/O |
|------|----------|----------|
| 접근 방식 | 연속 블록 | 흩어진 블록 |
| 헤드 이동 | 거의 없음 | 빈번 |
| 성능 | 빠름 | 느림 |
| DB에서 영향 | Full Scan | 인덱스 접근 |

---

# 핵심 결론

1. 데이터베이스 성능은 디스크 I/O에 의해 결정된다.
2. 특히 랜덤 I/O가 성능 병목의 핵심이다.
3. SSD는 HDD보다 랜덤 I/O에 강하다.
4. 인덱스는 랜덤 I/O를 줄이기 위한 구조이다.

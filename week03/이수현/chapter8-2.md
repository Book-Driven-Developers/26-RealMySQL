# 8.2 인덱스란?

---

## 인덱스의 개념

많은 사람이 인덱스를 설명할 때 **책의 찾아보기(색인)** 에 비유한다.

- 책의 맨 뒤에 있는 찾아보기는
    - 단어 → 페이지 번호
    - 정렬된 구조
    - 원하는 내용을 빠르게 찾기 위함

DBMS의 인덱스도 동일한 개념이다.

> 인덱스는 **데이터를 빠르게 찾기 위한 자료구조**

테이블의 모든 데이터를 처음부터 끝까지 검색하면 시간이 오래 걸린다.  
따라서 특정 컬럼을 기준으로 **정렬된 구조를 별도로 만들어**  
데이터 검색 속도를 향상시킨다.

---

## 자료구조와 인덱스의 관계

책에서는 인덱스를 설명하면서 자료구조와 비교한다.

### 1. SortedList (정렬 리스트)

- 데이터를 항상 정렬 상태로 유지
- 검색은 빠름 (이진 탐색 가능)
- 하지만 INSERT / UPDATE / DELETE 비용이 큼

### 2. ArrayList (비정렬 리스트)

- 저장은 빠름
- 검색은 느림 (선형 탐색 필요)

DBMS 인덱스는 **정렬된 자료구조**의 특성을 가진다.  
따라서 다음과 같은 특징이 있다.

| 구분 | 특징 |
|------|------|
| SELECT | 매우 빠름 |
| INSERT | 상대적으로 느림 |
| UPDATE | 느림 |
| DELETE | 느림 |

---

## 인덱스의 본질

인덱스는 검색 속도를 향상시키는 대신  
데이터 변경 작업의 비용을 증가시킨다.

즉,

> 인덱스는 읽기(SELECT) 성능을 위해 쓰기 성능을 희생하는 구조

따라서 모든 컬럼에 인덱스를 생성하면 오히려 성능이 저하될 수 있다.

---

# 인덱스의 분류

인덱스는 여러 기준에 따라 나눌 수 있다.

---

## 1. 역할 기준 분류

### Primary Key (기본 키 인덱스)

- 테이블의 대표 키
- 중복 불가
- NULL 불가
- 레코드 식별 목적

InnoDB에서는 Primary Key가 **클러스터링 인덱스**가 된다.

---

### Secondary Index (보조 인덱스)

- 기본 키 외의 인덱스
- 중복 허용 가능
- 검색 성능 향상을 위한 용도

---

## 2. 데이터 저장 방식(알고리즘) 기준 분류

대표적으로 다음 두 가지가 있다.

### B-Tree 인덱스

- 가장 일반적인 인덱스
- 대부분의 RDBMS에서 기본 인덱스 구조
- 범위 검색에 강함

### Hash 인덱스

- 해시 함수 기반
- 정확한 일치 검색에 매우 빠름
- 범위 검색 불가능

※ MySQL InnoDB는 기본적으로 B-Tree를 사용한다.

---

## 3. 중복 허용 여부 기준

### Unique Index

- 중복 허용하지 않음
- WHERE col = value 조건에서 매우 효율적

### Non-Unique Index

- 중복 허용
- 일반적인 검색 인덱스

---

## 인덱스의 장단점

### ✅ 장점

- 검색 속도 향상
- WHERE 조건 검색 최적화
- 정렬(ORDER BY) 최적화 가능

### ❌ 단점

- INSERT / UPDATE / DELETE 성능 저하
- 저장 공간 추가 사용
- 인덱스 관리 비용 발생

---

# 핵심 정리

1. 인덱스는 데이터를 빠르게 찾기 위한 정렬된 자료구조이다.
2. SELECT는 빠르게, INSERT/UPDATE/DELETE는 느리게 만든다.
3. Primary Key와 Secondary Index로 나뉜다.
4. 대부분의 RDBMS는 B-Tree 기반 인덱스를 사용한다.
5. 모든 컬럼에 인덱스를 생성하는 것은 위험하다.

---

# 한 줄 요약

> 인덱스는 "검색 속도를 얻고 쓰기 성능을 지불하는" 구조이다.

# 8.3.4 B-Tree 인덱스를 통한 데이터 읽기

MySQL이 B-Tree 인덱스를 통해 데이터를 어떻게 읽는지 이해하는 것은
실행 계획(EXPLAIN)을 해석하는 데 매우 중요하다.

B-Tree 인덱스를 사용하는 방식은 크게 다음과 같다.

1. 인덱스 레인지 스캔 (Index Range Scan)
2. 인덱스 풀 스캔 (Index Full Scan)
3. 루스 인덱스 스캔 (Loose Index Scan)
4. 인덱스 스킵 스캔 (Index Skip Scan)

--------------------------------------------------

## 8.3.4.1 인덱스 레인지 스캔 (Index Range Scan)

가장 일반적이고 효율적인 인덱스 사용 방식이다.

### 동작 과정

1) 인덱스 트리에서 조건을 만족하는 시작 위치 탐색
2) 해당 위치부터 필요한 범위까지 리프 노드를 순차적으로 읽음
3) (InnoDB의 경우) PK를 통해 실제 데이터 페이지 접근

예시:

```sql
SELECT *
FROM employees
WHERE first_name BETWEEN 'Ebbe' AND 'Gad';
```

위 쿼리는 다음과 같이 동작한다.

- 루트 → 브랜치 → 리프 노드에서 시작점 탐색
- 조건 범위에 해당하는 리프 구간을 순차적으로 읽음
- 필요 시 실제 데이터 페이지 접근

### 특징

- 가장 효율적인 인덱스 접근 방식
- 필요한 범위만 읽는다
- WHERE 조건이 인덱스 컬럼과 일치해야 함
- ORDER BY 최적화 가능

--------------------------------------------------

### 커버링 인덱스 (Covering Index)

쿼리에 필요한 컬럼이 모두 인덱스에 포함되어 있으면
데이터 파일을 읽지 않고 인덱스만으로 처리 가능하다.

예:

```sql
EXPLAIN SELECT gender, birth_date
FROM employees
WHERE gender = 'M';
```

Extra:
Using index

→ 데이터 페이지 접근 없음
→ 디스크 I/O 감소
→ 매우 빠른 처리

--------------------------------------------------

## 8.3.4.2 인덱스 풀 스캔 (Index Full Scan)

인덱스 전체를 처음부터 끝까지 읽는 방식이다.

### 사용되는 경우

- WHERE 조건이 인덱스 첫 컬럼과 일치하지 않을 때
- 인덱스가 테이블보다 작은 경우

예:

```sql
ALTER TABLE employees
ADD INDEX ix_gender_birthdate (gender, birth_date);

SELECT *
FROM employees
WHERE birth_date >= '1965-02-01';
```

- 선행 컬럼(gender)에 조건이 없음
- 인덱스를 앞에서부터 전부 읽음

### 특징

- 테이블 풀 스캔보다는 나을 수 있음
- 하지만 효율적인 방식은 아님
- 복합 인덱스의 선행 컬럼이 매우 중요함

--------------------------------------------------

## 8.3.4.3 루스 인덱스 스캔 (Loose Index Scan)

일반 인덱스 레인지 스캔보다 덜 촘촘하게 읽는 방식이다.
주로 GROUP BY, DISTINCT, MIN/MAX 최적화에 사용된다.

예:

```sql
SELECT dept_no, MIN(emp_no)
FROM dept_emp
WHERE dept_no BETWEEN 'd002' AND 'd004'
GROUP BY dept_no;
```

동작 방식:

- 각 그룹의 첫 번째 값만 읽는다
- 모든 레코드를 읽지 않는다
- 필요한 키만 건너뛰며 읽는다

### 특징

- GROUP BY 최적화에 매우 유리
- 디스크 I/O 감소
- 인덱스를 부분적으로 활용

--------------------------------------------------

## 8.3.4.4 인덱스 스킵 스캔 (Index Skip Scan)

MySQL 8.0부터 도입된 기능이다.

복합 인덱스에서 선행 컬럼 조건 없이
후행 컬럼 조건만으로 인덱스를 활용하는 방식이다.

예:

복합 인덱스:
(gender, birth_date)

기존에는 다음 쿼리에서 인덱스를 사용할 수 없었다.

```sql
SELECT *
FROM employees
WHERE birth_date >= '1965-02-01';
```

하지만 스킵 스캔은 내부적으로 다음과 같이 처리한다.

1) gender = 'M' 조건으로 범위 검색
2) gender = 'F' 조건으로 범위 검색
3) 결과를 합침

즉, 내부적으로 여러 번 인덱스 레인지 스캔을 수행한다.

실행 계획 예:

```sql
SET optimizer_switch='skip_scan=on';

EXPLAIN
SELECT gender, birth_date
FROM employees
WHERE birth_date >= '1965-02-01';
```

Extra:
Using index for skip scan

### 제약 조건

- 선행 컬럼의 유니크 값 개수가 적어야 함
- 후행 컬럼에 조건이 존재해야 함
- 옵티마이저가 비용 비교 후 선택

--------------------------------------------------

## 인덱스 사용 여부 판단 기준

옵티마이저는 다음을 비교한다.

- 인덱스를 통한 접근 비용
- 테이블 풀 스캔 비용

일반적으로

전체 레코드의 20~25% 이상 읽어야 한다면
테이블 풀 스캔이 더 효율적일 수 있다.

--------------------------------------------------

## 핵심 정리

1. 인덱스 레인지 스캔이 가장 이상적인 방식이다.
2. 복합 인덱스는 선행 컬럼 조건이 매우 중요하다.
3. 커버링 인덱스는 디스크 접근을 제거한다.
4. 루스 인덱스 스캔은 그룹 단위 처리에 유리하다.
5. 인덱스 스킵 스캔은 MySQL 8.0의 최적화 기능이다.
6. 인덱스를 사용한다고 항상 빠른 것은 아니다.

# 8.3 B-Tree 인덱스

---

# 8.3.1 구조 및 특성

B-Tree는 DBMS에서 가장 일반적으로 사용되는 인덱스 구조이다.  
Balanced Tree의 약자로, 항상 균형을 유지하는 트리 구조이다.

## 기본 구조

B-Tree는 다음 3가지 노드로 구성된다.

- **Root Node (루트 노드)**
- **Branch Node (브랜치 노드)**
- **Leaf Node (리프 노드)**

### 탐색 과정

1. 루트 노드에서 시작
2. 브랜치 노드를 거쳐
3. 최종적으로 리프 노드에 도달

리프 노드에는 실제 **데이터 레코드의 위치 정보**가 저장된다.

---

## 인덱스와 데이터 파일의 관계

### MyISAM

- 인덱스와 데이터 파일이 분리되어 있음
- 리프 노드에는 **데이터 파일의 물리적 주소** 저장

### InnoDB

- Primary Key 기준으로 데이터 자체가 정렬 저장
- Secondary Index의 리프 노드에는
    - 물리 주소 대신 **Primary Key 값** 저장
- Secondary Index → PK 인덱스를 한 번 더 탐색해야 실제 레코드 접근 가능

👉 InnoDB는 PK를 중심으로 클러스터링 구조를 가진다.

---

## 데이터 파일 정렬 여부

- 인덱스는 항상 정렬되어 있음
- 하지만 데이터 파일은 항상 정렬된 상태가 아님
- INSERT 순서대로 저장되지 않음

---

# 8.3.2 B-Tree 인덱스 추가 및 삭제

---

## 8.3.2.1 인덱스 추가

새로운 값이 추가되면:

1. B-Tree에서 적절한 위치 탐색
2. 리프 노드에 삽입
3. 노드가 가득 찬 경우 → 노드 분리(Split) 발생

### 비용

- 일반적으로 테이블에 인덱스가 3개 있다면
    - INSERT 시 인덱스 작업은 3번 수행
- 대략 테이블 작업 비용의 1~1.5배 수준

### MyISAM vs InnoDB

- MyISAM: 즉시 인덱스 반영
- InnoDB: 체인지 버퍼(Change Buffer)를 통해 지연 반영 가능

---

## 8.3.2.2 인덱스 삭제

- 해당 키가 저장된 리프 노드를 찾아 삭제 표시
- 실제 공간 정리는 즉시 일어나지 않을 수도 있음
- 디스크 I/O 발생 가능

MyISAM/MEMORY는 즉시 처리  
InnoDB는 내부적으로 처리

---

## 8.3.2.3 인덱스 변경 (UPDATE)

인덱스 키가 변경되면:

1. 기존 키 삭제
2. 새로운 키 삽입

즉, UPDATE는 내부적으로
> DELETE + INSERT

작업으로 처리된다.

---

## 8.3.2.4 인덱스 검색

B-Tree 검색은 항상 루트 노드에서 시작한다.

### 사용되는 경우

- 100% 일치 검색
- 부분 일치 검색
- 범위 검색

InnoDB에서는 인덱스를 통해 레코드를 잠그는 기능도 제공한다.

---

# 8.3.3 B-Tree 인덱스 사용에 영향을 미치는 요소

B-Tree의 성능은 다음 요소에 의해 결정된다.

---

## 8.3.3.1 인덱스 키의 크기

InnoDB는 데이터를 **페이지(Page)** 단위로 관리한다.  
기본 페이지 크기: 16KB

### 한 페이지에 저장 가능한 인덱스 개수

- 키 크기가 작을수록
    - 한 페이지에 더 많은 키 저장 가능
    - 트리 깊이 감소
    - 디스크 I/O 감소

예:
- 키 크기 16바이트
- 키 크기 32바이트

→ 키가 커질수록 저장 가능한 개수 감소

### 결론

> 인덱스 키는 가능한 작게 유지하는 것이 좋다.

---

## 8.3.3.2 B-Tree 깊이 (Depth)

트리 깊이는 성능에 직접적인 영향을 준다.

- 깊이가 1 증가 → 디스크 I/O 1회 증가
- 일반적으로 3~4 수준 유지

키 크기가 증가하면:

- 한 페이지에 저장 가능한 키 수 감소
- 트리 깊이 증가
- 검색 비용 증가

---

## 8.3.3.3 선택도 (Selectivity)

선택도 = 유니크 값의 비율

### 계산 방식
선택도 = 유니크 값 수 / 전체 레코드 수


예:
- 전체 10,000건
- 유니크 값 100개
  → 선택도 낮음

- 전체 10,000건
- 유니크 값 9,000개
  → 선택도 높음

### 선택도가 중요한 이유

- 선택도가 높을수록
  - 조건에 해당하는 레코드 수 적음
  - 인덱스 효율 높음

- 선택도가 낮으면
  - 많은 레코드 읽어야 함
  - 옵티마이저가 인덱스 사용하지 않을 수도 있음

---

## 8.3.3.4 읽어야 하는 레코드 수

옵티마이저는 다음을 비교한다.

- 인덱스를 사용해서 읽는 비용
- 전체 테이블 스캔 비용

일반적으로:

> 전체 레코드의 20~25% 이상 읽어야 한다면  
> 테이블 풀 스캔이 더 효율적일 수 있다.

---

# 핵심 정리

1. B-Tree는 균형 트리 구조이다.
2. InnoDB는 Secondary Index에서 PK를 한 번 더 조회한다.
3. INSERT/DELETE/UPDATE는 인덱스 유지 비용이 발생한다.
4. 인덱스 키는 작을수록 좋다.
5. 트리 깊이는 낮을수록 좋다.
6. 선택도가 높을수록 인덱스 효율이 좋다.
7. 전체의 20~25% 이상 읽으면 풀스캔이 더 나을 수 있다.

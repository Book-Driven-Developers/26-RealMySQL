# 8-3 딥다이브 - 커버링 인덱스와 PK 재탐색

> InnoDB에서 세컨더리 인덱스를 통해 레코드를 읽을 때 "반드시 프라이머리 키 B-Tree를 다시 한번 검색해야 한다"는 설명이 커버링 인덱스에도 해당되는가?

---

## 배경: InnoDB 세컨더리 인덱스의 일반적인 동작

InnoDB의 세컨더리 인덱스 리프 노드는 레코드의 **물리적 주소 대신 프라이머리 키 값**을 저장한다.

```
세컨더리 인덱스 리프 노드
┌─────────────────────────────┐
│  인덱스 키 값 | 프라이머리 키 값  │
└─────────────────────────────┘
```

그래서 쿼리가 인덱스에 없는 칼럼을 요구할 경우, 두 번의 B-Tree 탐색이 발생한다.

```
세컨더리 인덱스 탐색 → PK 값 획득 → 클러스터 인덱스(PK B-Tree) 재탐색 → 레코드 반환
```

---

## 커버링 인덱스: PK 재탐색의 예외

쿼리가 필요로 하는 **모든 칼럼이 이미 인덱스 안에 존재**한다면, 클러스터 인덱스를 재탐색할 필요가 없다.

```sql
-- 인덱스: INDEX ix_country (country)
SELECT country FROM tb_city WHERE country = 'KOREA';
```

`country` 값이 이미 세컨더리 인덱스에 있으므로:

```
세컨더리 인덱스 탐색 → 리프 노드에서 바로 반환 ✅
```

EXPLAIN 실행 시 `Extra` 컬럼에 `Using index` 가 표시되면 커버링 인덱스가 작동하고 있다는 신호다.

| 상황 | PK B-Tree 재탐색 |
|---|---|
| 일반 세컨더리 인덱스 | O (필요) |
| **커버링 인덱스** | **X (불필요)** |

---

## PK 칼럼은 명시하지 않아도 커버링 인덱스 대상

InnoDB는 세컨더리 인덱스 리프 노드에 **PK 값을 항상 함께 저장**한다. 따라서 인덱스 정의에 PK 칼럼을 명시하지 않아도, PK를 포함하는 쿼리는 커버링 인덱스로 처리될 수 있다.

```sql
-- 인덱스: INDEX ix_country (country)
-- PK: id

SELECT id, country FROM tb_city WHERE country = 'KOREA';
```

`id`는 인덱스 정의에 없지만, 리프 노드에 이미 존재하기 때문에 클러스터 인덱스 재탐색 없이 처리된다.

> **설계 관점**: PK를 함께 조회하는 쿼리라면, 복합 인덱스에 PK 칼럼을 명시하지 않아도 커버링 인덱스 효과를 얻을 수 있다.

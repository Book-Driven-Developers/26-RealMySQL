# 8장 인덱스 - 디스크 읽기 방식 & 인덱스란?

---

## 8.1 디스크 읽기 방식

인덱스에만 의존적인 용어는 아니지만, 이 장에서 자주 언급되는 **랜덤(Random) I/O**, **순차(Sequential) I/O** 와 같은 디스크 읽기 방식을 먼저 이해하고 인덱스를 살펴본다.

### 8.1.1 하드 디스크 드라이브(HDD)와 솔리드 스테이트 드라이브(SSD)

- 컴퓨터에서 CPU나 메모리 같은 주요 장치는 대부분 **전자식 장치**지만 HDD는 **기계식 장치**다.
- 데이터베이스 서버에서는 항상 디스크 장치가 병목이 되며, HDD를 대체하기 위해 **SSD(Solid State Drive)** 가 많이 출시되고 있다.
- SSD는 기존 HDD와 같은 인터페이스(SATA나 SAS)를 지원하므로, 내장 디스크나 DAS/SAN에 그대로 사용할 수 있다.

SSD는 플래시 메모리를 장착해 **원판을 기계적으로 회전시킬 필요가 없으므로** 데이터를 훨씬 빠르게 읽고 쓸 수 있다. 플래시 메모리는 전원이 공급되지 않아도 데이터가 삭제되지 않는다.

> **SSD vs HDD 성능 벤치마크**
> - SSD: 초당 436개의 트랜잭션 처리
> - HDD: 초당 60개의 트랜잭션 처리

일반적인 웹 서비스(OLTP) 환경의 데이터베이스에서는 SSD가 HDD보다 훨씬 빠르다.

---

### 8.1.2 랜덤 I/O와 순차 I/O

**랜덤 I/O** 는 HDD에서 플래터(원판)를 돌려서 읽어야 할 데이터가 저장된 위치로 디스크 헤더를 이동시킨 다음 데이터를 읽는 것을 의미하며, 사실 **순차 I/O도 이 작업 과정은 같다**.

| 구분 | 디스크 헤더 이동 횟수 |
|---|---|
| 순차 I/O (3개 페이지 기록) | 1번 |
| 랜덤 I/O (3개 페이지 기록) | 3번 |

→ 디스크의 성능은 **디스크 헤더의 위치 이동 없이 얼마나 많은 데이터를 한 번에 기록하느냐**에 의해 결정된다. 그래서 여러 번 쓰기를 요청하는 **랜덤 I/O 작업이 부하가 훨씬 크다**.

SSD 드라이브에서도 랜덤 I/O는 여전히 순차 I/O보다 전체 스루풋(Throughput)이 떨어진다.

> **참고 - 인덱스 레인지 스캔 vs 풀 테이블 스캔**
> - 인덱스 레인지 스캔: **랜덤 I/O** 사용
> - 풀 테이블 스캔: **순차 I/O** 사용
>
> 큰 테이블의 레코드 대부분을 읽는 작업에서는 인덱스를 사용하지 않고 풀 테이블 스캔을 사용하도록 유도하는 것이 더 효율적인 이유가 바로 이것이다.

**쿼리를 튜닝해서 랜덤 I/O를 순차 I/O로 바꿔서 실행하는 방법은 그다지 많지 않다.** 일반적으로 쿼리를 튜닝하는 것은 **랜덤 I/O 자체를 줄이는 것**이 목적이다. 즉, 쿼리를 처리하는 데 꼭 필요한 데이터만 읽도록 쿼리를 개선하는 것을 의미한다.

> **참고 - RAID 컨트롤러 캐시와 캐시 메모리**
> 일반적으로 DBMS 서버는 RAID 컨트롤러를 장착하는데, RAID 컨트롤러의 캐시 메모리는 순차 I/O를 효율적으로 처리할 수 있게 변환하는 역할을 한다. 하드 디스크뿐만 아니라 SSD를 사용하는 경우에도 RAID 컨트롤러는 중요한 역할을 하기 때문에 RAID 컨트롤러와 RAID 컨트롤러에 장착된 캐시의 성능을 무시하지 말자.

## 8.2 인덱스란?

많은 사람이 인덱스를 언급할 때는 항상 책의 맨 끝에 있는 **찾아보기(색인)** 로 설명한다. 책의 마지막에 있는 찾아보기가 인덱스에 비유된다면, 책의 내용은 **데이터 파일**에 해당한다고 볼 수 있다.

DBMS도 데이터 테이블의 모든 데이터를 검색해서 결과를 가져오려면 시간이 오래 걸린다. 그래서 칼럼(또는 칼럼들)의 값과 해당 레코드가 저장된 주소를 **키와 값의 쌍(Key-Value pair)** 으로 삼아 인덱스를 만들어 두는 것이다.

- 책의 찾아보기: 가나다 순서로 정렬 → DBMS 인덱스도 마찬가지로 **칼럼의 값을 주어진 순서로 미리 정렬해서 보관**한다.
- **SortedList**: 저장되는 값을 항상 정렬된 상태로 유지하는 자료 구조 → 인덱스와 비슷
- **ArrayList**: 값이 저장된 순서 그대로 유지하는 자료 구조 → 데이터 파일과 비슷

결론적으로 DBMS에서 인덱스는 **데이터의 저장(INSERT, UPDATE, DELETE) 성능을 희생하고** 그 대신 **데이터의 읽기 속도를 높이는 기능**이다.

> 테이블의 인덱스를 하나 더 추가할지 말지는, **읽기 속도를 얼마나 빠르게 만들어야 하느냐**와 **데이터 변경 작업이 얼마나 자주 일어나느냐**에 따라 결정해야 한다.
> SELECT 쿼리 문장의 WHERE 조건절에 사용되는 칼럼이라고 해서 전부 인덱스로 생성하면 **데이터 저장 성능이 떨어지고 인덱스의 크기가 비대해져 오히려 역효과만 불러올 수 있다**.

### 인덱스 분류

#### 1. 역할별 분류

| 구분 | 설명 |
|---|---|
| **프라이머리 키(Primary key)** | 레코드를 대표하는 칼럼의 값으로 만들어진 인덱스. NULL 불허, 중복 불허. 테이블에서 해당 레코드를 식별할 수 있는 기준값 (식별자라고도 함). |
| **세컨더리 인덱스(Secondary index)** | 프라이머리 키를 제외한 나머지 모든 인덱스. 유니크 인덱스는 별도로 분류하기도 함. |

#### 2. 알고리즘별 분류

| 알고리즘 | 설명 |
|---|---|
| **B-Tree 인덱스** | 가장 일반적으로 사용. 칼럼의 값을 변형하지 않고 원래 값을 이용해 인덱싱. MySQL의 R-Tree 인덱스도 B-Tree의 응용 알고리즘. |
| **Hash 인덱스** | 칼럼의 값으로 해시값을 계산해서 인덱싱. 매우 빠른 검색 지원. 하지만 값을 변형해서 인덱싱하므로 **전방(Prefix) 일치** 나 **범위 검색** 시 사용 불가. 주로 메모리 기반 데이터베이스에서 사용. |

> MySQL 8.0에서는 **Fractal-Tree 인덱스**와 **Merge-Tree 인덱스**도 개발됐지만, 이 책의 범위를 넘으므로 생략.

#### 3. 중복 허용 여부별 분류

| 구분 | 설명 |
|---|---|
| **유니크 인덱스(Unique)** | 인덱스가 유니크한, 즉 같은 값이 1개만 존재. 옵티마이저에게 동등 조건으로 검색하면 항상 1건의 레코드만 찾으면 된다는 것을 알려주는 **중요한 힌트** 역할. MySQL의 처리 방식의 변화나 차이가 상당히 달라짐. |
| **논유니크 인덱스(Non-Unique)** | 같은 값이 1개 이상 존재할 수 있는 인덱스. |

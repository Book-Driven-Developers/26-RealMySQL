# 딥다이브: 인덱스와 I/O 전략

> 클러스터형 인덱스와 세컨더리 인덱스의 구조적 차이가
> I/O 비용에 어떤 영향을 주는지, 커버링 인덱스와 선택도(Selectivity)까지 분석한다.

---

## 1. 클러스터형 인덱스 = 실제 데이터 그 자체

InnoDB에서 가장 중요한 구조적 특성이다.

```
❌ 오개념:
   [실제 데이터 저장소] ← 별도 존재
   [PK 인덱스]         ← 별도 존재, 포인터로 위를 가리킴

✅ 실제 InnoDB 구조:
   [PK 클러스터형 인덱스 = 실제 행 데이터]  ← 하나로 합쳐져 있음
```

```
클러스터형 인덱스 리프 페이지 내부:
┌─────────────────────────────────────────┐
│ PK=1 | name="Alice" | age=25 | ...전체행│  ← 실제 데이터가 여기 있음
│ PK=2 | name="Bob"   | age=30 | ...전체행│
│ PK=3 | name="Carol" | age=22 | ...전체행│
└─────────────────────────────────────────┘
```

B-Tree 탐색 시 루트 → 브랜치 → 리프로 여러 페이지를 거치지만,
루트/브랜치 페이지는 **Buffer Pool** 에 캐싱되어 있어 실질적으로 리프 페이지 접근 1번만 디스크 I/O가 발생한다.

## 2. 클러스터형 인덱스의 I/O 패턴 - 단건 vs 범위

```
❌ 오개념: PK 조회 → 순차 I/O → 디스크 헤드 이동 1회

✅ 실제: 조회 종류에 따라 다르다
```

### 단건 PK 조회

```sql
SELECT * FROM users WHERE id = 50000;
```

```
B-Tree 탐색:
루트 페이지 → 브랜치 페이지 → 리프 페이지(= 실제 데이터)

= 포인트 조회 (Point Lookup)
= 순차 I/O가 아님
```

"1회에 가깝다"는 사실이지만, 순차 I/O여서가 아니다.
루트/브랜치 페이지가 Buffer Pool에 캐싱되어 있어 리프 페이지 1번만 디스크 I/O가 발생하기 때문이다.

### PK 범위 조회

```sql
SELECT * FROM users WHERE id BETWEEN 1 AND 10000;
```

```
리프 페이지들을 포인터로 따라가며 순서대로 읽음
Page3 → Page7 → Page15 → ...

= 준순차 I/O
= 세컨더리 인덱스보다 훨씬 효율적
```

이것이 클러스터형 인덱스가 "순차에 가깝다"고 하는 진짜 이유다.

### 조회 종류별 정리

| 조회 종류 | I/O 패턴 | 이유 |
|-----------|----------|------|
| PK 단건 조회 | 포인트 조회 | B-Tree 탐색, Buffer Pool이 보완 |
| PK 범위 조회 | 준순차 I/O | 리프 페이지가 PK 순서로 논리 연결 |
| 세컨더리 범위 조회 | 랜덤 I/O | 이중 조회 + 흩어진 위치 |

> 클러스터형 인덱스가 유리한 이유는 "헤드 이동 1회"여서가 아니라,
> **이중 조회가 없고, 데이터가 PK 순서로 논리 정렬되어 있어** 범위 조회 시 이동 경로가 예측 가능하기 때문이다.

## 3. 세컨더리 인덱스와 이중 조회(Double Lookup)

세컨더리 인덱스 리프 노드에는 **인덱스 키 + PK 값**만 저장된다. 실제 행 데이터가 아니다.

```
❌ 오개념:
   세컨더리 인덱스 리프: name="Alice" | → 실제 행 직접 포인터

✅ 실제 세컨더리 인덱스 리프:
   name="Alice" | PK=1   ← PK 값만 저장
   name="Bob"   | PK=2
```

따라서 세컨더리 인덱스 조회는 반드시 두 번의 B-Tree 탐색을 거친다.

```
SELECT * FROM users WHERE name = 'Alice';

1단계: 세컨더리 인덱스 B-Tree 탐색 → PK=1 획득
2단계: 클러스터형 인덱스 B-Tree 다시 탐색 → PK=1로 실제 행 조회

= 이중 조회 (Double Lookup)
= 랜덤 I/O 2회 발생
```

전체 구조를 그림으로 정리하면:

```
[세컨더리 인덱스: name B-Tree]     [클러스터형 인덱스 = 실제 데이터]
┌──────────────────┐               ┌──────────────────────────────┐
│ "Alice" | PK=1   │ → PK=1 조회 → │ PK=1 | Alice | 25 | ...전체행│
│ "Bob"   | PK=2   │ → PK=2 조회 → │ PK=2 | Bob   | 30 | ...전체행│
└──────────────────┘               └──────────────────────────────┘
  별도 B-Tree 구조                    이것 자체가 원본 데이터
```

## 4. Full Table Scan vs Index Scan 선택 기준

```sql
SELECT * FROM orders WHERE customer_id = 5;
-- customer_id에 세컨더리 인덱스 존재
-- customer_id = 5인 행이 10,000개 (전체의 30%)
```

```
인덱스 스캔:  랜덤 I/O 10,000번  = 70,000ms = 70초
Full Scan:   순차 I/O 1번        = 수백 ms
```

MySQL 옵티마이저는 전체 데이터 중 **일정 비율 이상**이 조회 대상이면 인덱스를 버리고 Full Table Scan을 선택한다.

```sql
EXPLAIN SELECT * FROM orders WHERE customer_id = 5;
-- 결과가 전체의 20~30% 이상 → type: ALL (Full Scan 선택)
-- 결과가 소수            → type: ref (Index 사용)
```

## 5. 커버링 인덱스(Covering Index)

세컨더리 인덱스의 리프 노드에는 **인덱스 키 + PK 값**이 저장된다.
조회하려는 컬럼이 이 안에 모두 존재하면, 클러스터형 인덱스를 추가 조회하지 않아도 된다.
이것을 **커버링 인덱스(Covering Index)** 라고 한다.

```
name 세컨더리 인덱스 리프 노드:
┌──────────────────────┐
│ name="Alice" | PK=1  │  ← name + PK 값이 함께 저장
│ name="Bob"   | PK=2  │
└──────────────────────┘
```

### 커버링 인덱스 적용 여부

```sql
SELECT name, id FROM users WHERE name = 'Alice';
-- name: 인덱스 키에 있음, id(PK): 리프에 있음
-- → 커버링 인덱스 적용, 클러스터형 인덱스 조회 없음 (랜덤 I/O 1회)

SELECT name, age FROM users WHERE name = 'Alice';
-- age: 인덱스에 없음
-- → 이중 조회 발생 (랜덤 I/O 2회)

SELECT * FROM users WHERE name = 'Alice';
-- 모든 컬럼 필요 → 이중 조회 발생
```

### EXPLAIN으로 확인

```sql
EXPLAIN SELECT name, id FROM users WHERE name = 'Alice';
-- Extra: Using index  ← 커버링 인덱스 적용

EXPLAIN SELECT name, age FROM users WHERE name = 'Alice';
-- Extra: (없음)       ← 이중 조회 발생
```

> 커버링 인덱스는 의도적으로 설계할 수 있다.
> 자주 조회하는 컬럼 조합을 인덱스에 포함시켜 이중 조회를 원천 차단하는 전략이다.

## 6. 커버링 인덱스 불가 시에도 인덱스가 유리한 이유

```sql
-- name 세컨더리 인덱스 존재, age는 인덱스에 없음
SELECT name, age FROM users WHERE name = 'Alice';
```

### 매칭 건수가 적을 때 (예: 전체 10만 건 중 5건)

```
인덱스 사용:
  세컨더리 인덱스 탐색 → PK 5개 획득
  클러스터형 인덱스 5번 랜덤 조회
  = 총 I/O: 소량

Full Table Scan:
  10만 건의 모든 페이지를 순차로 읽음
  = 총 I/O: 전체 데이터
```

→ 이중 조회 5번이 전체 스캔보다 훨씬 싸므로 **인덱스가 유리**하다.

### 매칭 건수가 많을 때 (예: 전체 10만 건 중 3만 건)

```
인덱스 사용:
  클러스터형 인덱스 3만 번 랜덤 조회
  = 랜덤 I/O 3만 번

Full Table Scan:
  전체 페이지 순차 읽기 1번
  = 순차 I/O
```

→ **Full Table Scan이 유리**하다. 옵티마이저가 인덱스를 무시하고 Full Scan을 선택한다.

### 선택도(Selectivity)

이 판단 기준이 되는 개념이 **선택도(Selectivity)** 다.

```
선택도 = 조건에 매칭되는 행 수 / 전체 행 수

선택도 낮음 (0.001%) → 인덱스 매우 유리
선택도 높음 (30%~)   → Full Table Scan이 유리
```

옵티마이저는 테이블 통계 정보를 보고 선택도를 계산해서 자동으로 실행 계획을 선택한다.

```sql
EXPLAIN SELECT name, age FROM users WHERE name = 'Alice';
-- type = ref  → 인덱스 사용 (선택도 낮음)
-- type = ALL  → Full Scan 선택 (선택도 높음)
```

> 커버링 인덱스가 불가하더라도 인덱스는 **"읽어야 할 행의 범위를 좁히는 역할"** 을 한다.
> 매칭 건수가 적을수록 이중 조회 비용보다 범위 축소 효과가 크기 때문에 인덱스가 유리하다.

## 7. I/O 방식 종합 정리

| 조회 방식 | I/O 종류 | 언제 유리? |
|-----------|----------|-----------|
| Full Table Scan | 순차 I/O | 결과가 많을 때 |
| 세컨더리 인덱스 스캔 | 랜덤 I/O (이중 조회) | 결과가 적을 때 (소수 행) |
| 세컨더리 인덱스 + 커버링 | 랜덤 I/O (1회) | 조회 컬럼이 인덱스 안에 있을 때 |
| PK(클러스터형) 단건 | 포인트 조회 | Buffer Pool이 보완 |
| PK(클러스터형) 범위 | 준순차 I/O | 범위 조회에 강함 |

> **핵심 원칙**: 인덱스가 있다고 항상 빠른 것이 아니다.
> 옵티마이저는 랜덤 I/O 비용과 선택도를 계산해서 Full Table Scan이 더 유리하면 인덱스를 사용하지 않는다.

## 다음 탐구 주제

- **Auto Increment PK vs UUID PK의 I/O 차이**
  Auto Increment는 항상 마지막 페이지에 삽입되어 페이지 분할이 없다.
  반면 UUID처럼 랜덤한 값을 PK로 쓰면 페이지 내부 정렬 과정에서 페이지 분할(Page Split)이 자주 발생한다. 이것이 성능과 공간 낭비에 어떤 영향을 주는가?

- **Redo Log가 순차적으로 쓸 수 있는 이유**
  InnoDB가 랜덤 I/O를 순차 I/O로 변환하기 위해 Redo Log를 사용한다고 했는데,
  왜 Redo Log는 순차적으로 쓸 수 있는 구조인지 분석.

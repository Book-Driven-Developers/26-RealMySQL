# 딥다이브: 디스크 I/O 원리 (HDD / SSD)

> HDD의 물리적 동작 구조부터 랜덤 I/O와 순차 I/O의 비용 차이,
> SSD에서도 순차 I/O가 유리한 이유, 그리고 섹터와 페이지의 관계까지 분석한다.

---

## 1. HDD의 물리적 동작 원리

HDD에서 데이터를 읽으려면 다음 3단계를 거친다.

```
          ┌─────────────────────────┐
          │  스핀들(Spindle)        │  ← 회전축
          │  ┌───────────────────┐  │
          │  │   플래터(Platter)  │  │  ← 자성 원판 (데이터 저장)
          │  │   ○ ─ Track      │  │  ← 동심원 형태의 트랙
          │  │   Sector(섹터)   │  │  ← 트랙을 조각낸 단위 (보통 512B or 4KB)
          │  └───────────────────┘  │
          │  [헤드(R/W Head)] ──→   │  ← 데이터 읽기/쓰기 바늘
          └─────────────────────────┘
```

| 단계 | 이름 | 설명 | 소요 시간 |
|------|------|------|-----------|
| 1 | **탐색 시간(Seek Time)** | 헤드가 해당 트랙 위로 이동 | ~5~10ms |
| 2 | **회전 지연(Rotational Latency)** | 원하는 섹터가 헤드 아래로 올 때까지 대기 | ~2~4ms |
| 3 | **전송 시간(Transfer Time)** | 실제 데이터 읽기/쓰기 | 매우 짧음 |

전체 I/O 시간의 90% 이상이 1, 2번(이동+대기)에 소비된다.

---

## 2. 랜덤 I/O vs 순차 I/O의 비용 차이

```
순차 I/O: [섹터1][섹터2][섹터3][섹터4]  ← 연속된 위치
          → 헤드 이동 1번, 이후 연속으로 읽음

랜덤 I/O: [섹터1] ... [섹터500] ... [섹터23] ... [섹터901]  ← 흩어진 위치
          → 데이터마다 헤드 이동 발생
```

10,000개 데이터를 읽는다고 가정하면:

```
순차 I/O: Seek 1번      = 7ms
랜덤 I/O: Seek 10,000번 = 70,000ms = 70초
```

이것이 MySQL 옵티마이저가 **Full Table Scan vs Index Scan** 전략을 고민하는 근본적인 이유다.

---

## 3. SSD에서도 순차 I/O가 유리한 이유

SSD는 플래터와 헤드가 없어 Seek Time이 거의 0에 가깝다.
그러나 랜덤 I/O는 SSD에서도 여전히 순차 I/O보다 불리하다.

| | HDD | SSD |
|--|-----|-----|
| 랜덤 읽기 | ~100 IOPS | ~100,000 IOPS |
| 순차 읽기 | ~150 MB/s | ~550 MB/s |
| 랜덤 vs 순차 격차 | **엄청남** | **상대적으로 작지만 존재함** |

### 3-1. OS Prefetcher (미리 읽기)

OS는 순차 접근 패턴을 감지하면 다음 페이지를 미리 캐시에 적재한다.

```
순차 I/O: [페이지 1] 요청 → OS가 [페이지 2,3,4...] 미리 캐시 적재
          → 다음 요청은 이미 메모리에 존재 → 디스크 접근 0번

랜덤 I/O: [페이지 1] 요청 → 다음에 뭘 읽을지 OS가 예측 불가
          → 매번 디스크 직접 접근
```

### 3-2. FTL (Flash Translation Layer)

SSD 내부에는 논리 주소(OS가 아는 주소)를 물리 주소(실제 NAND 셀 위치)로 변환하는 테이블이 존재한다.

```
논리 주소(LBA) → 물리 주소(NAND 셀)
  LBA 1000   →  Cell Block 47, Page 3
  LBA 1001   →  Cell Block 12, Page 8  ← 랜덤이면 변환 테이블 조회 비용 반복
```

랜덤 I/O가 많을수록 FTL이 수만 개의 변환 테이블을 실시간으로 조회해야 한다.

### 3-3. Write Amplification (쓰기 증폭)

SSD는 NAND 플래시 특성상 **덮어쓰기(Overwrite)가 불가능**하다.

```
HDD: 섹터 5번에 새 데이터 → 그냥 덮어씀

SSD: 셀에 새 데이터 쓰려면
     1. 해당 블록(수백 KB) 전체를 임시 공간에 복사
     2. 블록 전체를 지움 (Erase)  ← NAND 특성상 블록 단위로만 지울 수 있음
     3. 수정된 내용을 포함해서 전체 블록을 다시 씀
```

랜덤 쓰기가 많으면 1바이트 수정에도 수백 KB 블록 전체를 지우고 다시 써야 한다.
이것이 **쓰기 증폭(Write Amplification)** 이며 SSD 수명 단축과 성능 저하로 이어진다.

### 3-4. MySQL의 대응 - 랜덤 I/O를 순차 I/O로 변환

```
랜덤 쓰기 발생
      ↓
InnoDB: "디스크에 바로 쓰지 않겠다"
      ↓
변경 내용을 순차적으로 Redo Log에 기록 (Sequential Write)
      ↓
Buffer Pool에서 모아두었다가 한 번에 디스크에 Flush (Sequential Write)
```

InnoDB 설계의 핵심은 **랜덤 I/O를 순차 I/O로 변환**하는 것이다.

## 4. 섹터와 페이지의 관계 - 레이어 분리

페이지와 섹터는 같은 레이어가 아니다. 담당자가 다르다.

```
InnoDB         "page 3번 읽어줘 (16KB)"
   ↓
파일시스템      ".ibd 파일의 offset 48KB 위치에서 16KB 읽어"
   ↓
OS             "디스크 블록 12, 13, 14, 15번 읽어" (4KB 블록 × 4)
   ↓
디스크 컨트롤러  섹터 단위로 실제 읽기 수행
```

InnoDB는 섹터를 전혀 모른다. 파일의 어느 offset에서 16KB를 요청할 뿐이고, 나머지는 OS/파일시스템이 처리한다.

```
InnoDB 페이지 (16KB)       ← InnoDB 관리 단위 (실제 행 데이터 존재)
    ↓ 구성됨
OS 블록 (4KB) × 4          ← 운영체제 관리 단위
    ↓ 구성됨
디스크 섹터 (512B) × 32    ← 하드웨어 물리 단위
```

### 페이지 내부 섹터는 연속인가?

**일반적으로는 연속이다.** 파일시스템의 Extent 할당 방식 덕분이다.

```
InnoDB가 .ibd 파일을 확장할 때
→ 파일시스템이 1MB 단위(Extent)로 연속된 디스크 블록을 한번에 확보
→ 그 안의 블록들은 물리적으로 연속

┌────────────────────────────────┐
│ Extent (1MB = 연속된 디스크 블록) │
│  [Page1 섹터들] [Page2 섹터들]  │  ← 같은 Extent 안에서는 연속
└────────────────────────────────┘
```

단, **디스크 단편화(Fragmentation)** 가 심해지면 깨진다.

```
단편화 발생 시:
.ibd 파일 자체가 디스크 여기저기 흩어짐
→ Page3의 16KB가 섹터 100~115 / 섹터 830~845 로 쪼개질 수 있음
→ OS가 두 번의 다른 위치 읽기 수행
→ HDD에서는 헤드 이동 발생
```

이것이 HDD에서 **디스크 조각 모음(Defragmentation)** 이 필요한 이유다.
SSD는 물리적 이동이 없으므로 단편화의 영향이 훨씬 적다.

### 레이어별 보장 범위 정리

| 레이어 | 담당자 | 보장 여부 |
|--------|--------|-----------|
| 페이지 내부 섹터 연속성 | OS/파일시스템 | 일반적으로 연속, 단편화 시 깨짐 |
| 페이지간 논리 순서 | InnoDB (연결 포인터) | 항상 보장 |
| 페이지간 물리 순서 | OS/파일시스템 | 보장 안 됨 |

> InnoDB의 관심사는 **페이지** 단위다. 섹터 연속성은 파일시스템에 위임한다.
> 그래서 InnoDB 성능 튜닝은 섹터가 아닌 **페이지 단위**로 이야기한다.

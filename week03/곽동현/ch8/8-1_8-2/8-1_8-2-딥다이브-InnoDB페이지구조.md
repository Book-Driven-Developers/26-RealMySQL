# 딥다이브: InnoDB 페이지 구조와 물리적 저장

> InnoDB가 데이터를 어떤 단위로 관리하는지,
> 페이지들이 논리적으로 어떻게 연결되는지,
> 그리고 클러스터형 인덱스의 물리적 순서 보장 여부를 분석한다.

---

## 1. InnoDB 페이지 구조

InnoDB는 **16KB 페이지(Page)** 단위로 데이터를 관리한다.
이 페이지들은 **이중 연결 리스트(Doubly Linked List)** 로 논리적으로 연결된다.

```
┌─────────────────────────────┐
│ Page 3  (16KB)              │
│  [PK=1, 데이터]              │
│  [PK=2, 데이터]              │  ← 페이지 내부는 PK 순서로 정렬
│  [PK=3, 데이터]              │
│  next_page → Page 7         │  ← 다음 페이지 포인터 (논리적 연결)
└─────────────────────────────┘
         ↓ (물리적 위치와 무관)
┌─────────────────────────────┐
│ Page 7  (16KB)              │
│  [PK=4, 데이터]              │
│  [PK=5, 데이터]              │
│  prev_page → Page 3         │
│  next_page → Page 15        │
└─────────────────────────────┘
```

---

## 2. 클러스터형 인덱스 ≠ 물리적 순서 보장

클러스터형 인덱스는 **논리적 순서**를 보장하지, **물리적(섹터) 순서**를 보장하지 않는다.

```
PK=3 삭제 후 PK=6 삽입:

물리적 페이지 순서: [PK=1][PK=2][PK=6(빈공간에 삽입)][PK=4][PK=5]
논리적 PK 순서:    1 → 2 → 4 → 5 → 6  (포인터로 유지)
```

데이터는 삽입 순서대로 빈 공간에 저장되기 때문에 물리적 위치가 흩어진다.
InnoDB는 포인터로 논리적 순서만 유지할 뿐이다.

진짜 물리적 순서를 맞추려면 테이블 재구성이 필요하다.

```sql
OPTIMIZE TABLE orders;
-- 또는
ALTER TABLE orders ENGINE=InnoDB;
-- PK 순서로 물리적 재작성 (이 시점만 논리 순서 = 물리 순서)
```

## 3. InnoDB 테이블스페이스 공간 관리

```
.ibd 파일 (테이블스페이스)
┌──────────────────────────────────────────┐
│  Extent 1 (1MB = 64 pages)               │
│  Extent 2 (1MB = 64 pages)               │
│  Extent 3 (1MB = 64 pages)               │
└──────────────────────────────────────────┘
```

- InnoDB 내부에서 **Free List / Used List** 로 페이지를 관리한다.
- 페이지가 필요하면 Free List에서 꺼낸다.
- 여유 공간이 없으면 OS에게 새 Extent 요청 → 파일 크기 증가.
- **OS는 파일 블록의 위치만 결정**하고, InnoDB는 파일 안에서 페이지를 논리적으로 관리한다.

---

## 4. Full Table Scan의 실제 동작

Full Table Scan은 페이지 연결 포인터를 따라 전체 페이지를 순서대로 읽는다.

```
Full Table Scan 실행:

Page 3 읽기  → next_page 포인터 따라가기
Page 7 읽기  → next_page 포인터 따라가기
Page 15 읽기 → next_page 포인터 따라가기
...
```

페이지들이 물리적으로 흩어져 있어도 InnoDB의 **Read-Ahead(미리 읽기)** 가 보완한다.

```
InnoDB Read-Ahead:
"어차피 다음 페이지도 읽을 거잖아"
→ 한 번에 여러 페이지를 묶어서 읽음
→ I/O 횟수 감소
```

> Full Table Scan = 순차 I/O라고 하는 이유는 페이지 연결 포인터를 순서대로 따라가며
> Read-Ahead로 묶어 읽기 때문이다. 완전한 물리적 순차는 아니지만 랜덤 I/O보다 훨씬 효율적이다.
